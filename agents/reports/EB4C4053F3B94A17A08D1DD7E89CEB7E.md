## Advisories (non-blocking)

- **Dead code dropped.** The file `ob-onchange-functions.js` is a shared core utilities file. The
  functions `organizationCurrency`, `processDefinitionUIPattern`, and `colorSelection` are not bound to
  any parameter of this process and were not migrated (dead-code rule).
- **Cloning.** The four overdue parameters share one `agingProcessDefinitionOverdue` body; it is pasted
  identically into all four. Because the body is short and self-contained, it lives directly in each
  parameter rather than in `em_etmeta_payscript_logic`.
- **Sibling process.** `0D37A9F6109549DEB058373EF2DAEB6A` (Aging Balance for Receivables) uses the same
  two functions; the same code applies there once it is migrated.
- **valueMap semantics.** Classic mutated `gl.valueMap` directly. The first migration translated this to
  `getValueMap()`/`setValueMap()`, which proved incorrect (see `## Updates`): `getValueMap()` returns an
  **array** so the classic object-key assignment was lost, and `setValue(id)` alone never sets the
  display `$_identifier`. The org handler now uses `setValueFromRecord({id, _identifier})`, which sets
  the value and its identifier together — the idiomatic new-UI equivalent of the classic
  `valueMap[id]=label; setValue(id)`.
- **Chained comparison.** The original chained `column1 < column2 && ... < column4` is kept as a single
  boolean (`ascending`), not as nested ternaries; no rule violated.

## Updates (2026-06-11) — two bugs found in manual testing, fixed

Manual testing surfaced two incorrect behaviors (no console errors). Root causes and fixes:

### Bug 1 — overdue validation always warned for valid values, then reset the field to 0

**Cause (platform, fixed — no JS change).** The overdue parameters are Integer (`reference "11"`) with
**string** defaults (`"30"`, `"60"`, `"90"`, `"120"`). In the new UI `form.getItem('ColumnN').getValue()`
returned the raw **string**, so `column1 < column2 && … < column4` compared **lexicographically** —
`"90" < "120"` is `false` (`"9" > "1"`) — and the warning fired for valid input. `item.setValue('')`
then became `0` because the field is mandatory. Classic worked because SmartClient Integer items return
**numbers** from `getValue()`.

**Fix (platform).** `item.getValue()` now coerces to a number for numeric parameters
(Integer / Number / Quantity / Decimal), mirroring SmartClient, so the **unmodified** migrated hook runs
correctly. Implemented in:
- `utils/form/constants.ts` — new reusable `isNumericReference(reference)` predicate (single source of
  truth for the numeric reference ids).
- `utils/processes/definition/scriptProxies.ts` — `coerceParamValue(raw, parameter)`; `createItemProxy`
  accepts the resolved `ProcessParameter` and coerces numeric values in `getValue` (empty / null /
  already-numeric pass through unchanged; non-numeric params are untouched). New `findParameter(name,
  parameters)` resolves the parameter by name, `dBColumnName`, or map key — the `parameters` map is keyed
  by `dBColumnName`, so a lookup by the resolved `name` would miss it; `resolveFormKey` now delegates to
  `findParameter`. Threaded through `createFormProxy.getItem`, `getFields`, and
  `useParameterChangeHooks.runHook`.

**The overdue JS hook is unchanged** — no manual paste needed for the four Group parameters.

### Bug 2 — changing Organization did not update General Ledger

**Cause (migrated JS).** The first migration translated the classic `gl.valueMap[id]=label; setValue(id)`
into `getValueMap()`/`setValueMap()`/`setValue()`. But `getValueMap()` returns a **`ListOption[]` array**,
so the classic object-key assignment `valueMap[id]=label` was dropped on normalization, and `setValue(id)`
alone never sets the selector's display `$_identifier`. A process selector only renders a value that is
not in the loaded options when `$_identifier` is set, so the GL appeared unchanged.

**Fix (migrated JS — apply manually).** The Organization `em_etmeta_on_parameter_change` callback now
calls `gl.setValueFromRecord({ id: data.response.value, _identifier: data.response.identifier })`, which
sets the value **and** the identifier together (the idiomatic new-UI equivalent of the classic pattern).
The corrected code is already shown above in "Paste into the Organization parameter". **No platform
change** was required for this bug; `setValueFromRecord` is already a live method on the onChange path.
