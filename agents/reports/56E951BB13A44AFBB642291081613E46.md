## Generated code per field

### `obuiapp_parameter` → **Organization** (`AD_Org`, `1EE24A173A464164B46519D7FDA3A667`) → `em_etmeta_on_parameter_change`

```js
(item, view, form) => {
  const CHANGE_ACCT_SCHEMA_HANDLER =
    "com.etendoerp.financial.reports.advanced.handler.ChangeAcctSchemaActionHandler";
  const GENERAL_LEDGER_FIELD = "GL_Journal_ID";

  const organization = item.getValue() ? item.getValue() : "";

  const setAcctSchema = (response, data) => {
    const generalLedger = form.getItem(GENERAL_LEDGER_FIELD);
    if (generalLedger) {
      generalLedger.setValueFromRecord({
        id: data.acctSchema,
        _identifier: data.acctSchemaIdIdentifier,
      });
    }
  };

  if (organization !== "") {
    OB.RemoteCallManager.call(
      CHANGE_ACCT_SCHEMA_HANDLER,
      { organization: organization },
      {},
      setAcctSchema
    );
  }
}
```

Compile-check: evaluates to a function, arity 3 — OK.

## Advisories (non-blocking)

1. **Dead code dropped.** Three ETFRA functions (`showDatesFieldsOnChange`, `showHideDimensions`,
   `showDocumentNoField`) live in the shared family but are not bound to this process; they were not
   migrated. They belong to sibling ETFRA reports — migrate them only when those processes are in scope,
   per the cluster cloning rule. (For example, `showDatesFieldsOnChange` is the in-scope hook for D37588.)
2. **`valueMap` + `refreshDisplayValue` collapsed to `setValueFromRecord`.** The classic callback sets
   the selector value (`setValue`), injects the display label into `valueMap`, then forces a redraw
   (`refreshDisplayValue`). In the new UI these three steps are a single `setValueFromRecord({ id,
   _identifier })`, which sets both the value and the displayed identifier for a value not in the loaded
   option list. Behaviorally equivalent; the migrated body relies on the platform's display handling
   rather than a manual redraw.
3. **Null-guard added.** The classic code assumes `getItem('GL_Journal_ID')` always returns a widget.
   The migrated body guards the lookup (`if (generalLedger)`), so a missing/renamed General Ledger field
   degrades gracefully instead of throwing inside the callback. Functionally equivalent for the present
   field set.
4. **Empty-organization no-op preserved.** The classic guard `if (organization !== '')` is kept, so
   clearing the Organization field does not fire the handler — identical to Classic.
5. **Backend handler dependency.** The flow calls the Java handler
   `com.etendoerp.financial.reports.advanced.handler.ChangeAcctSchemaActionHandler` (via
   `OB.RemoteCallManager.call` → `callAction`), exactly as Classic does. QA requires the module deployed
   and the handler returning `{ acctSchema: <id>, acctSchemaIdIdentifier: <label> }`. This is unchanged
   backend behavior, not a migration gap.
6. **No global namespace bootstrap needed.** The classic file starts with `OB.ETFRA = {}` /
   `OB.ETFRA.OnChange = {}` to register the namespace. In the new UI each hook owns its scope, so no
   namespace self-registration is required for this single self-contained `onChange`; `em_etmeta_payscript_logic`
   stays empty.
7. **Cluster cloning note.** This is the ETFRA namespace-sharing cluster. The same `organizationOnChange`
   body must be cloned into `636EF6F0F8B64E94A8247930569B98CA` (Journal Entries Report Advanced), which
   also binds it. A future fix to this logic must be applied to each member row independently.

## Manual-test checklist

Run in the new UI, on the **General Ledger Report Advanced** process dialog.

1. **Open the dialog.** Confirm it opens with the standard parameter form (no custom component) and no
   console errors.
2. **Select an Organization.** Pick a value in the **Organization** parameter. Expect: the
   **General Ledger** parameter is auto-populated with the accounting schema returned by the handler,
   and the selector shows the correct **label** (not a raw id).
3. **Change to a different Organization.** Expect the **General Ledger** value and its displayed label to
   update to the new organization's accounting schema.
4. **Clear the Organization** (set it to empty). Expect **no** call to the handler and the General Ledger
   field is left unchanged (no error) — matching the classic empty-organization guard.
5. **Network check (optional).** On each non-empty Organization change, confirm a single POST to the
   `ChangeAcctSchemaActionHandler` with `{ organization: <selected org id> }`, and that the UI applies the
   returned `acctSchema` / `acctSchemaIdIdentifier`.
6. **Label correctness.** Confirm the General Ledger selector renders the returned identifier as its
   visible text even if that schema was not previously in the field's loaded option list (validates
   `setValueFromRecord`).
7. **No console errors** during any of the above.
