## In-scope code: `OB.Jobs.ProcessShipment.onLoad(view)` (lines 136–188)

Behaviour:
1. Reads the `DocAction` item and the selected records from the launching grid.
2. Builds distinct `documentStatuses` from the selection. **Note:** unlike the Orders/Invoices variants,
   this loop builds **only** `documentStatuses` — it never populates `documentActions` (the array is
   declared but left empty; see the faithful-bug note below).
3. Calls `com.smf.jobs.defaults.ProcessShipmentDefaults` via `OB.RemoteCallManager.call` with
   `tableId = "319"`.
4. In the callback: filters the `DocAction` value map to the server-returned `actions`, applies it with
   `docActionField.setValueMap(newValues)`, then programmatically selects an option —
   `setValueProgrammatically(documentActions[0])` if available, else
   `setValueProgrammatically(getFirstOptionValue())`.
5. If `actions.length > 0` and the OK button is disabled, enables it: `view.okButton.enable()` (guarded by
   `view.okButton && !view.okButton.isEnabled()`).

### Faithful-bug note: `documentActions` is always empty here

In `ProcessShipment.onLoad` the selection loop (Classic lines 145–150) populates `documentStatuses` but
**not** `documentActions` (the Orders/Invoices variants populate both). Since the callback later tests
`newValues[documentActions[0]]` and `documentActions[0]` is always `undefined`, the record-action branch is
never taken — the field always falls back to `setValueProgrammatically(getFirstOptionValue())` (the first
allowed option). This is a Classic quirk, not a platform limitation. The migrated body **reproduces it
exactly** (keeps `documentActions` declared and unpopulated) so behaviour matches Classic 1:1; it is not
"fixed".

### Process `em_etmeta_onload` — entity `obuiapp_process` (PASTE)

Faithful port of `OB.Jobs.ProcessShipment.onLoad`. Two-parameter signature `(_process, view) => { … }`
because the runtime calls `fn(processDefinition, onLoadView)` — the canonical `view` (carrying `theForm`,
`selectedRecords`) is the second positional argument; a one-parameter `(view) =>` would bind `view` to
`processDefinition`, making `view.theForm` `undefined` and throwing on the first executable line. Remaps
`view.parentWindow.view.viewGrid.getSelectedRecords()` → `view.selectedRecords`, `Array.add` → `.push`,
and the `currentValues[action]` object idiom → `currentValues.find(...)` (because `getValueMap()` returns a
`ListOption[]` array). `documentActions` is kept declared-but-unpopulated to preserve the Classic quirk
(always falls back to the first option). The repeated handler name and table id are hoisted to constants.

```js
(_process, view) => {
  const SHIPMENT_DEFAULTS_HANDLER = 'com.smf.jobs.defaults.ProcessShipmentDefaults';
  const SHIPMENT_TABLE_ID = '319';
  const form = view.theForm;
  const docActionField = form.getItem('DocAction');
  const selectedRecords = view.selectedRecords || [];
  const documentStatuses = [];
  const documentActions = [];

  for (let i = 0; i < selectedRecords.length; i++) {
    const record = selectedRecords[i];
    if (!documentStatuses.includes(record.documentStatus)) {
      documentStatuses.push(record.documentStatus);
    }
  }

  OB.RemoteCallManager.call(
    SHIPMENT_DEFAULTS_HANDLER,
    { documentStatuses: documentStatuses, isProcessing: '', tableId: SHIPMENT_TABLE_ID },
    {},
    (response, data, request) => {
      const actions = data.actions || [];
      const currentValues = docActionField.getValueMap();
      const newValues = {};

      for (let i = 0; i < actions.length; i++) {
        const action = actions[i];
        const match = currentValues.find((option) => option.value === action || option.id === action);
        if (match) {
          newValues[action] = match.label;
        }
      }

      docActionField.setValueMap(newValues);
      // Avoid setting an action that is on the record(s) but not in the dropdown list.
      if (newValues[documentActions[0]]) {
        docActionField.setValueProgrammatically(documentActions[0]);
      } else {
        docActionField.setValueProgrammatically(docActionField.getFirstOptionValue());
      }

      // If the OK button is not enabled but there are actions to select, enable it.
      if (actions.length > 0 && view.okButton && !view.okButton.isEnabled()) {
        view.okButton.enable();
      }
    }
  );
}
```

Compile-check: written to `/tmp/babel_onload.js` and validated as a bare arrow function
(`typeof f === "function"` → OK).

### `em_etmeta_onprocess` — entity `obuiapp_process`

**LEAVE EMPTY.**

Justification (what it does / what it makes work / what is lost / why it is added or not):

- **What it does / would do.** `em_etmeta_onprocess` runs when the user presses Execute. With it empty and
  `classname` set (`com.smf.jobs.defaults.ProcessShipment`, i.e. `javaClassName` populated), the runtime
  routes through the standard direct-Java path: `useProcessExecution.ts` `handleExecute` →
  `handleDirectJavaProcessExecute` → `executeJavaProcess(buildStandardJavaPayload(...))`, which POSTs the
  standard payload (record context + full `_params` + `_buttonValue` + `_entityName`) to `_action =
  com.smf.jobs.defaults.ProcessShipment` and dispatches server `responseActions`.
- **What it makes work.** The actual Shipment processing (the configured Java class running on the selected
  shipment record(s)). The classic `processRecords.js` defines **no** `clientsidevalidation`/`onProcess`
  for ProcessShipment — only `onLoad` — so there is no client-side submit logic to port. The standard
  direct-Java path already performs the submit.
- **What would be lost by adding code here.** Adding a hand-built submit wrapper (as some pre-existing
  rows in this family carry) would force the non-empty branch and **lose** what the standard path provides:
  the launching-record context merge (`buildProcessPayload`), the full merged `_params` (a superset of any
  hand-picked subset), and registry-first `responseActions` dispatch with retry/refresh/toast/close. It
  would duplicate the framework's default execution with strictly less context.
- **Why it is added (it is not).** There is no source behaviour to reproduce and the standard path is a
  strict superset, so the column is intentionally left EMPTY — equal-or-better behaviour for this process.

## Advisories (non-blocking)

1. **Dangling `onchangefunction`.** "Document Action" binds
   `OB.Jobs.ProcessShipment.onChangeDocumentAction`, which is undefined in the source — a no-op in Classic
   (copy-paste leftover from the Invoices variant). Left EMPTY; not migrated. If the business actually
   wants the Invoices-style "show Void/POReference fields on VO/RC" behaviour for Shipment, that is a **new
   feature**, not a migration, and must be requested explicitly.
2. **Faithful Classic quirk: `documentActions` never populated.** `ProcessShipment.onLoad` only builds
   `documentStatuses`, so the record-action pre-selection is dead code and the field always falls back to
   the first allowed option. Reproduced exactly; not "fixed".
3. **`Array.add` → `.push`.** Classic uses `documentStatuses.add(...)` (SmartClient `Array` augmentation),
   ported to `.push(...)`. Not a platform gap.
4. **`viewGrid.getSelectedRecords()` → `view.selectedRecords`.** Resolved via the onLoad `hookData` per the
   architecture doc.
5. **Family cloning.** The `processRecords` family shares the `onLoad` shape; each process is migrated on
   its own row. This row's onLoad differs from the Orders rows (handler `ProcessShipmentDefaults`,
   `tableId = "319"`, and the `documentActions`-not-built quirk). A future fix to the family logic must be
   re-applied per row.
