## 4. Feasibility gate

The process's central behavior is: press **Process** → backend validates and returns a custom
`EAPM_Popup` response action → the client opens a **runtime-built parameter form** → the user fills it →
OK submits that form to a **second** action handler (`PaymentExecutionProcessActionHandler`) and refreshes
the `payment_pick` grid with a success/error banner.

Every mechanism this requires now has a faithful target in the substrate (proofs in §3). The reachability
concern that blocked the prior pass is resolved: the `EAPM_Popup` dispatch is **not** the dead
`em_etmeta_onprocess`/Pick&Execute path — it travels the **response-action** path
(`executeJavaProcess` → `readDispatchableResponseActions` → registry-first `executeJSON`), which **does**
run for the Window-Reference Process submit. The builder is registered at modal-open time
(`evaluateModuleScope` runs the module body against the shared OB shim, and `OB.*` writes persist:
`moduleScope.ts:120-136`), so it is already in the registry when the Process button fires later. Per the
conservative gate, with no `best-effort` or `unsupported` entry remaining, the process is **migratable**.

### `obuiapp_process` — `em_etmeta_payscript_logic`

Registers the `EAPM_Popup` builder into the shared OB action registry. When the backend returns the
`EAPM_Popup` response action, the builder opens the dynamic form, and on Confirm POSTs the collected
values to the second handler, then refreshes the grid and shows the banner.

```js
// @module-scope
const EAPM_POPUP_ACTION = "EAPM_Popup";
const SECOND_HANDLER = "com.etendoerp.advpaymentmngt.actionHandler.PaymentExecutionProcessActionHandler";
const PAYMENT_PICK_GRID = "payment_pick";
const SEVERITY_SUCCESS = "success";
const SEVERITY_ERROR = "error";
const GENERIC_ERROR = "Error executing the process";

const refreshPaymentGrid = () => {
  OB.Utilities.Action.executeJSON([{ refreshGrid: {} }, { refreshGridParameter: { gridName: PAYMENT_PICK_GRID } }]);
};

const handleExecutionResponse = (data) => {
  if (!data) {
    messageBar.setMessage(SEVERITY_ERROR, null, GENERIC_ERROR);
    return;
  }
  if (data.success) {
    refreshPaymentGrid();
    messageBar.setMessage(SEVERITY_SUCCESS, null, data.message);
  } else {
    messageBar.setMessage(SEVERITY_ERROR, null, data.error || GENERIC_ERROR);
  }
};

OB.Utilities.Action.set(EAPM_POPUP_ACTION, async (actionData) => {
  const collected = await openDynamicForm({
    title: actionData.executionProcessName,
    fields: actionData.processParameters,
  });
  if (!collected) {
    return;
  }
  OB.RemoteCallManager.call(
    SECOND_HANDLER,
    { _params: actionData, processParameters: collected },
    {},
    (rpcResponse, data) => handleExecutionResponse(data)
  );
});

return {};
```

## 6. Advisories (non-blocking)

- **Identical onChange body on two params.** The classic single function is intentionally duplicated into
  both parameter columns (the platform binds onChange per parameter; there is no shared registration slot
  for a parameter hook). This is the faithful 1:1 translation, not accidental duplication.
- **Banner instead of `showMsgInProcessView`.** The classic OK callback used
  `OB.Utilities.Action.execute("showMsgInProcessView", paramObj)`; the migration uses
  `messageBar.setMessage(...)`, the documented in-modal banner equivalent (same visible outcome).
- **Field scope.** `openDynamicForm` renders only `TEXT`/`CHECK` field types — exactly what the classic
  popup emits (`OBTextItem`/`OBCheckboxItem`). Richer types would be an additive future extension; not
  needed here.
- **Cancel safety.** If the user cancels the dynamic form (or the modal closes), `openDynamicForm` resolves
  to `null` and the builder returns without POSTing — matching the classic Cancel button's `closeClick()`.
- `OB.EAPM.*` / `EAPM_*` are specific to this process; no cluster-cloning obligation.

   wrongly refreshed.
