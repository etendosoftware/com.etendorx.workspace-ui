## 4. Feasibility gate

The process's central behavior is: press **Process** → backend validates and returns a custom
`EAPM_Popup` response action → the client opens a **runtime-built parameter form in a popup** → the user
fills it → OK submits that form to a **different** action handler and refreshes the grid.

The mechanism that renders and runs that popup (`isc.defineClass` / `isc.OBPopup` / `isc.DynamicForm` /
`isc.OBFormButton` / `isc.VLayout`, plus the `EAPM_Popup` named action that builds them) has **no
equivalent** in the new UI substrate. Per the conservative gate, at least one mechanism is `unsupported`,
so the process is **blocked**. Emitting only the onChange toggle while silently dropping the popup-driven
execution flow would remove the process's primary behavior, which the gate forbids.

### What the new UI would need to unblock this

1. A way to render an **action-triggered nested dialog containing a dynamically-built parameter form**
   (fields described by server JSON: checkbox/text items with defaults), the new-UI analogue of
   `isc.DynamicForm` inside `isc.OBPopup`. The current `confirm`/`warn`/`say` helpers are plain-text only,
   and `etmetaCustomComponent` is a process-open rendering flag, not an action-time dialog.
2. A wiring so a backend `responseActions` entry with a custom key (`EAPM_Popup`) carrying **data only**
   can resolve to a script-provided builder. Today `custom` only runs `paramObj.func` when the payload
   itself contains a function; a named, data-only custom action defined in `em_etmeta_payscript_logic`
   has no dispatch route from `responseActionDispatcher`.
3. The OK handler then needs to POST the collected form values to a **second, named** action handler
   (`PaymentExecutionProcessActionHandler`) — `OB.RemoteCallManager.call` covers this once items 1–2 exist.
