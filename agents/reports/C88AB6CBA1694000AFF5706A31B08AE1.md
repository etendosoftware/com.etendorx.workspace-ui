## 3. Feasibility gate — BLOCKED

The single in-scope hook cannot be migrated with behavioral parity because it depends on two
platform capabilities the new UI does not provide. (The earlier `BigDecimal`-not-injected gap is
now resolved — see §0.)

1. **`BigDecimal.prototype.setScale(2)`** — `CalculateSelected` scales each per-currency accumulator
   to 2 decimals (`...add(...).setScale(2)`, `ob-rem-utilities.js:93`). The now-injected `BigDecimal`
   global implements only `add` / `subtract` / `compareTo` / `toString` / `toNumber` /
   `prototype.ZERO` (`utils/ob/bigDecimal.ts`); calling `setScale` throws
   `TypeError: ...setScale is not a function`. Dropping the scaling or emulating it with `Number`
   would change rounding/scale and break amount parity with the server-side check, so it is not a
   faithful translation.

2. **`OB.Constants.FIELDSEPARATOR` / `OB.Constants.IDENTIFIER`** — used to build the grid record key
   for the display currency (`'currency' + sep + identifier`, `ob-rem-utilities.js:77`). The `OB`
   shim exposes `PropertyStore`, `I18N`, `Format`, `Utilities`, `Styles`, `MessageBar`,
   `TestRegistry`, `RemoteCallManager`, `Datasource` — but **not `Constants`** (`interface OBShim` in
   `utils/ob/types.ts` has no `Constants` member). Reading `OB.Constants.FIELDSEPARATOR` throws.

Because at least one entry is `unsupported`, the conservative gate applies: no migrated code is
generated. The process stays `blocked` in the inventory.

### Platform changes

1. **`BigDecimal.setScale(scale)` — added** (`packages/MainUI/utils/ob/bigDecimal.ts`). The class now
   tracks an optional fixed `scale`; `setScale(scale)` rounds the value to `scale` decimals with
   `ROUND_HALF_UP` and returns a **new** `BigDecimal` whose `toString()` renders exactly that many
   decimals (zero-padded, e.g. `new BigDecimal("5").setScale(2).toString() === "5.00"`). For the
   ≤2-decimal money amounts these scripts handle nothing is ever discarded, so the result is identical
   to the classic `setScale(scale)` default (`ROUND_UNNECESSARY`) while only zero-padding. `add`/
   `subtract`/`compareTo`/`toNumber`/`prototype.ZERO` are unchanged; `compareTo` ignores scale, so the
   `compareTo(new BigDecimal('0')) === 0` "drop empty currency" check works as in the classic code.

   `OBShim.Constants`; `packages/MainUI/utils/ob/obShim.ts`). Exposes `FIELDSEPARATOR` (`"$"`) and
   `IDENTIFIER` (`"_identifier"`, reusing `IDENTIFIER_KEY` from `utils/columnsConstants.tsx`). No new
   behavior — existing constants surfaced under the Classic namespace. Building the currency key
   `'currency' + OB.Constants.FIELDSEPARATOR + OB.Constants.IDENTIFIER` now yields
   `'currency$_identifier'` without throwing.

No wiring change: the `onGridLoad` hook is compiled with `scriptHookContext`
(`ProcessDefinitionModal.tsx`), which already carries the `OB` shim and `BigDecimal`, so both fixes

```js
(grid, view, parameters) => {
  const ZERO = new BigDecimal('0');
  const SCALE = 2;
  const currencyIsoAttribute = 'currency' + OB.Constants.FIELDSEPARATOR + OB.Constants.IDENTIFIER;
  const totals = {};
  const selectedRecords = grid.getSelectedRecords();
  for (let i = 0; i < selectedRecords.length; i++) {
    const record = selectedRecords[i];
    const currencyIso = record[currencyIsoAttribute];
    let recordAmount = record.amount;
    if (!record.receipt) {
      recordAmount = -recordAmount;
    }
    if (!totals[currencyIso]) {
      totals[currencyIso] = ZERO;
    }
    totals[currencyIso] = totals[currencyIso].add(new BigDecimal(recordAmount.toString())).setScale(SCALE);
    if (totals[currencyIso].compareTo(ZERO) === 0) {
      delete totals[currencyIso];
    }
  }
  let total = '';
  for (const currencyIso in totals) {
    if (Object.prototype.hasOwnProperty.call(totals, currencyIso)) {
      total += totals[currencyIso].toString() + ' ' + currencyIso + ' ';
    }
  }
  if (!total) {
    total = '0';
  }
  grid.view.theForm.getItem('total').setValue(total);
}
```
