- **"Update Invariant Characteristics" popup is out of scope (dropped per the dead-code rule).** The bulk of
  `productCharacteristicsProcess.js` (`:20-53` and `:66-323`) implements a custom SmartClient popup
  (`isc.UpdateInvariantCharacteristicsPopup`, a subclass of `isc.OBPopup`) that builds a tree-reference combo
  per product characteristic, collects the user's selections, and POSTs them to
  `org.openbravo.client.application.event.UpdateInvariantCharacteristicsHandler`. This popup is launched by a
  **custom toolbar button** registered through `OB.ToolbarUtils` / the button registry — **not** by any
  Defined-Process hook column (verified: `on_load_function`, `clientsidevalidation`, `on_refresh_function`
  all empty; the parameter carries no `onchangefunction`). Because it is unreachable from any in-scope
  Defined-Process hook, it is left out of this migration (Section 10.3 dead-code rule). Migrating that button
  would be a **separate effort**: in principle it maps to the new-UI dynamic-form path
  (`openDynamicForm` + `OB.Utilities.Action.set` + `OB.RemoteCallManager.call`), but it is not part of the
  Defined-Process JS this migration covers and is not triggered by the Manage Variants process dialog.
- **`setMessage` signature normalization.** Classic calls `setMessage(severity, text)` (2 args); the new-UI
  message bar is `setMessage(severity, title, text)` (3 args). Ported with `title = null` so the same text

- **`grid.view.messageBar` → `view.messageBar`.** The `onGridLoad` signature `(grid, view, parameters)`
  supplies the `view` directly, so the Classic `grid.view` walk is unnecessary and the migrated body uses the
  argument for clarity and guaranteed availability.
- **`getData().getLength()` kept verbatim.** Equivalent to `grid.getTotalRows()`; the proxy supports both. No

## Updates

### 2026-06-17 — SUBSTRATE FIX: `onGridLoad` message bar on an empty grid (platform, JS untouched)

**Symptom (manual QA, item 5 of the checklist above).** With a product that has **no variants**, the grid
loads zero rows; the migrated `em_etmeta_on_grid_load` enters the `recordCount === 0` branch and calls
`view.messageBar.setMessage(isc.OBMessageBar.TYPE_INFO, null, OB.I18N.getLabel('NoVariants'))` (confirmed
via logs), but **no banner appears and no console error**. The migrated JS is correct; the defect is 100%
platform-side.

**Root cause (two-stage, confirmed statically).** `WindowReferenceGrid`'s datasource-sync effect fired
`onGridLoad` / `dataArrived` on *any* `rawRecords` change, including the **pre-fetch empty placeholder**
(initial render / the `skip` phase, before a real datasource response):
1. On open it fired `onGridLoad` with a phantom `rowCount = 0` → `setMessage` ran **and** advanced the
   dedup ref (`rawRecordsStringRef = "[]"`).
2. The modal's clear-on-open effect (`ProcessDefinitionModal.tsx`, `messageBar.hide()`) runs *after* the
   grid's effect (React fires child effects before parent), wiping the just-set message.
3. The genuine empty delivery (`[]`) then matched the poisoned dedup ref → **early return → `onGridLoad`
   never fired again** → banner never re-set. No error, condition correct — exactly the symptom. This is
   the previously-documented "`on_grid_load` sees `rowCount=0` on open" advisory, now manifesting as a real
   bug whenever the load-time hook acts on an empty grid.

**Fix (frontend, additive, non-regressive).**
- `useDatasource` now returns `hasFirstFetchCompleted` (already tracked internally; "has a real fetch
  completed?", distinct from `loaded`, which is also true during `skip`).
- New pure exported helper `decideDatasourceSync(rawRecords, hasFirstFetchCompleted, lastSignature)`
  returns `{ shouldSync, signature }`. It proceeds only when a real fetch has completed **and** the
  serialized content changed; while gated it returns the signature **unchanged**, so the pre-fetch
  placeholder never poisons the dedup ref. The sync effect now delegates its gate/dedup to it.
- Net effect: `onGridLoad` / `dataArrived` fire on the first **delivered** result (async, hence after the
  clear-on-open commit), so an empty result set reaches the hook exactly once and the `NoVariants` banner
  is set after the clear and stays visible.

**Why it mirrors Classic.** SmartClient fires `dataArrived` / `ongridloadfunction` on data **arrival**,
not on the pre-fetch empty state. The gate restores that contract.

**Non-regression.** Before the first fetch the placeholder sync was a no-op (`setLocalRecords([])` on an
already-empty grid); gating it changes no end state. Grids that always `skip` (no datasource) never had a
real load and now simply don't fire a phantom one. Non-empty grids previously fired `onGridLoad` twice
(phantom 0-row + real) and now fire once with the real rows — closer to Classic, not a regression.

→ 59 suites / 726 tests green, including new `describe("decideDatasourceSync")` (6 cases) and
`describe("hasFirstFetchCompleted")` (3 cases); `npx biome check` on the four touched files → 0 new
diagnostics (remaining ones are pre-existing `noExplicitAny` / `useExhaustiveDependencies` outside the
