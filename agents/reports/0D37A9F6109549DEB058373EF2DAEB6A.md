## Advisories (non-blocking)

- **Dead code dropped.** `ob-onchange-functions.js` is a shared core utilities file. `organizationCurrency`,
  `processDefinitionUIPattern`, and `colorSelection` are not bound to any parameter of this process and
  were not migrated (dead-code rule).
- **Cloning.** The four overdue parameters share one `agingProcessDefinitionOverdue` body; it is pasted
  identically into all four. The body is short and self-contained, so it lives directly in each parameter
  rather than in `em_etmeta_payscript_logic`.
- **Sibling parity.** Code is byte-identical to the Payables sibling
  (`EB4C4053F3B94A17A08D1DD7E89CEB7E`), including its two confirmed fixes: the platform numeric coercion
  for Integer overdue values (no JS change needed) and `setValueFromRecord({id, _identifier})` in the
  Organization callback. Bindings were re-verified against this process's own metadata before reuse.
- **Chained comparison.** The original chained `column1 < column2 && … < column4` is kept as a single
  boolean (`ascending`), not as nested ternaries; no rule violated.

1. **Open the process.** Open "Aging Balance Process Definition for Receivables". Confirm the dialog opens
   normally (no onLoad behavior expected).
2. **Overdue ascending valid.** Set Group One/Two/Three/Four to ascending values (e.g. 30, 60, 90, 120).
   On committing each, confirm no warning banner appears and any existing banner is hidden. (Specifically
   verify the previously-buggy case `90 < 120` is treated as valid — numeric, not lexicographic.)
3. **Overdue not ascending.** With all four set, change one so the sequence is not strictly ascending
   (e.g. Group Three = 30 while Group Two = 60). On commit, confirm: the just-changed field is cleared and
   a warning banner shows the `OBUIAPP_OverdueNotValid` label text.
4. **Partial values.** Leave one of the four groups empty. Confirm no warning fires regardless of the
   others (validation only runs when all four are set).
5. **Recovery.** After a warning, set a valid ascending sequence and confirm the banner hides.
6. **Organization → General Ledger.** Change the Organization parameter. Confirm the General Ledger
   (AccSchema) field is populated with the GL returned by the action handler for that organization, and
   that its displayed identifier (label) is correct. Try a second organization and confirm the GL updates
   again.
7. **Organization with no GL / error.** If an organization has no associated GL, confirm there is no JS
   error and the field behaves gracefully.
