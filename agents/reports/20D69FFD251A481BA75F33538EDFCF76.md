## Verified substrate (why this is now unblocked)

1. **Grid visibility — supported.** `scriptProxies.ts` exposes `GridProxy.show()/hide()` wired by
   `buildGridVisibility(controller, paramName)` to `FieldController.setDisplayed(paramName, …)`.
   `assignItemCanvas` wires it for `canvas.viewGrid`, and `WindowReferenceGrid.tsx` passes the same
   visibility to the `onGridLoad` grid arg. The `Accounts` parameter has no static `displayLogic`, so
   the script-set flag wins. `view.theForm.getItem('Accounts').canvas.viewGrid.hide()/show()` toggles
   the grid parameter — the defining UX (grid hidden until "Search", then shown) is reproducible.
2. **`paramObj._processView` — false negative.** The actions are registered **inside `onLoad`**, which
   receives `view`; the registered closures capture `view` lexically and call
   `view.theForm.getItem(...)` directly. `OB.Utilities.Action` keeps one shared registry per modal
   (`ob/action.ts`), so actions registered in `onLoad` are visible to the dispatcher, and `executeJSON`
   is registry-first, so the registered closures run.
3. **`onGridLoad` selection / mutation — supported.** The `onGridLoad` grid arg is built with the live
   `GridController` (`WindowReferenceGrid.tsx`), so `grid.data.allRows` returns the loaded rows and
   `grid.setEditValue(rowIndex, colName, value)` overlays the edit through the grid's change handler.
   Classic `selectAllRecords()` + `getSelection()` here means "iterate the loaded rows"; the in-place
   `item.amount = totalAmount` maps to `grid.setEditValue(index, 'amt', total)`.

### `em_etmeta_payscript_logic` — entity `obuiapp_process`

```js
const ACCOUNTS_PARAM = 'Accounts';
const AMOUNT_PARAM = 'amount';
const AMOUNT_COLUMN = 'amt';

const state = { lines: null };

const sumAccountAmount = (rowId) => {
  const lines = state.lines;
  if (!lines || !Array.isArray(lines.accounts)) {
    return null;
  }
  const matching = lines.accounts.filter((account) => account.id === rowId);
  if (matching.length === 0) {
    return null;
  }
  return matching.reduce((total, account) => total + parseFloat(account.amt), 0);
};

return { ACCOUNTS_PARAM, AMOUNT_PARAM, AMOUNT_COLUMN, state, sumAccountAmount };
```

### `em_etmeta_on_grid_load` — parameter `Accounts` (`obuiapp_parameter`)

```js
(grid, view, parameters) => {
  try {
    const rows = grid.data.allRows;
    rows.forEach((row, index) => {
      const total = sumAccountAmount(row.id);
      if (total !== null) {
        grid.setEditValue(index, AMOUNT_COLUMN, total);
      }
    });
  } catch (error) {
    console.error('Error during matching:', error);
  }
}
```

## Advisories (non-blocking)

- **Internal function names are inverted vs their roles.** `onLoadProcess` is the process `onLoad`;
  `OB.ETVATR.onLoad` is the `Accounts` `onGridLoad`. Trust the legacy columns, not the names.
- **Grid amount column is `amt`, not `amount`.** Classic wrote `item.amount = totalAmount` on the
  selection record, but the grid's displayed/editable Amount field (`AD_FIELD` `2D4F8DA8…`) is backed by
  AD_COLUMN `amt` (the `Accounts` window-reference grid), and is read-only. The migrated `onGridLoad`
  writes through `grid.setEditValue(index, 'amt', total)`, which is the documented edit path for a grid
  cell. If manual QA shows the computed total not reflected in the grid, the column key is the first
  thing to revisit (`amt` vs `amount`).
- **Action `set` moved into `onLoad`.** Classic registered the three actions at module top level; in the
  new UI the module body (`em_etmeta_payscript_logic`) has no `view`, so the registrations live in
  `onLoad` (which receives `view`). The shared `state` and the `sumAccountAmount` helper stay in module
  scope and are referenced by bare name from both `onLoad` and `onGridLoad`.
- **`updateVATTotal` target field.** `getItem('amount')` resolves by dBColumnName `amount` to the
  scalar `Amount` parameter (the dialog total). Confirmed against `obuiapp_parameter` (`columnname =
  amount`).
- **`try/catch` preserved.** The match loop keeps the classic `console.error` on failure.
