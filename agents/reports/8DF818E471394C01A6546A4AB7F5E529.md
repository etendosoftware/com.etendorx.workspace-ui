## In-scope code: `OB.Jobs.ProcessOrders.onLoad(view)` (lines 79–134)

Behavior:
1. Reads the `DocAction` item and the selected records from the launching grid.
2. Builds distinct `documentStatuses` / `documentActions` from the selection.
3. Calls `com.smf.jobs.defaults.ProcessOrdersDefaults` via `OB.RemoteCallManager.call`.
4. In the callback: filters the `DocAction` value map to the server-returned `actions`, applies it with
   `docActionField.setValueMap(newValues)`, then programmatically selects an option —
   `setValueProgrammatically(documentActions[0])` if available, else
   `setValueProgrammatically(getFirstOptionValue())`.
5. If `actions.length > 0` and the OK button is disabled, enables it: `view.okButton.enable()`
   (guarded by `view.okButton && !view.okButton.isEnabled()`).

### Process `em_etmeta_onload` — entity `obuiapp_process` (PASTE)

Faithful port of `OB.Jobs.ProcessOrders.onLoad`. **Signature corrected to two parameters**
`(_process, view) => { … }`: the runtime invokes onLoad as
`executeStringFunction(onLoadScript, scriptHookContext, button.processDefinition, onLoadView)`, and
`executeStringFunction(code, context, ...args)` calls `fn(...args)` — so the first positional argument
is `processDefinition` and the canonical `view` (carrying `theForm`, `selectedRecords`, `tabId`) is the
**second**. The previous single-parameter `(view) =>` bound `view` to `processDefinition`, making
`view.theForm` `undefined` and throwing a TypeError on the first executable line. The first parameter is
unused here, so it is named `_process`.

Remaps the selection / tabId DOM walks to the onLoad `hookData`, `Array.add` → `.push`, and the
`currentValues[action]` object idiom → `currentValues.find(...)` because `getValueMap()` returns a
`ListOption[]` array.

```js
(_process, view) => {
  const form = view.theForm;
  const docActionField = form.getItem('DocAction');
  const selectedRecords = view.selectedRecords || [];
  const tabId = view.tabId;
  const documentStatuses = [];
  const documentActions = [];

  for (let i = 0; i < selectedRecords.length; i++) {
    const record = selectedRecords[i];
    if (!documentStatuses.includes(record.documentStatus)) {
      documentStatuses.push(record.documentStatus);
    }
    if (!documentActions.includes(record.documentAction)) {
      documentActions.push(record.documentAction);
    }
  }

  OB.RemoteCallManager.call(
    'com.smf.jobs.defaults.ProcessOrdersDefaults',
    { documentStatuses: documentStatuses, isProcessing: '', tabId: tabId },
    {},
    (response, data, request) => {
      const actions = data.actions || [];
      const currentValues = docActionField.getValueMap();
      const newValues = {};

      for (let i = 0; i < actions.length; i++) {
        const action = actions[i];
        const match = currentValues.find((option) => option.value === action || option.id === action);
        if (match) {
          newValues[action] = match.label;
        }
      }

      docActionField.setValueMap(newValues);
      // Avoid setting an action that is on the record(s) but not in the dropdown list.
      if (newValues[documentActions[0]]) {
        docActionField.setValueProgrammatically(documentActions[0]);
      } else {
        docActionField.setValueProgrammatically(docActionField.getFirstOptionValue());
      }

      // If the OK button is not enabled but there are actions to select, enable it.
      if (actions.length > 0 && view.okButton && !view.okButton.isEnabled()) {
        view.okButton.enable();
      }
    }
  );
}
```

### `em_etmeta_onprocess` — entity `obuiapp_process`

**RECOMMENDATION: LEAVE EMPTY** (clear the pre-existing wrapper). The classic file has no
`onProcess`/`clientsidevalidation` for ProcessOrders — only `OB.Jobs.ProcessOrders.onLoad`. The
investigation below justifies, against the runtime code (read-only), why clearing the pre-existing
wrapper is safe.

Runtime evidence (canonical sources under `client/packages/MainUI`):
- `components/ProcessModal/hooks/useProcessExecution.ts` `handleExecute` (l.713–732):
  ```js
  if (!etmetaOnprocess && javaClassName) {
    await handleDirectJavaProcessExecute(actionValue);   // STANDARD path
    return;
  }
  if (!etmetaOnprocess || !shouldRunProcessLifecycleHooks({ tab, callerField })) return;
  ```
- `handleDirectJavaProcessExecute` (l.664–672) → `executeJavaProcess(buildStandardJavaPayload(...))`.
- `buildStandardJavaPayload` (l.628–662) and `buildKernelEndpoint` (l.98–110).
- DB for this process: `classname = com.smf.jobs.defaults.ProcessOrders` (so `javaClassName` is set),
  `uipattern = A`.

**1. What does the pre-existing `em_etmeta_onprocess` currently do?**
It is a submit wrapper `async (process, params) => { … }` that:
- reads `recordIds`, `DocAction`, `windowId`, `entityName`, `buttonValue` from `params`;
- builds `?processId=<process.id>&windowId=<windowId>&_action=com.smf.jobs.defaults.ProcessOrders`;
- POSTs `{ recordIds, _buttonValue: params.buttonValue, _params: { DocAction }, _entityName }` via
  `Metadata.kernelClient.post`;
- throws on `!ok` and `return data`.
It hand-builds the exact same standard Java-process submit the framework already performs — it is a
manual re-implementation of the default execution, not client-side validation and not any
ProcessOrders-specific logic.

**2. What does it make work? Is it required, or does the standard submit work without it?**
It is **not required**. When `em_etmeta_onprocess` is empty and `javaClassName` is set, `handleExecute`
takes the standard branch `handleDirectJavaProcessExecute` → `executeJavaProcess(buildStandardJavaPayload())`,
which POSTs to the **same** endpoint (`buildKernelEndpoint` sets `_action` to the configured
`javaClassName` = `com.smf.jobs.defaults.ProcessOrders`) and runs the configured Java class. So with the
column empty, the process still executes through the standard path. The wrapper's second-branch
machinery (`view.executeProcess()`, toast/close handling) is only reached when the column is non-empty;
emptying it routes execution through the dedicated direct-Java path instead.

**3. What is lost by deleting it?**
Nothing that the standard path does not already cover — and the standard path is in fact a **superset**:
- Same endpoint and same `_action` (`com.smf.jobs.defaults.ProcessOrders`), same `recordIds`,
  `_buttonValue`, `_entityName`.
- `_params`: the wrapper sends only `{ DocAction }`; `buildStandardJavaPayload` sends the **full** merged
  form values (which include `DocAction`) plus the button-list param — a strict superset.
- The standard path additionally merges the launching record context (`buildProcessPayload(record, tab, …)`)
  and process-specific PK fields (`buildProcessSpecificFields`), which the hand-built wrapper omits.
- The standard path also dispatches server `responseActions` registry-first and handles
  retry/refresh/toast/close (`executeJavaProcess`), which the bare `return data` wrapper does not.
No ProcessOrders behavior is unique to the wrapper; the classic file defines none.

**4. Why was it added initially? Is it safe to delete?**
It is an **auto-generated / boilerplate submit stub**, not a deliberate ProcessOrders implementation:
the classic `processRecords.js` has no `onProcess`/`clientsidevalidation` for ProcessOrders, so there
was no source behavior to port; its shape (generic `recordIds`/`_buttonValue`/`_params`/`_entityName`
post to `javaClassName`) is exactly the framework's own standard payload, re-expressed by hand. It
duplicates the default execution and, by being present, forces the non-empty branch instead of the
purpose-built direct-Java path — losing the record-context merge and `responseActions` dispatch the
standard path provides. **Conclusion: it is safe (and preferable) to clear it.** Emptying the column is
strictly equal-or-better behavior for this process.
