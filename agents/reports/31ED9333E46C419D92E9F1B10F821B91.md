## status: blocked

The classic file is not Defined-Process dialog JavaScript. It does not populate or read any
`em_etmeta_*` hook (`onLoad` / `onProcess` / `onRefresh` / `onChange` / `onGridLoad`). Its only job is
to register a global **toolbar button** ("Clone") on every window tab via
`OB.ToolbarUtils.createCloneButton` -> `OB.ToolbarRegistry.registerButton`, injected at classic-UI
startup by a `ComponentProvider`. The process appears in the inventory only through **signal 3** (a
hardcoded `processId` inside a `.js`), not through dialog hooks.

The behavior lives in a layer (the window toolbar) that the process-script runtime does not cover, so
it cannot be expressed through the metadata columns. No code was generated, because anything placed in
`em_etmeta_*` would never execute (there is no migrated trigger to fire it) and would give a false sense
of migration.

## What the new UI lacks (to unblock)

1. **Toolbar-button registration API.** The "Clone" button must appear on the toolbar of every window
   tab. The process runtime executes code only inside the process modal; there is no way from
   `em_etmeta_*` to inject a toolbar button into windows. This is exactly the startup-injected
   `ComponentProvider` resource the inventory flags as not replicable in Next.js.
2. **Tab `view` with editable `viewGrid` + `view.editRecord`.** The callback inserts the clone into the
   tab grid cache, selects it, scrolls, and optionally opens it in edit mode. A process hook only gets
   the dialog's `view`, and the grid proxy lacks `addToCacheData`, `scrollToRow`, `doSelectSingleRecord`,
   `editRecord`.
3. **Dialog with custom buttons** (`isc.ask` with `OBUIAPP_CloneButton` / `OBUIAPP_CloneChildrenButton`):
   only plain-text `confirm`/`warn`/`say` are available.
