runs server-side.

## Why it was unblocked (2026-06-11)

The process was previously blocked because the new UI did not inject the global `BigDecimal` and
`OB.Utilities.Number.JSToOBMasked` could not format a `BigDecimal`. Two additive platform changes
lifted that — no `Number` rewrite, so decimal parity is preserved:

1. **Global `BigDecimal` injected into the process-script context.** Shim
   `packages/MainUI/utils/ob/bigDecimal.ts` — an immutable class backed by `bignumber.js@^9.3.1`
   reproducing the exact classic surface used here: `new BigDecimal(String(x))`,
   `BigDecimal.prototype.ZERO`, `.add`, `.subtract`, `.compareTo`, `.toString()`/`.toNumber()`. Wired
   in `buildProcessScriptContext` (`utils/processes/definition/utils.ts:247`) and propagated to all
   hooks via `scriptHookContext`. (`multiply`/`divide` are intentionally omitted — Java
   `BigDecimal.divide` needs an explicit scale + `RoundingMode`, and this process uses neither.)
2. **`OB.Utilities.Number.JSToOBMasked` now formats decimal-like values.** `utils/ob/number.ts` gained
   a `toFiniteNumber()` coercion (duck-typed `toNumber()`), so a `BigDecimal` (or raw `BigNumber`)
   passed as the first argument is formatted correctly; any other input is returned unchanged (classic
   behaviour). The classic `JSToOBMasked(blineAmt, …)` call now works verbatim.

`em_etmeta_onprocess`:

```js
async (process, view) => {
  const HIDE_POPUP_PROPERTY = 'APRM_MATCHSTATEMENT_HIDE_PARTIALMATCH_POPUP';
  const NO_RECORD_LABEL = 'APRM_SELECT_RECORD_ERROR';
  const SPLIT_CONFIRM_LABEL = 'APRM_SplitBankStatementLineConfirm';

  const formatAmount = (amount) =>
    OB.Utilities.Number.JSToOBMasked(
      amount,
      OB.Format.defaultNumericMask,
      OB.Format.defaultDecimalSymbol,
      OB.Format.defaultGroupingSymbol,
      OB.Format.defaultGroupingSize
    );

  const contextInfo = view.getContextInfo();
  const trxParameter = contextInfo && contextInfo.findtransactiontomatch;
  const trxSelection = trxParameter && trxParameter._selection;

  if (!trxSelection || !trxSelection[0]) {
    view.messageBar.setMessage(
      isc.OBMessageBar.TYPE_ERROR,
      null,
      OB.I18N.getLabel(NO_RECORD_LABEL)
    );
    return { severity: 'error', text: OB.I18N.getLabel(NO_RECORD_LABEL) };
  }

  const blineAmt = new BigDecimal(String(view.callerField.record.amount));
  let totalTrxAmt = BigDecimal.prototype.ZERO;
  for (const trx of trxSelection) {
    const trxDepositAmt = new BigDecimal(String(trx.depositAmount));
    const trxPaymentAmt = new BigDecimal(String(trx.paymentAmount));
    totalTrxAmt = totalTrxAmt.add(trxDepositAmt.subtract(trxPaymentAmt));
  }

  const splitRequired = totalTrxAmt.compareTo(blineAmt) !== 0;
  if (!splitRequired) {
    return (await view.executeProcess())?.message;
  }

  const hideSplitConfirmation =
    OB.PropertyStore.get(HIDE_POPUP_PROPERTY, view.windowId) === 'Y';
  if (hideSplitConfirmation) {
    return (await view.executeProcess())?.message;
  }

  const confirmed = await confirm(
    OB.I18N.getLabel(SPLIT_CONFIRM_LABEL, [
      formatAmount(blineAmt),
      formatAmount(totalTrxAmt)
    ])
  );
  if (!confirmed) {
    return { severity: 'error', text: '' };
  }
  return (await view.executeProcess())?.message;
}
```

Compile-check: `new Function("return ("+body+")")` evaluates to `typeof === "function"` → OK.

## Advisories (non-blocking)

- **Dropped test-mode branch.** The classic `if (isc.isA.emptyObject(OB.TestRegistry.registry)) { isc.confirm(...) } else { execute(true) }` is dead in production (the registry is always empty, so the `if` is always true). Only the `confirm(...)` path is ported; `isc.isA` / `OB.TestRegistry.registry` are unshimmed by design.
- **`actionHandlerCall()` → `await view.executeProcess()`**; the return is `(await view.executeProcess())?.message` so the standard onProcess flow surfaces the server message exactly once.
- **Cancel branch semantics.** Classic `clientSideValidationFail()` on confirm-cancel simply aborts with no new message. It is mapped to `return { severity: 'error', text: '' }` (empty text) to abort the submit without surfacing a misleading error. The no-selection case keeps the real `APRM_SELECT_RECORD_ERROR` message both in the message bar and the return.
- **Selection guard simplified.** The classic `typeof view.getContextInfo === 'function' && view.callerField.view...` capability checks are unnecessary in the new UI (these accessors are always live on the view proxy); the migrated body keeps the behaviorally meaningful selection check only.

## Launch restriction — not manually testable in this environment

This process is **not exercisable through the UI in the current environment**, so the manual-test
checklist below was **not run**. The migrated code stands on static verification only. This is a
launcher/data gap, not a code defect.

Two independent blockers, both confirmed against `etendodev`:

1. **The launch field is permanently hidden.** The process column `EM_Aprm_Findtransactionspd`
   (`ad_column_id 394FD1143C904A8C9E6A2AD42085CB9B`) is bound to exactly one field,
   `ad_field FC6A539FAE444706A27E013B9D62D10F` on the **Financial Account → Account** tab, whose
   `displaylogic` is the literal constant **`false`** (`isdisplayed = Y`). A display logic of `false`
   never evaluates true, so the button never renders. This is the classic SmartClient pattern: "Find
   Transactions to Match" was never launched from a visible form button — it is invoked
   **programmatically from the bank-statement reconciliation flow ("Match Statement" / Add
   Transaction)**, where the reconciliation grid sets `view.callerField` to the **bank-statement line**
   being reconciled (the source of `callerField.record.amount`). The hidden column is only a metadata
   anchor for the process.

2. **No data to drive it.** There are **zero unmatched bank-statement lines anywhere in the database**
   (all 141 statements, all in the F&B International Group client, are fully reconciled; the QA Testing
   client — the only client the QA Admin role can see — has **no bank statements at all**). The caller
   record (an unmatched statement line carrying `amount`) therefore does not exist.

### What would be required to test it

- **A launcher in the new UI** that opens this process with `view.callerField` pointing at a
  bank-statement line — i.e. the new UI's equivalent of the classic "Match Statement" reconciliation
  surface. Whether the new UI implements this surface is **unconfirmed**; if it does not, the migrated
  `em_etmeta_onprocess` has no caller regardless of the field's display logic.
- **Test data:** a bank statement with at least one **unmatched line** (the caller), on a financial
  account that also has **unmatched financial-account transactions** (the grid candidates). In QA
  Testing, the account `Accounting Documents DOLLAR` (org USA, USD) has 9 unmatched transactions
  available for the grid, but a statement + unmatched line would have to be created first.

## Manual-test checklist

> **Not executed in this environment** (see [§ Launch restriction](#launch-restriction--not-manually-testable-in-this-environment)).
> Retained as the QA plan for when a reconciliation launcher and statement-line data become available.

1. **Open the process** "Find Transactions to Match" from a bank-statement line (the launching record must carry an `amount`). The modal opens with the transactions grid.
2. **No selection → error.** Without selecting any transaction, press Done/OK. Expect: an error banner in the dialog reading the `APRM_SELECT_RECORD_ERROR` label; the dialog stays open; no server call.
3. **Exact match → direct execute.** Select transaction(s) whose Σ(deposit − payment) equals the line amount exactly. Press Done. Expect: no split confirmation; the process executes server-side and the result message is shown once.
4. **Partial match → split confirm (popup enabled).** With the preference `APRM_MATCHSTATEMENT_HIDE_PARTIALMATCH_POPUP` **not** set to `Y`, select transaction(s) whose total differs from the line amount. Press Done. Expect: a confirmation dialog whose text shows the line amount and the transactions total, both **formatted with the locale's decimal/grouping symbols** (verify `BigDecimal` amounts render correctly, not as `[object Object]`).
   - **Confirm (OK)** → the process executes and shows its result.
   - **Cancel** → the submit aborts, the dialog stays open, no error message is shown.
5. **Partial match → popup suppressed.** Set the preference `APRM_MATCHSTATEMENT_HIDE_PARTIALMATCH_POPUP = Y`. Repeat step 4's partial selection and press Done. Expect: no confirmation dialog; the process executes directly.
6. **Decimal parity.** Use amounts with fractional cents (e.g. line 100.05, transactions 100.04) so the comparison hinges on decimal precision. Confirm the split decision matches Classic (it must treat 100.05 vs 100.04 as a split, i.e. `BigDecimal` parity, not lossy float rounding).
