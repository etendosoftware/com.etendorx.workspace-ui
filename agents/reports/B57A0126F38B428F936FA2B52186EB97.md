## Generated code per field

### `em_etmeta_on_parameter_change` â€” parameter "Emisor de Factura" (`AD_Org_ID`)
```js
(item, view, form, grid) => {
  const ORG_FIELD = 'AD_Org_ID';
  const ACTION_HANDLER =
    'com.etendoerp.verifactu.action_handler.AddOrganizationInfoActionHandler';

  const orgItem = form.getItem(ORG_FIELD);
  const organization = orgItem ? orgItem.getValue() : '';

  if (organization === '') {
    return;
  }

  const callback = (response, data) => {
    form.getItem('LegalName').setValue(data.legalName);
    form.getItem('NIF').setValue(data.nif);
    form.redraw();
  };

  OB.RemoteCallManager.call(ACTION_HANDLER, { organization: organization }, {}, callback);
}
```

## Advisories (non-blocking)
- The classic global namespace `OB.ETVFAC` is not needed: handlers are not cross-referenced, so no module scope is registered.
- The message id `ETVFAC_DATA_ID_ISEMPTY` was not found in `ad_message` in `etendodev` (module not checked out as source â€” `âš  deploy`). `OB.I18N.getLabel` falls back to the key, matching classic behavior. Confirm the label is installed in the target environment so the user sees translated text.
- The classic `recipientID` handler shows a warning banner whenever the field is non-empty (the label literally says "is empty" but the guard fires on non-empty). The migration reproduces this exact behavior verbatim; it is a pre-existing classic quirk, not introduced here.
- `view.messageBar.setMessage(...)` is only meaningful inside the process dialog; the banner shows in the in-dialog message bar.
