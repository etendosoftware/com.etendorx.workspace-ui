## Generated code per field

### `obuiapp_parameter` → **General Accounting Report** → `em_etmeta_on_parameter_change`

```js
(item, view, form) => {
  const SHOW_DATES_HANDLER =
    "com.etendoerp.financial.reports.advanced.handler.ShowDatesField";
  const balanceFields = ["CurrentDate", "AsOfReferenceDate", "AsOfReferenceDate2", "AsOfReferenceDate3"];
  const rangeFields = [
    "FromRefDate",
    "ToRefDate",
    "DateFrom",
    "DateTo",
    "FromRefDate2",
    "ToRefDate2",
    "FromRefDate3",
    "ToRefDate3",
  ];

  const setVisibility = (fieldNames, visible) => {
    fieldNames.forEach((fieldName) => {
      const field = form.getItem(fieldName);
      if (field) {
        if (visible) {
          field.show();
        } else {
          field.hide();
        }
      }
    });
  };

  const balanceValue = form.getItem("C_Acct_Rpt_ID").getValue();

  const hideFields = (response, data) => {
    if (data.balance) {
      setVisibility(rangeFields, false);
      setVisibility(balanceFields, true);
    } else {
      setVisibility(balanceFields, false);
      setVisibility(rangeFields, true);
    }
  };

  OB.RemoteCallManager.call(SHOW_DATES_HANDLER, { balanceValue: balanceValue }, {}, hideFields);
}
```

Compile-check: evaluates to a function, arity 3 — OK.

## Advisories (non-blocking)

1. **Dead code dropped.** Three ETFRA functions (`organizationOnChange`, `showHideDimensions`,
   `showDocumentNoField`) live in the shared family but are not bound to this process; they were not
   migrated. They belong to the sibling ETFRA reports — migrate them only when those processes are
   in scope, per the cluster cloning rule (Section 10.5).
2. **Refactor vs. line-for-line.** The classic callback uses one big if/else that calls `.hide()`/
   `.show()` on each field individually. The migrated body keeps identical behavior but factors the
   field lists into two arrays plus a `setVisibility` helper to avoid repetition and reduce cognitive
   complexity. The visible/hidden sets and the `data.balance` branch logic are unchanged.
3. **Null-guard added.** The classic code assumes every `getItem(...)` returns a widget. The migrated
   body guards each lookup (`if (field)`), so a missing/renamed field degrades gracefully instead of
   throwing inside the callback. Functionally equivalent for the present field set.
4. **Backend handler dependency.** The flow calls the Java handler
   `com.etendoerp.financial.reports.advanced.handler.ShowDatesField` (via `OB.RemoteCallManager.call`
   → `callAction`), exactly as Classic does. The compiled handler class was not found in the local
   source checkout (deployed module only). QA requires the module deployed and the handler returning
   `{ balance: <boolean> }`. This is unchanged backend behavior, not a migration gap.
5. **No global namespace bootstrap needed.** The classic files start with `OB.ETFRA = {}` /
   `OB.ETFRA.OnChange = {}` to register the namespace. In the new UI each hook owns its scope, so no
   namespace self-registration is required for this single self-contained `onChange`.

## Manual-test checklist

Run in the new UI, on the **Balance Sheet and P&L Structure advanced** process dialog.

1. **Open the dialog.** Confirm it opens with the standard parameter form (no custom component).
2. **Select a "balance"-type General Accounting Report** (one for which `ShowDatesField` returns
   `balance = true`). Expect: `As Of Date`, `As Of Reference Date`, `As Of Reference Date 2`,
   `As Of Reference Date 3` become **visible**; `From Reference Date`, `To Reference Date`,
   `Starting Date`, `Ending Date`, `From Reference Date 2`, `To Reference Date 2`,
   `From Reference Date 3`, `To Reference Date 3` become **hidden**.
3. **Select a non-balance General Accounting Report** (`balance = false`). Expect the inverse: the
   four "As Of …" fields **hidden**; the eight range/reference-date fields **visible**.
4. **Toggle back and forth** between a balance and a non-balance report a few times. Expect the
   visibility to switch correctly each time with no stale/leftover fields.
5. **Network check (optional).** On each change, confirm a single POST to the `ShowDatesField`
   handler with `{ balanceValue: <selected report id> }` and that the UI reacts to the returned
   `balance` flag.
6. **No console errors** during any of the above.
