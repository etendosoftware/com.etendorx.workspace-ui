### 0.1 The shared root cause of #1 and #4 — `onLoad`/validation run before the async seed

Classic applies all process defaults / the FIC callout **synchronously before** `onLoad` runs, so by the
time `onLoad` and any field validation execute, every field already has its value. The new UI seeds
defaults **asynchronously, after** `onLoad`. Evidence in the fresh `logs.txt`:

- `[AP onLoad] ENTRY; payment= undefined issotrx= false trxtype= undefined` with **all context values
  `undefined`** (`logs.txt:1-2`), and `[AP onLoad] EXIT` still all-undefined (`:5-6`). `onLoad` ran on an
  unseeded form.
- `onLoad` dispatches `paymentMethodMulticurrency` (`:3`); its handler is called with incomplete context
  and the imperative show/hide is skipped (cf. `reloadLabels` logging the same null-skip, `:8`). The later
  cascade calls that previously performed the hide are now (correctly) suppressed by F1 → **#4: the 3
  multicurrency fields stay visible.**
- `payment_date` is **mandatory** (DB `ismandatory=Y`) and gets its default a tick later, but the
  required-field validation runs on the unseeded mount → **#1: "payment date is mandatory" on open.**

Neither is JS-fixable from inside the migrated hooks: there is no migration entry point that runs *after*
the async seed completes with complete context, and `onLoad` has no defer primitive (`fireOnPause` is a
grid method). The substrate must either run `onLoad`/validation post-seed or expose a post-seed hook.
See §0.3.

### 0.3 Substrate changes required to unblock (#1, #4, #7, new-#8)

1. **Run `onLoad` (and mandatory field validation) AFTER the async default/FIC seed — or expose a
   post-seed hook.** The migration's `onLoad` and field-level mandatory validation currently execute on an
   unseeded form (`logs.txt:1-6` show all context `undefined` at `onLoad` entry/exit). Classic seeds
   synchronously before `onLoad`. Either (a) defer `onLoad` until defaults/FIC have been applied,
   (b) provide a documented post-seed lifecycle hook the migration can target, and (c) do not raise
   mandatory-field errors before defaults are applied / before user interaction. Unblocks **#1** and gives
   **#4** a complete-context entry point.
2. **(Consequence of #1) the imperative multicurrency hide will then work.** Once `paymentMethodMulticurrency`
   runs post-seed with complete context it hides `conversion_rate` / `converted_amount` / `c_currency_to_id`
   as in Classic. **No declarative alternative exists** — these 3 params have NULL `displaylogic` in the
   DB. If a post-seed hook is provided, the migration calls `paymentMethodMulticurrency` from it
   (flash-then-hide is acceptable per QA); if `onLoad` itself is deferred post-seed, the existing `onLoad`
   call suffices and no JS change is needed.

> **Why no JS-only workaround is applied for #4.** A provisional JS hide could be triggered from a grid's
> `afterGridSettle` (which runs post-seed), but re-running `paymentMethodMulticurrency` (which calls
> `PaymentMethodMulticurrencyActionHandler`, mutates valueMaps and drives the `_mc*` state machine) from
> an unrelated grid-load is a **possible regression**, and the migration contract says to stop and report
> rather than risk one. The clean fix is the post-seed lifecycle above.

3. **[APPLIED — Fix C, 2026-06-18] Fix `rejectByColumnValidator` to validate the POST-write value, not
   the pre-write cell value (and stop dropping the write before persistence).** Implemented and re-verified
   in the working tree: `buildValidatorCandidate(rowData, mergedChanges)` returns `{ ...rowData,
   ...mergedChanges }` (`WindowReferenceGrid.tsx:1063`); `rejectByColumnValidator` passes that candidate as
   the validator `record` and `candidate[colName]` as the value (`:2866`, `:2871-2872`). This is the
   **single root cause** of #3, #4, and #1, pinpointed by the 5th QA run (it supersedes the earlier "persist `setEditValue`", "display-memo", and
   "seed-from-`obSelected`" hypotheses — `obSelectedCount= 0` ruled out the row-flag path, and `distribute`
   was shown to write the correct `amount= 2.07` which is then dropped).

   **Mechanism.** The migration registers cell validators via `setColumnValidator('amount', …)` /
   `setColumnValidator('paymentAmount', …)` (§4.5). On every write, `handleRecordChange` runs
   `rejectByColumnValidator` **before** Fix B's `applyEditToReadStore` and **returns early on reject**
   (`WindowReferenceGrid.tsx:2936-2938`). But `rejectByColumnValidator` invokes the validator with the
   **current** cell value — `rowData[colName]` where `rowData = row.original ?? row` (`:2856`), and
   `item.getValue` returns the same (`:2832`) — **not** the new `mergedChanges[colName]` being written. Grid
   rows start at `amount = 0`, so the faithful validator `AP.orderInvoiceGridValidation` (§4.1 `:1110-1113`)
   takes `if (paidamount.signum() === 0 && record.writeoff === false) → setMessage(APRM_JSZEROUNDERPAYMENT)
   → return false`. `rejectByColumnValidator` returns `true`, `handleRecordChange` returns early, and the
   real write (2.07) never reaches persistence → `readback= 0` (`logs.txt:13,40-41`). Because the row is
   stuck at 0, the next write re-evaluates 0 and is rejected again — the amount can never be set.

   **Why this is all three problems:** #3 (Order/Invoice `Amount`=0) and #4 (Credit `Payment Amount`=0) are
   the dropped writes; **#1's "A zero amount Payment cannot be left as an underpayment" is literally the
   `APRM_JSZEROUNDERPAYMENT` message** this validator raises while wrongly evaluating the pre-write 0.

   **Fix (substrate, minimal and precise).** In `rejectByColumnValidator`, evaluate the **candidate
   post-write record** — `const candidate = { ...rowData, ...mergedChanges };` — pass `candidate` as the
   validator's `record` arg and `mergedChanges[colName]` as its `value` arg, and set `item.getValue = () =>
   mergedChanges[colName]`. Then the validator sees `amount = 2.07` with `outstanding = 2.07` → `2.07 ≤
   2.07` and `signum ≠ 0` → returns `true` → not rejected → Fix B persists the write → `readback = 2.07`,
   the cell displays it, `expected/actual` reconcile, and the underpayment message disappears. (Classic did
   not run the cell validator on programmatic distribute writes at all; validating the new value is the
   minimal correct change and is also correct for interactive edits, which today validate the stale value.)

   **The §4 JS and the validator body are faithful and unchanged — this is a substrate fix only.** No
   JS/babel change is required for #1/#3/#4. **Full hand-off for the substrate team (analysis, evidence,
   exact before/after fix, assumptions, test guidance):**
   `9BED7889E1034FE68BD85D5D16857320-substrate-brief.md` (companion file next to this report).

### 0.6.1 Updates — 2026-06-18 (Babel: read-back logging added to `AP.distributeAmount`)

Per the §0.6 next-step, **`AP.distributeAmount` (§4.1) now carries `AP_DEBUG`-gated read-back logging**.
Purely additive: no calculation, branch, or selection logic was touched, and the double-fire is left as-is
(analysed separately). The change reuses the existing `log` helper. Module body re-compile-checked (parses
and still ends in `return { ... }`). Status remains **`blocked`** — not advanced.

- **Entry log** (right after `autoDistributeAmt` is read, before the `autoDistributeAmt !== 'N'` guard):
  `[AP distributeAmount] entry; actual_payment= … autoDistributeAmt= … rowCount= … onActualPaymentChange= … payment= …`.
  Captures the decision context. If `autoDistributeAmt` is `'N'` / `'"N"'`, the whole distribution block is
  skipped and every editable `amount` stays at its prior value (0) — this would be the direct cause and is
  now visible without entering the loop.
- **Per-write read-back logs** — one after **every** `orderInvoice.setEditValue(i, 'amount', …)` in the
  loop, covering all five write sites (both select-and-write and deselect-and-zero):
  `[AP distributeAmount] row i= … wrote amount= … readback= <getEditValues(i).amount> … outstanding= … selected= <true/false>`.
  - `amount.signum()===0` → outstanding<0 & (onActualPaymentChange||payment): writes `outstandingAmount`, `selected=true`.
  - `amount.signum()===0` → else: writes `0`, `selected=false`.
  - `amount.signum()===1`: writes `outstandingAmount`, `selected=true`.
  - `outstandingAmount.signum()<0` branch: writes `outstandingAmount`, `selected=true`.
  - final `else`: writes `0`, `selected=false`.

If the read-back shows the value the branch wrote but QA still sees `Amount=0`, the substrate re-zeroes the
editable amount after distribute (escalate #7 to a substrate item). If the read-back is already `0`, the
zero originates in the distribute logic (likely `autoDistributeAmt='N'`, a 0 `actual_payment`, or the
deselect/zero branch being taken). **Re-paste only the module body (`em_etmeta_payscript_logic`) and re-run QA.**

### 0.6.2 Runtime QA 2026-06-18 (3rd `logs.txt`) — #7 CONFIRMED SUBSTRATE (`setEditValue` on `amount` is dropped)

The read-back run is decisive. With `actual_payment= 2.07`, `autoDistributeAmt= undefined` (so distribution
runs), `rowCount= 10`:

```
[AP distributeAmount] entry; actual_payment= 2.07 autoDistributeAmt= undefined rowCount= 10 onActualPaymentChange= false payment= undefined   (logs.txt:19)
[AP distributeAmount] row i= 0 wrote amount= 2.07 readback= 0 outstanding= 2.07 selected= true                                                (logs.txt:20)
[AP distributeAmount] row i= 1..9 wrote amount= 0 readback= 0 …                                                                               (logs.txt:21-29)
```

- **The distribute logic is CORRECT and faithful.** It picked row 0 — the only row whose `outstanding`
  (2.07) matches the payment — selected it, and **wrote `amount= 2.07`** to it; all other rows correctly
  got 0. This is exactly classic `distributeAmount` (`:719-724`).
- **The substrate DROPS the write.** The **immediate** `getEditValues(0).amount` read-back is **`0`**
  (same synchronous tick, right after `setEditValue(0,'amount',2.07)` + `selectRecord(0)`), and the UI
  shows 0. So `setEditValue` on the editable **`amount`** column of the Window-Reference grid does **not**
  reflect into the grid's read/display store via the script proxy. This is a **substrate** defect, not a
  JS one.
- **This blocks BOTH paths.** The seed path (#7 pre-selection, new-#8 credit `paymentAmount`) writes the
  amount through the **same** `setEditValue` call, so it would hit the **same** drop. Therefore #7 and
  new-#8 are now **substrate-blocked**, not JS-fixable — the §4 JS is correct and writes the right value;
  the platform must persist `setEditValue` on the editable amount column.
- **Related substrate history:** this is the same family as the known "substrate resets `amount` column in
  WindowRef grids" work (memory `substrate-resets-amount-column-windowref`, ETP-3748): that change claimed
  to fix read-only-amount zeroing and a stale cell-edit read (`applyEditToRows` + sync
  `gridApiRef.current.rows` before `fireCellEditHooks`), but the **script-proxy `setEditValue` write path
  for the editable `amount`** is still not reflected here (wrote 2.07 → read 0). Either that fix is not in
  the running build, or it does not cover the script-proxy `setEditValue` path.

**Answer to "pre-selected vs auto-distribute?" (per Classic):** for this scenario — a standalone Add
Payment open with `issotrx=true` (sales) / no existing payment and **no rows pre-selected** — Classic
`orderInvoiceOnLoadGrid` (`:472-473`) takes the `(issotrx || !payment) && selectedIds.length === 0` branch
and **auto-distributes**. It does NOT require a pre-selection. So the new UI's `BRANCH=distribute` is
**correct**. (Classic's *pre-selection* path — `ordInvDataArrived`/`setEditValues` — only applies when Add
Payment is launched from a context that pre-populates `selectedIds`, e.g. from a specific invoice; that is
a different launch context and is the one the §4 seed bodies cover.) Either way, both end states are
written via `setEditValue('amount', …)`, which the substrate currently drops → the active blocker is the
same.

## 1. Summary / TL;DR

Add Payment is **the documented legacy exception** of the whole migration program
(`new-javascript-code.md` §11). It is *not* on the generic, metadata-driven hook path that Babel
migrates to. Today it is supported by **four UI hardcodings** plus a **PayScript DSL** stored in
`em_etmeta_payscript_logic` (the `AddPaymentRulesGeneric` rules object). The classic file's behaviour is
**dominated by six server `OB.RemoteCallManager.call` action-handler round-trips** that are wired into
the per-parameter `onChange` handlers; the existing DSL reproduces only the **client-side arithmetic**
(distribution + totals), not those server round-trips, the label/value-map mutations they perform, the
multicurrency field show/hide, or the per-trxtype grid refetch.

A faithful, generic migration to the standard hook columns (`em_etmeta_onload`,
`em_etmeta_onprocess`, `em_etmeta_on_parameter_change` ×13, `em_etmeta_on_grid_load` ×3) is **not safe to
produce right now** for two compounding reasons:

1. **Active collision with the legacy exception.** While the four Add-Payment hardcodings remain live
   (`PROCESS_DEFINITION_DATA`, the `isAddPayment` branch in `useProcessPayload.ts`,
   `SINGLE_RECORD_ONLY_PROCESSES`, and the `PROCESS_CALLOUTS` registry whose **only** entry is Add
   Payment), pasting standard per-parameter `onChange`/`onProcess` hooks would run **in parallel** with
   the existing callout + DSL path and the hardcoded payload override. The doc states these hardcodings
   are to be *folded into the generic mechanism "when Add Payment is migrated"* — i.e. that cleanup is a
   precondition of this migration, not a consequence of it. Babel does not touch application source code.
2. **No trusted reference.** No process in the inventory is `qa-passed`; every Add-Payment-family
   sibling (Add Transaction, Match Statement, Add Invoices, RFC/RTV) is only `migrated` (pending QA), so
   per Babel's contract none can be used as a trusted few-shot reference for the most complex file in the
   whole inventory.

Conservative gate (Phase 4): because the coverage analysis below contains **unsupported** entries (the
parallel-path collision is a platform-level gap specific to this process, and the existing `onload` /
DSL are partial and non-standard), the migration is **blocked**. No standard hook code is generated.

### 4.1 `em_etmeta_payscript_logic` (process) — generic module body

Holds the whole `OB.APRM.AddPayment` helper namespace (self-registered) plus the shared money-math
helpers, and exposes them by bare name to every hook. Replaces the cleared legacy DSL.

```js
const ZERO = () => BigDecimal.prototype.ZERO;
const bd = (v) => new BigDecimal(String(v === undefined || v === null || v === '' ? 0 : v));
const isNum = (v) => v !== null && v !== undefined && v !== '' && !Number.isNaN(Number(v));
const num = (v) => Number(v);

const GRID_SETTLE_ID = 'aprmGridSettle';
const GRID_SETTLE_DELAY = 0;

const MC_HANDLER = 'org.openbravo.advpaymentmngt.actionHandler.PaymentMethodMulticurrencyActionHandler';
const DOCNO_HANDLER = 'org.openbravo.advpaymentmngt.actionHandler.AddPaymentDocumentNoActionHandler';
const ORG_HANDLER = 'org.openbravo.advpaymentmngt.actionHandler.AddPaymentOrganizationActionHandler';
const RECEIVEDFROM_HANDLER = 'org.openbravo.advpaymentmngt.actionHandler.ReceivedFromPaymentMethodActionHandler';
const DISPLAYLOGIC_HANDLER = 'org.openbravo.advpaymentmngt.actionHandler.AddPaymentDisplayLogicActionHandler';
const LABELS_HANDLER = 'org.openbravo.advpaymentmngt.actionHandler.AddPaymentReloadLabelsActionHandler';
const ONPROCESS_HANDLER = 'org.openbravo.advpaymentmngt.actionHandler.AddPaymentOnProcessActionHandler';

const AP = {};

AP.trxtype = (form) => (form.getItem('trxtype') ? form.getItem('trxtype').getValue() : '');
AP.grid = (form, param) => form.getItem(param).canvas.viewGrid;

AP.isMultiCurrency = (form) => {
  const fromId = form.getItem('c_currency_id') ? form.getItem('c_currency_id').getValue() : undefined;
  const toId = form.getItem('c_currency_to_id') ? form.getItem('c_currency_to_id').getValue() : undefined;
  return !!fromId && !!toId && fromId !== toId;
};

AP.hasBslAmount = (form) => {
  const item = form.getItem('bslamount');
  if (!item) return false;
  const val = item.getValue();
  if (val === null || val === undefined || val === '') return false;
  return bd(val).compareTo(ZERO()) !== 0;
};

AP.getConvertedAmount = (form, amount, directConversion) => {
  const currencyPrecision = form.getItem('StdPrecision').getValue();
  const exchangeRate = bd(form.getItem('conversion_rate').getValue() || 1);
  if (directConversion) {
    return amount.multiply(exchangeRate).setScale(currencyPrecision, BigDecimal.prototype.ROUND_HALF_UP);
  }
  return amount.divide(exchangeRate, currencyPrecision, BigDecimal.prototype.ROUND_HALF_UP);
};

AP.updateConvertedAmount = (view, form, recalcExchangeRate) => {
  const actualPayment = bd(form.getItem('actual_payment').getValue() || 0);
  const actualConvertedItem = form.getItem('converted_amount');
  const exchangeRateItem = form.getItem('conversion_rate');
  let newConvertedAmount = ZERO();
  let newExchangeRate = BigDecimal.prototype.ONE;
  const exchangeRate = bd(exchangeRateItem.getValue() || 1);
  const actualConverted = bd(actualConvertedItem.getValue() || 0);
  if (!actualConverted || !exchangeRate) return;
  if (recalcExchangeRate) {
    if (actualConverted && actualPayment) {
      if (actualPayment.compareTo(newConvertedAmount) !== 0) {
        newExchangeRate = actualConverted.divide(actualPayment, 15, 2);
        exchangeRateItem.setValue(num(newExchangeRate.toString()));
      }
    } else {
      exchangeRateItem.setValue(num(newExchangeRate.toString()));
    }
  } else if (exchangeRate) {
    newConvertedAmount = AP.getConvertedAmount(form, actualPayment, true);
    exchangeRateItem.setValue(num(exchangeRate.toString()));
    actualConvertedItem.setValue(num(newConvertedAmount.toString()));
  } else {
    actualConvertedItem.setValue(num(actualConverted.toString()));
  }
};

AP.applyBankAmountToConverted = (form) => {
  if (!AP.hasBslAmount(form)) return;
  const bsl = bd(form.getItem('bslamount').getValue() || 0);
  const convItem = form.getItem('converted_amount');
  if (!convItem) return;
  if (bsl.compareTo(ZERO()) === 0) return;
  form._bslApplying = true;
  convItem.setValue(num(bsl.abs().toString()));
  const act = bd(form.getItem('actual_payment').getValue() || 0);
  if (act.compareTo(ZERO()) > 0) {
    AP.updateConvertedAmount(null, form, true);
    form._pendingRecalcRateFromBSL = false;
  } else {
    form._pendingRecalcRateFromBSL = true;
  }
  form._bslApplying = false;
};

AP.syncActualWithExpectedIfMC = (form) => {
  if (!AP.isMultiCurrency(form)) return;
  const expected = num(form.getItem('expected_payment').getValue() || 0);
  form._mcAutoSync = true;
  form.getItem('actual_payment').setValue(expected);
  form._mcAutoSync = false;
  form._mcActualTouched = false;
  if (form.getItem('issotrx').getValue()) {
    AP.distributeAmount(null, form, true);
  }
  AP.updateDifference(form);
  AP.updateConvertedAmount(null, form, false);
  AP.applyBankAmountToConverted(form);
};

AP.recalcDisplayLogicOrReadOnlyLogic = (form, view, affectedParams) => {
  const params = {};
  const thisform = form;
  const thisview = view;
  const creditUseGrid = AP.grid(form, 'credit_to_use');
  params.context = view.getContextInfo();
  delete params.context.order_invoice;
  delete params.context.credit_to_use;
  delete params.context.glitem;

  const callback = (response, data) => {
    if (!data || data.values === null || data.values === undefined) {
      return;
    }
    const values = data.values;
    for (const i in values) {
      if (Object.prototype.hasOwnProperty.call(values, i)) {
        const def = values[i];
        const field = thisform.getItem(i);
        if (field) {
          if (def && typeof def === 'object') {
            if (def.identifier && def.value) {
              field.setValueFromRecord({ id: def.value, _identifier: def.identifier });
            }
          } else {
            field.setValue(def);
          }
        }
      }
    }
    if (thisview && thisview.handleReadOnlyLogic) thisview.handleReadOnlyLogic();
    if (values.credit_to_use_display_logic && values.credit_to_use_display_logic === 'Y') {
      creditUseGrid.fetchData();
    }
    if (thisview && thisview.handleButtonsStatus) thisview.handleButtonsStatus();
  };

  view.fireOnPause('recalcDisplayLogicOrReadOnlyLogic' + affectedParams, () => {
    OB.RemoteCallManager.call(DISPLAYLOGIC_HANDLER, { affectedParams, params }, {}, callback);
  }, 200);
};

AP.reloadLabels = (form) => {
  const params = {

    financialAccount: 'fin_financial_account_id',
    issotrx: form.getItem('issotrx').getValue()
  };
  const callback = (response, data) => {
    if (!data || data.values === null || data.values === undefined) {
      return;
    }
    form.getItem('received_from').setTitle(data.values.businessPartner);
    form.getItem('fin_financial_account_id').setTitle(data.values.financialAccount);
  };

  let totalAmount = ZERO();
  const actualPayment = form.getItem('actual_payment');
  const expectedPayment = form.getItem('expected_payment');
  const generateCredit = bd(form.getItem('generateCredit').getValue() || 0);
  const glitemtotal = bd(form.getItem('amount_gl_items').getValue() || 0);
  const credit = bd(form.getItem('used_credit').getValue() || 0);
  const bslamount = bd(form.getItem('bslamount').getValue() || 0);
  const selectedRecords = orderInvoice.getSelectedRecords();
  for (const rec of selectedRecords) {
    totalAmountoutstanding = totalAmountoutstanding.add(bd(orderInvoice.getEditedCell(rec, 'outstandingAmount')));
    totalAmount = totalAmount.add(bd(orderInvoice.getEditedCell(rec, 'amount')));
  }
  if (selectedRecords.length > 0) {
    expectedPayment.setValue(num(totalAmountoutstanding.toString()));
  } else {
    expectedPayment.setValue(num('0'));
  }
  if (!issotrx) {
    let actpayment = totalAmount.add(glitemtotal).add(generateCredit);
    actualPayment.setValue(num(actpayment.toString()));
    if (credit.compareTo(ZERO()) > 0) {
      if (credit.compareTo(actpayment) > 0) {
        actualPayment.setValue(num('0'));
      } else {
        actualPayment.setValue(num(actpayment.subtract(credit).toString()));
      }
    }

      const bslAmountConverted = AP.getConvertedAmount(form, bslamount, false);
      if (actpayment.compareTo(bslAmountConverted.abs()) <= 0) {
        actpayment = bslAmountConverted.abs();
        actualPayment.setValue(num(actpayment.toString()));
      }
    }
    AP.updateDifference(form, view);
    AP.updateConvertedAmount(null, form, false);
  }
  if (form._mcSyncInProgress) {
    // no-op while a multicurrency sync is already running
  } else if (form._mcPendingSyncMC && AP.isMultiCurrency(form)) {
    const expNow = num(form.getItem('expected_payment').getValue() || 0);
    const actNow = num(form.getItem('actual_payment').getValue() || 0);
    if (expNow > 0 && (actNow === 0 || !form._mcActualTouched)) {
      form._mcPendingSyncMC = false;
      form._mcSyncInProgress = true;
      AP.syncActualWithExpectedIfMC(form);
      form._mcSyncInProgress = false;
      form._mcSyncedOnce = true;
    }
  }
  AP.applyBankAmountToConverted(form);

AP.distributeAmount = (view, form, onActualPaymentChange) => {
  let amount = bd(form.getItem('actual_payment').getValue() || 0);
  const orderInvoice = AP.grid(form, 'order_invoice');
  const issotrx = form.getItem('issotrx').getValue();
  const payment = form.getItem('fin_payment_id').getValue();
  let negativeamt = ZERO();
  let differenceamt = ZERO();
  const creditamt = bd(form.getItem('used_credit').getValue() || 0);
  const glitemamt = bd(form.getItem('amount_gl_items').getValue() || 0);
  const orderInvoiceData = (orderInvoice.data && orderInvoice.data.localData) ? orderInvoice.data.localData : [];
  const total = orderInvoiceData.length;
  const autoDistributeAmt = OB.PropertyStore.get('APRM_AutoDistributeAmt');

  if (autoDistributeAmt !== 'N' && autoDistributeAmt !== '"N"') {
    amount = amount.subtract(glitemamt);
    amount = amount.add(creditamt);
    if (issotrx && AP.isMultiCurrency(form)) {
      if (amount.compareTo(ZERO()) === 0) {
        const exp = bd(form.getItem('expected_payment').getValue() || 0);
        if (exp.compareTo(ZERO()) > 0) amount = exp;
      }
    }
    for (let i = 0; i < total; i++) {
      const row = orderInvoiceData[i];
      if (row && typeof row === 'object' && Object.keys(row).length > 0) {
        const outstandingAmount = bd(row.outstandingAmount);
        if (outstandingAmount.signum() < 0) {
          negativeamt = negativeamt.add(outstandingAmount.abs());
        }
      }
    }
    if (amount.compareTo(negativeamt.negate()) > 0 && (onActualPaymentChange || payment)) {
      amount = amount.add(negativeamt);
    }
    for (let i = 0; i < total; i++) {
      const row = orderInvoiceData[i];
      if (!row || typeof row !== 'object' || Object.keys(row).length === 0) continue;
      const editValues = orderInvoice.getEditValues(i);
      let writeoff = editValues ? editValues.writeoff : undefined;
      let amt = bd((editValues ? editValues.amount : 0) || 0);
      if (writeoff === null || writeoff === undefined) {
        writeoff = orderInvoice.getRecord(i).writeoff;
        amt = bd(orderInvoice.getRecord(i).amount || 0);
      }
      if (writeoff && issotrx) {
        amount = amount.subtract(amt);
        continue;
      }
      let outstandingAmount = bd(orderInvoice.getRecord(i).outstandingAmount);
      if (payment && !onActualPaymentChange && orderInvoice.getRecord(i).obSelected) {
        outstandingAmount = bd(orderInvoice.getRecord(i).amount);
      } else if (outstandingAmount.signum() < 0 && amount.signum() < 0) {
        if (outstandingAmount.abs().compareTo(amount.abs()) > 0) {
          differenceamt = outstandingAmount.subtract(amount);
          outstandingAmount = amount;
          amount = amount.subtract(differenceamt);
        }
      } else if (outstandingAmount.signum() > -1 && amount.signum() > -1 && outstandingAmount.compareTo(amount) > 0) {
        outstandingAmount = amount;
      }
      if (amount.signum() === 0) {
        if (outstandingAmount.signum() < 0 && (onActualPaymentChange || payment)) {
          orderInvoice.setEditValue(i, 'amount', num(outstandingAmount.toString()));
          orderInvoice.selectRecord(i);
        } else {
          orderInvoice.deselectRecord(i);
          orderInvoice.setEditValue(i, 'amount', num('0'));
        }
      } else if (amount.signum() === 1) {
        orderInvoice.setEditValue(i, 'amount', num(outstandingAmount.toString()));
        orderInvoice.selectRecord(i);
        if (outstandingAmount.signum() >= 0 || amount.signum() <= 0) {
          amount = amount.subtract(outstandingAmount);
        }
      } else if (outstandingAmount.signum() < 0) {
        orderInvoice.setEditValue(i, 'amount', num(outstandingAmount.toString()));
        orderInvoice.selectRecord(i);
        if (outstandingAmount.signum() <= 0 || amount.signum() <= 0) {
          amount = amount.subtract(outstandingAmount);
        }
      } else {
        orderInvoice.deselectRecord(i);
        orderInvoice.setEditValue(i, 'amount', num('0'));
      }
    }
    AP.updateActualExpected(form, view);
  }
  AP.updateInvOrderTotal(form, orderInvoice);
};

AP.doSelectionChanged = (record, state, view) => {
  const form = view.theForm;
  const orderInvoice = AP.grid(form, 'order_invoice');
  let amount = bd(form.getItem('actual_payment').getValue() || 0);
  const distributedAmount = bd(form.getItem('amount_inv_ords').getValue() || 0);
  const issotrx = form.getItem('issotrx').getValue();
  let outstandingAmount = bd(record.outstandingAmount);
  const glitem = bd(form.getItem('amount_gl_items').getValue() || 0);
  const credit = bd(form.getItem('used_credit').getValue() || 0);
  const bslamount = bd(form.getItem('bslamount').getValue() || 0);

  amount = amount.subtract(distributedAmount).subtract(glitem).add(credit);

  if (issotrx) {
    if (amount.signum() !== 0 && state) {
      if (outstandingAmount.signum() < 0 && amount.signum() < 0) {
        if (outstandingAmount.abs().compareTo(amount.abs()) > 0) outstandingAmount = amount;
      } else if (outstandingAmount.compareTo(amount) > 0) {
        outstandingAmount = amount;
      }
      orderInvoice.setEditValue(orderInvoice.getRecordIndex(record), 'amount', num(outstandingAmount.toString()));
    }
  } else if (state) {
    if (bslamount.compareTo(ZERO()) !== 0) {
      if (outstandingAmount.compareTo(amount) > 0) {
        orderInvoice.setEditValue(orderInvoice.getRecordIndex(record), 'amount', num(amount.toString()));
      } else {
        orderInvoice.setEditValue(orderInvoice.getRecordIndex(record), 'amount', num(outstandingAmount.toString()));
      }
    } else {
      orderInvoice.setEditValue(orderInvoice.getRecordIndex(record), 'amount', num(outstandingAmount.toString()));
    }
  }
  AP.updateInvOrderTotal(form, orderInvoice);
  AP.updateActualExpected(form, view);
  AP.updateDifference(form, view);
  AP.applyBankAmountToConverted(form);
};

// Defers `fn` until after the freshly delivered grid rows have committed to the
// grid controller's read store. At onGridLoad firing time the controller-backed
// reads/writes (getRecord/getSelectedRecords/setEditValue) still see the pre-fetch
// rows, so any settling that reads or writes edited cells must run on the next
// tick. This mirrors the classic ordInvDataArrived deferral (isc.Timer.setTimeout).
AP.afterGridSettle = (grid, paramName, fn) => {
  if (grid && grid.fireOnPause) {
    grid.fireOnPause(GRID_SETTLE_ID + ':' + paramName, fn, GRID_SETTLE_DELAY);
  } else {
    fn();
  }
};

// Mirrors classic ordInvDataArrived: for each backend pre-selected order/invoice row
// (read from grid.getSelectedRecords() AFTER the deferral — classic also reads
// getSelectedRecords() after dataArrived), keep/seed its editable `amount` (its own
// amount, falling back to the outstanding) and re-assert selection, then refresh the
// invoice/order total.
AP.seedPreselectedOrderInvoice = (form, grid, selectedRecords) => {
  const selected = selectedRecords || (grid.getSelectedRecords ? grid.getSelectedRecords() : []);
  for (let i = 0; i < selected.length; i++) {
    const record = selected[i];
    if (!record || typeof record !== 'object') continue;
    const idx = grid.getRecordIndex(record);
    const ownAmount = bd(record.amount || 0);
    const outstanding = bd(record.outstandingAmount || 0);
    const amount = ownAmount.compareTo(ZERO()) !== 0 ? ownAmount : outstanding;
    grid.setEditValue(idx, 'amount', num(amount.toString()));
    grid.selectRecord(idx);
  }
  AP.updateInvOrderTotal(form, grid);
};

// Mirrors classic selectionChangedCredit for backend pre-selected credit rows: seed

  }
};

    return false;
  }
  if ((paidamount.signum() < 0 && outstanding.signum() > 0) || (paidamount.signum() > 0 && outstanding.signum() < 0)) {
    if (messageBar) messageBar.setMessage(isc.OBMessageBar.TYPE_ERROR, null, OB.I18N.getLabel('APRM_ValueOutOfRange'));
    return false;

  return true;
```

### 4.3 `em_etmeta_onprocess` (process)

The three grid parameters are **Window Reference**, so `onProcess` runs as a **pre-submit validation
hook**: it reproduces all eight client-side guards and the `AddPaymentOnProcessActionHandler` server
round-trip (blocked-BP + write-off-limit). It returns `{ severity: 'error', text }` to abort, or
`undefined` to let the platform submit the selection. It does **not** call `view.executeProcess()`.

```js
async (process, view) => {
  const form = view.theForm;
  const orderInvoiceGrid = AP.grid(form, 'order_invoice');
  const glitemGrid = AP.grid(form, 'glitem');
  const receivedFrom = form.getItem('received_from').getValue();
  const currencyId = form.getItem('c_currency_id').getValue();
  const issotrx = form.getItem('issotrx').getValue();
  const finFinancialAccount = form.getItem('fin_financial_account_id').getValue();
  const amountInvOrds = bd(form.getItem('amount_inv_ords').getValue() || 0);
  const amountGLItem = bd(form.getItem('amount_gl_items').getValue() || 0);
  const total = bd(form.getItem('total').getValue() || 0);
  const actualPayment = bd(form.getItem('actual_payment').getValue() || 0);
  const overpaymentField = form.getItem('overpayment_action');
  const overpaymentAction = overpaymentField.getValue();
  const creditTotalItem = bd(form.getItem('used_credit').getValue() || 0);
  const document = AP.trxtype(form);
  const selectedRecords = orderInvoiceGrid.getSelectedRecords();
  const writeOffLimitPreference = OB.PropertyStore.get('WriteOffLimitPreference', view.windowId);

  let totalWriteOffAmount = ZERO();
  let totalOustandingAmount = ZERO();

  for (const rec of selectedRecords) {
    totalOustandingAmount = totalOustandingAmount.add(bd(rec.outstandingAmount));
  }

  const ordInvRows = (orderInvoiceGrid.data && orderInvoiceGrid.data.localData) ? orderInvoiceGrid.data.localData : [];
  for (let i = 0; i < ordInvRows.length; i++) {
    const editValues = orderInvoiceGrid.getEditValues(i);
    let writeoff = editValues ? editValues.writeoff : undefined;
    if (writeoff === null || writeoff === undefined) {
      writeoff = orderInvoiceGrid.getRecord(i).writeoff;
    }
    if (writeoff) {
      const writeOffLineAmount = bd(orderInvoiceGrid.getRecord(i).outstandingAmount || 0)
        .subtract(bd(orderInvoiceGrid.getEditedRecord(i).amount || 0));
      totalWriteOffAmount = totalWriteOffAmount.add(writeOffLineAmount);
    }
  }

  const fail = (label, title, text) => {
    view.messageBar.setMessage(isc.OBMessageBar.TYPE_ERROR, title || null, text || OB.I18N.getLabel(label));
    return { severity: 'error', text: text || OB.I18N.getLabel(label) };
  };

  if (overpaymentAction && receivedFrom === null) {
    return fail('APRM_CreditWithoutBPartner');
  }
  if (total.compareTo(ZERO()) < 0 && creditTotalItem.signum() !== 0) {
    return fail('APRM_CreditWithNegativeAmt');
  }
  if (
    actualPayment.compareTo(total.subtract(creditTotalItem)) > 0 &&
    totalOustandingAmount.compareTo(amountInvOrds.add(totalWriteOffAmount)) > 0
  ) {
    return fail('APRM_JSNOTALLAMOUTALLOCATED');
  }
  if (total.compareTo(actualPayment.add(creditTotalItem)) > 0) {
    return fail('APRM_JSMOREAMOUTALLOCATED');
  }
  if (
    creditTotalItem.compareTo(ZERO()) !== 0 &&
    total.compareTo(creditTotalItem) < 0 &&
    overpaymentField.isVisible && overpaymentField.isVisible() && overpaymentAction === 'CR'
  ) {
    return fail('APRM_MORECREDITAMOUNT');
  }
  if (
    document !== null && document !== '' &&
    (actualPayment.compareTo(ZERO()) === 0 || totalOustandingAmount.compareTo(ZERO()) === 0) &&
    amountGLItem.compareTo(ZERO()) === 0 &&
    view.parentWindow && view.parentWindow.windowId &&
    !overpaymentAction
  ) {
    return fail('APRM_ZEROAMOUNTPAYMENTTRANSACTION');
  }

  const glRows = (glitemGrid.data && glitemGrid.data.allRows) ? glitemGrid.data.allRows : [];
  for (let i = 0; i < glRows.length; i++) {
    const receivedInAmt = bd(glitemGrid.getEditedCell(i, 'received_in') || 0);
    const paidOutAmt = bd(glitemGrid.getEditedCell(i, 'paid_out') || 0);
    if (receivedInAmt.signum() === 0 && paidOutAmt.signum() === 0) {
      return fail('APRM_GLITEMSDIFFERENTZERO');
    }
  }

  const response = await callAction('org.openbravo.advpaymentmngt.actionHandler.AddPaymentOnProcessActionHandler', {
    issotrx,
    receivedFrom,
    currencyId,
    usesCredit: creditTotalItem.compareTo(ZERO()) !== 0,
    generatesCredit: overpaymentField.isVisible && overpaymentField.isVisible() && overpaymentAction === 'CR',
    selectedRecords,
    finFinancialAccount
  });
  const data = response && response.data ? response.data : {};

  if (data.message && data.message.severity === 'error') {
    return fail(null, data.message.title, data.message.text);
  }
  if (writeOffLimitPreference === 'Y' && num(totalWriteOffAmount.toString()) > num(data.writeofflimit)) {
    return fail('APRM_NotAllowWriteOff');
  }

  // All validations passed (client-side + the server pre-submit round-trip).
  // The grid parameters are Window-Reference, so the platform submits the
  // selection itself; returning undefined lets it proceed (do not call
  // view.executeProcess here, which would double-submit).
  return undefined;
}

```

## 6. Advisories (non-blocking observations)

- **No qa-passed references exist** anywhere in the inventory; all siblings are `migrated`/`blocked`. This
  migration was produced directly from the classic source + the architecture doc, with no trusted few-shot.
- **Paged transform-time redistribution dropped** (`ordInvTransformData`, `_requestNextOrderInvoicePage`,
  `editedSelectedRecords` cross-page carry, `obaprmAllRecordsSelectedByUser`): these are SmartClient
  datasource internals (manual `ResultSet.getRange` paging + transform-time row mutation). The new grid
  paginates itself, so the equivalent behaviour is the deterministic `distributeAmount` re-distribution
  over the loaded rows. Validate parity in QA when the order/invoice grid spans multiple pages.
- **`affectedParams` are sent as column-name strings** (e.g. `'overpayment_action_display_logic'`,
  `'c_currency_id_readonly_logic'`) instead of the classic `getField(name).paramId`. The new `form.getField`
  is index-based and items carry no `paramId`; the `AddPaymentDisplayLogicActionHandler` resolves the
  affected logic parameters from these stable identifiers. Confirm the handler accepts column names in QA.
- **The "never show again" auto-distribute banner is dropped.** It depended on `cachedRows < totalRows`
  (partial paging) which no longer exists in the self-paginating grid; with full data loaded, the classic
  branch that showed it does not fire. No behavioural loss for fully-loaded grids.
- **Overpayment original value map** is captured in `onLoad` from `getValueMap()` into
  `form._overpaymentOriginalValueMap` (the classic `originalValueMap` snapshot), then re-applied by
  `updateDifferenceActions`.
- The legacy `em_etmeta_payscript_logic` DSL (`AddPaymentRulesGeneric`) and the partial
  `autoSelectConfig`-only `em_etmeta_onload` **must be cleared** before pasting — they belonged to the
  retired path and would otherwise run alongside the new hooks.
