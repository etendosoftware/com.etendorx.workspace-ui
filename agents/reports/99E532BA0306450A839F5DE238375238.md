# Migration report — Select Invoices and Orders

## Inputs

- **Classic file:** `erp/WebContent/web/org.openbravo.module.remittance/js/rem_addinvandord_utilities.js`
- **Process id:** `99E532BA0306450A839F5DE238375238`
- **Process name:** Select Invoices and Orders
- **Module:** `org.openbravo.module.remittance` (`OB.REM.ProcessPP.*`)
- **Date:** 2026-06-11

## status: blocked

The process is **not migratable with full behavioral parity** on the current new-UI substrate.
One in-scope, live behavior — recomputing the dialog total when the user **edits a per-row
payment amount inside the embedded grid** (`OB.REM.ProcessPP.realPaymentOnChange`) — has **no
faithful equivalent**: the new UI exposes no per-grid-cell `onChange` hook, and no
script-registerable callback re-fires when a grid cell is edited in place. Emitting code anyway
would silently drop that recomputation, so the migration is blocked rather than emitted.

A partial, unverified migration already sits in the DB columns of this process (an `onLoad`, a
`payscript_logic`). It only covers the selection-based total and does **not** address the
cell-edit case; it is therefore not parity-complete and is not trusted. (It also violates the
module-body contract — its `em_etmeta_payscript_logic` is a bare object literal, not a module
body ending in `return { … }`.)

## Why this is blocked (the unsupported gap, in detail)

`realPaymentOnChange` is bound to the editable `realpayment` cell and does:

1. read every selected record's `realpayment`, taking the freshly edited value for the changed row;
2. write that edited value back into the selected-records array; and
3. set `total count` to the new sum, then `form.redraw()`.

Its purpose: **when the user changes a row's payment amount (without changing which rows are
selected), the total must update.**

On the new-UI substrate this cannot fire:

- **No per-grid-cell onChange column.** `em_etmeta_on_parameter_change` targets scalar
  `obuiapp_parameter` rows. `realpayment` is an `AD_Field` **inside** the window-reference grid,
  not a process parameter, so there is no column to carry this hook.
- **No callback re-fires on a cell edit.** In `WindowReferenceGrid.tsx`:
  - `grid.selectionChanged` subscribers fire only on `[rowSelection]` changes
    (lines ~2671–2678) — i.e. selecting/deselecting rows, **not** editing a selected row's cell.
  - `grid.dataArrived` subscribers fire only when the **raw datasource payload** changes
    (guarded by `rawString === rawRecordsStringRef.current`, lines ~2115–2129) — not on a local
    edit.
  - the cell-edit path (`handleRecordChange`, lines ~2561–2615) updates `localRecords` and the
    selection structure but **invokes no script-registerable subscriber**.

Net effect of forcing a migration: the total would update on selection toggles (via `onSelect`'s
equivalent) but would **go stale whenever the user edits an amount** — a real, user-visible
behavior loss. Per Babel's conservative gate, that is a block, not a best-effort emit.

### What the platform would need to lift the block

Any **one** of the following, added to the client substrate, would make this process migratable:

- A **per-grid-field onChange** hook for window-reference grids (e.g. carry the field's
  `onchangefunction` into a new metadata channel and invoke it from `GridCellEditor` /
  `handleRecordChange`), passing `(item, view, form, grid)` with `item.record` = the edited row; **or**
- Have `handleRecordChange` **re-fire the `selectionChanged` subscribers** (or a new
  `editComplete`/`cellEdited` chained callback on the grid proxy) when an **edited row is part of
  the selection**, so a script registered from `onGridLoad` can recompute the total on cell edits; **or**
- A declarative "recompute field X = sum(selected.realpayment)" binding that recomputes on both
  selection change and cell edit.

Until one of those exists, `OB.REM.ProcessPP.realPaymentOnChange` cannot be reproduced faithfully.

## Advisories (non-blocking observations)

- **Cluster note (REM).** Per the architecture doc's cloning rule, the REM cluster
  (`ob-rem-utilities.js` + `rem_addinvandord_utilities.js`) shares the `OB.REM` namespace, not
  function calls. Once each process owns its scope the sharing disappears; nothing to clone here
  while blocked.
- **Sibling process.** `C88AB6CBA1694000AFF5706A31B08AE1` (Select Payments Pick and Edit), the
  other REM process, is `migrated` (not `qa-passed`) — so it is **not** a trusted reference. It was
  read only for style awareness. Notably it has **no editable in-grid amount with an onChange**: its
  total recomputes purely on selection (`em_etmeta_on_grid_load` + `getSelectedRecords`), which is
  exactly the supported subset; it does not exercise the gap that blocks this process.
- **DOM guard (`getActiveElement().className === 'OBGridBody'`).** This only guards *when* the
  `expectedDate <= dueDate` criteria is injected; the criteria itself is supported via `onLoad`
  `_filterExpressions`. The guard is not reproducible but is not the blocker.
- **`pneSelectedRecords` mutation.** The classic code mutates the selected-records array to stage
  edited amounts. The supported `grid.getSelectedRecords()` returns a read-only snapshot; the staging
  pattern is part of why the cell-edit recompute has no clean substrate path.

## Manual-test checklist (to run once the block is lifted and code is pasted)

These define full parity; they are listed now so QA is ready when the platform gap closes.

1. **Open dialog / grid load + due-date filter.** Launch *Select Invoices and Orders* from a
   remittance whose `dueDate` is set. Expect the **Pending Payments** grid to show only rows with
   `expectedDate <= remittance dueDate`. (Covers `onLoadGrid` criteria injection.)
2. **Pre-selected rows seed the total.** If rows arrive pre-selected, expect **Total Count** to
   equal the sum of their `realpayment` on open. (Covers `onLoadGrid` initial sum.)
3. **Select a row → total updates.** Select an additional row; expect **Total Count** to increase
   by that row's payment amount. Deselect it; expect the total to revert. (Covers `onSelect`.)
4. **Edit a selected row's payment amount → total updates.** With a row selected, change its
   **Real Payment** cell to a new value and commit (blur). Expect **Total Count** to recompute to the
   new sum **without** re-toggling the selection. **(This is the currently unsupported case — it
   must pass for parity.)**
5. **Edit then deselect/reselect.** Edit an amount, deselect the row, reselect it; expect the total
   to reflect the latest edited amount.
6. **Multiple currencies / many rows.** Select several rows with different amounts; expect the total
   to equal their exact sum, with no lexicographic ordering error (numeric param).
