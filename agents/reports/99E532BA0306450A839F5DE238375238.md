## Substrate verification (ETP-3748 `onRecordChange`) — VERIFIED

The Updates claim was checked against real source, end-to-end:

1. **Proxy is script-registerable.** `scriptProxies.ts` exposes `grid.onRecordChange = (fn) =>
   controller.onRecordChange(fn)` (line 881). `"onRecordChange"` is in the grid-method allowlist
   (line 482) **and** in `LIVE_GRID_METHODS` (line 513), so it becomes a live method once the modal
   injects a `GridController` (not a deferred/throwing stub).
2. **Controller registers the subscriber.** `createEmbeddedGridController` takes a `recordChangeSubs`
   array (line 1265); `controller.onRecordChange(fn)` pushes into it (lines 1332–1334).
3. **The array is the live ref.** The controller is constructed with `recordChangeSubsRef.current`
   (`WindowReferenceGrid.tsx` line 2824), so the controller's `recordChangeSubs` **is**
   `recordChangeSubsRef.current`.
4. **It fires on every real cell edit.** `handleRecordChange` (line 2721) calls `fireCellEditHooks`
   (line 2782) on every in-place edit; `fireCellEditHooks` iterates `recordChangeSubsRef.current` and
   invokes each subscriber as `fn(row, changes)` (lines 2692–2698), wrapped in try/catch. It is
   **additive**: a no-op when `recordChangeSubsRef.current.length === 0`
   (`if ((!hasSubs && !hasColumnOnChange) || …) return;`, line 2685).
5. **Documented.** `new-javascript-code.md` §8.5 lists
   `grid.onRecordChange((record, changes) => …)` → "on every in-place cell edit" → classic
   `realPaymentOnChange`-style recompute (line 613); the §10.2 equivalence table maps
   "field-level cell `onchangefunction` recompute (e.g. `realPaymentOnChange`)" →
   `grid.onRecordChange(...)` from `onGridLoad` (line 833) and the `pneSelectedRecords` staging array
   → "drop — recompute from `grid.getSelectedRecords()` + the `changes` arg" (line 834).

Conclusion: the hook exists, is script-registerable from `onGridLoad`, fires on real cell edits, and is
documented. The Updates claim is **confirmed**.

## Generated code per field (for manual pasting)

### `em_etmeta_onload` — entity `obuiapp_process`

```js
async (process, view) => {
  const GRID_KEY = 'pending payments';
  const remittanceDueDate = view.getContextInfo().inpduedate;
  if (!remittanceDueDate) {
    return;
  }
  return {
    _filterExpressions: {
      [GRID_KEY]: {
        expectedDate: { operator: 'lessOrEqual', value: remittanceDueDate }
      }
    }
  };
}
```

### `em_etmeta_on_grid_load` — parameter **Pending Payments** (`4A592FF004A8425999E67FDB5E7A8C66`)

```js
(grid, view, parameters) => {
  const TOTAL_ITEM = 'total count';
  const SCALE = 2;
  const sumSelected = () => {
    let total = BigDecimal.prototype.ZERO;
    const selected = grid.getSelectedRecords();
    for (let i = 0; i < selected.length; i++) {
      const amount = selected[i].realpayment;
      if (amount !== undefined && amount !== null && amount !== '') {
        total = total.add(new BigDecimal(String(amount)));
      }
    }
    return total.setScale(SCALE);
  };
  const writeTotal = () => {
    grid.view.theForm.getItem(TOTAL_ITEM).setValue(Number(sumSelected().toString()));
  };
  writeTotal();
  grid.selectionChanged = () => writeTotal();
  grid.onRecordChange((record, changes) => writeTotal());
}
```

This single `onGridLoad` body absorbs all three classic functions:
- the **seed** (classic `onLoadGrid` sum of pre-selected rows) via the initial `writeTotal()`;
- the **selection** recompute (classic `onSelect`) via `grid.selectionChanged`;
- the **cell-edit** recompute (classic `realPaymentOnChange`) via `grid.onRecordChange`.

## Advisories (non-blocking)

- **`form.redraw()` dropped.** The new-UI `setValue` updates the field reactively; an explicit redraw is
  not part of the supported surface and is unnecessary.
- **DOM focus guard dropped.** `getActiveElement().className === 'OBGridBody'` only gated *when* the
  due-date criteria was injected. The filter is now declarative in `onLoad`, applied once per open;
  `onRecordChange` fires only on real edits, so the guard's intent is preserved.
- **`pneSelectedRecords` staging dropped.** Per §10.2, the edited value is already merged into the
  selection before `onRecordChange` runs, so recomputing from `grid.getSelectedRecords()` is faithful;
  no manual write-back array is needed.
- **Numeric total.** `total count` is reference `12` (numeric), so the body writes a `number`
  (`Number(sumSelected().toString())`). The `BigDecimal` accumulation preserves scale/rounding before
  the conversion; this differs from the sibling P&E process which writes a multi-currency **string**
  because that grid is multi-currency — this one is single-amount.
- **`onSelect` semantics.** Classic `selectionChanged` here is used only to recompute the total, so the
  full-array `grid.selectionChanged = fn` subscriber (not the per-toggle `onSelectionToggle`) is the
  right target; it recomputes the whole sum on every selection change, matching the classic loop.
- **Sibling process untrusted.** `C88AB6CBA1694000AFF5706A31B08AE1` is `migrated` (not `qa-passed`); read
  for style only.

## Manual-test checklist (new UI)

1. **Grid load + due-date filter.** Launch *Select Invoices and Orders* from a remittance whose
   `dueDate` is set. Expect the **Pending Payments** grid to show only rows with
   `expectedDate <= remittance dueDate`. (Covers `onLoad` `_filterExpressions`.)
2. **Pre-selected rows seed the total.** If rows arrive pre-selected, **Total Count** equals the sum of
   their `realpayment` on open. (Covers the `onGridLoad` initial `writeTotal()`.)
3. **Select a row → total updates.** Select an additional row; **Total Count** increases by that row's
   amount. Deselect it; the total reverts. (Covers `grid.selectionChanged`.)
4. **Edit a selected row's payment amount → total updates.** With a row selected, change its
   **Real Payment** cell to a new value and commit (blur). **Total Count** recomputes to the new sum
   **without** re-toggling selection. **(This is the formerly-blocked case — must pass.)**
5. **Edit then deselect/reselect.** Edit an amount, deselect the row, reselect it; the total reflects
   the latest edited amount.
6. **Many rows / decimals.** Select several rows with decimal amounts; **Total Count** equals their
   exact sum to 2 decimals, with no rounding drift (BigDecimal) and no lexicographic error (numeric).
