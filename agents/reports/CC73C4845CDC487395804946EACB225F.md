## Classic file summary

`OB.APRM.FundsTranfer` declares two entry points:

- `onLoad(view)` — bound to `on_load_function`. Sets the `description` field to the label
  `FundsTransfer`.
- `depositToOnChange(item, view, form, grid)` — bound to the **Deposit To** parameter
  (`onchangefunction`, columnname `fin_financial_account_id`). On change it calls the server handler
  `FundsTransferOnChangeDepositToActionHandler` with `{ accountID }`, and in the callback writes the
  returned currency into the **Currency To** selector (`c_currency_to_id`) and redraws the form.

No `clientsidevalidation`, no `on_refresh_function`, no `ongridloadfunction`, no module-level helpers.

> Note: the process already carried an `em_etmeta_payscript_logic` PayScript-style draft in the DB.
> It was **not** produced or verified by this pipeline and does not reproduce the classic callback
> behavior (it never calls the server handler), so it is treated as untrusted and replaced by the
> faithful translation below (`em_etmeta_payscript_logic` is left empty).

### `em_etmeta_on_parameter_change` — entity `obuiapp_parameter`, parameter **Deposit To** (`fin_financial_account_id`, id `F4029E7726074FF2B6D3B59AE3072711`)

```js
(item, view, form, grid) => {
  const HANDLER =
    'org.openbravo.advpaymentmngt.actionHandler.FundsTransferOnChangeDepositToActionHandler';

  const callback = (response, data) => {
    if (!data || !data.currencyID) {
      return;
    }
    const currencyTo = form.getItem('c_currency_to_id');
    currencyTo.setValueFromRecord({
      id: data.currencyID,
      _identifier: data.currencyISO
    });
    form.redraw();
  };

  OB.RemoteCallManager.call(HANDLER, { accountID: item.getValue() }, {}, callback);
}
```

## Advisories (non-blocking)

1. **Value-map rewrite.** The classic code mutated `currencyTo.valueMap` directly and then called
   `setValue`. The new-UI item proxy exposes no live `.valueMap` property, so that pattern would be a
   silent no-op. Replaced with `setValueFromRecord({ id, _identifier })`, the supported equivalent that
   updates the selected value and its displayed ISO label together. Behavior is preserved.
2. **Untrusted DB draft replaced.** The pre-existing `em_etmeta_payscript_logic` (a `FundsTransferRules`
   compute/validate object) did not call the server handler and is not a faithful translation; it is
   superseded by the onChange above and should be cleared.
3. **Callback shape kept.** The classic `OB.RemoteCallManager.call(..., callback)` form is retained
   (over rewriting to `await`) for line-for-line fidelity, as the playbook recommends.

## Manual test checklist (new UI)

1. **onLoad — description default.** Open the Funds Transfer process modal. Expect the **Description**
   field to be pre-filled with the translated label for `FundsTransfer` (same text the classic dialog
   showed).
2. **onChange — Deposit To drives Currency To.** With the modal open, set **Deposit To** to a financial
   account whose currency is known (e.g. an account in USD). Expect **Currency To** to update
   automatically to that account's currency, displaying its ISO code, without manual selection.
3. **onChange — change again.** Change **Deposit To** to an account with a *different* currency. Expect
   **Currency To** to update to the new currency (no stale value).
4. **onChange — empty / no currency.** If an account resolves to no currency from the handler, expect
   **Currency To** to be left unchanged (no error, no clearing).
5. **No regressions.** Confirm no console error such as `"... is not implemented yet"` during open or
   while changing Deposit To.
