## What the new UI lacks (technical detail)

The in-scope onLoad ends with this exact block (lines 54â€“60), which decides the default
`Document Action` after narrowing the dropdown to the actions allowed by the server:

```js
docActionField.setValueMap(newValues);                 // supported (item.setValueMap)
if (newValues[documentActions[0]]) {
  docActionField.setValueProgrammatically(documentActions[0]);   // UNSUPPORTED
} else {
  docActionField.setValueProgrammatically(docActionField.getFirstOptionValue()); // UNSUPPORTED Ă—2
}
```

1. **`item.setValueProgrammatically(value)`** â€” the SmartClient method that sets a form
   item's value *and* runs its change handlers programmatically. The new-UI item proxy
   exposes `setValue(v)` and `setValueFromRecord(record)`, but **not**
   `setValueProgrammatically`. It is neither a live method nor a deferred ("not implemented
   yet") stub, so a call reaches `undefined()` and throws a raw `TypeError`. Whether
   `setValue` would also fire the dependent `Document Action` `onChange` (it drives
   `onChangeDocumentAction`, which shows the Void fields) is not documented, so substituting
   it is an unverified semantic change, not a faithful 1:1 translation.
   - **What would be required:** a documented `item.setValueProgrammatically` on the item
     proxy (with defined change-firing semantics), or an explicit entry in the
     `new-javascript-code.md` equivalence table stating `setValueProgrammatically â†’
     item.setValue` and confirming the onChange-firing behavior matches Classic.

2. **`item.getFirstOptionValue()`** â€” returns the first option of the selector's current
   value list (the Classic fallback when the record's own action is not among the allowed
   ones). The new-UI item proxy has **no** way to read a selector's option list: it offers
   `getValueMap()` (the valueâ†’label map) and `getValue()`, but nothing that yields "the first
   available option id". Reconstructing it from `getValueMap()` would be a non-1:1
   reinterpretation (the map is unordered keyâ†’label, not the rendered option order), so it
   cannot be done faithfully.
   - **What would be required:** a documented `item.getFirstOptionValue()` (or an ordered
     option-list accessor) on the item proxy, plus an equivalence-table entry.

Both gaps sit on the same onLoad branch, so neither can be worked around by dropping dead
code. The `onChangeDocumentAction` hook on its own (`show()` of the Void fields) is fully
supported, but a partial migration that silently drops the default-selection logic would
break parity and is exactly what this agent refuses to ship.

## Advisories (non-blocking observations)

- **`POReference` does not exist on this process.** The classic `ProcessInvoices.onLoad`
  hides `supplierReferenceField = form.getItem('POReference')` and `onChangeDocumentAction`
  shows it for window `183` with `< 2` records. This process has only `DocAction`,
  `VoidDate`, `VoidAccountingDate` â€” no `POReference`. In Classic, `getItem('POReference')`
  on a process without that field returns null and the `.hide()`/`.show()` is a no-op; the
  new UI behaves the same (proxy with no backing field). Carry it over verbatim once
  unblocked; it is inert here. (The `POReference` show path is only meaningful for the sibling
  processes that do declare it.)
- **SmartClient `Array.prototype.add`.** Lines 21/24 use `documentStatuses.add(...)` /
  `documentActions.add(...)`, a SmartClient extension. Migrate as native `.push(...)`.
- **Shared-file family.** `processRecords.js` backs 5 processes (`A5A9B914`, `B4A21A61`,
  `8DF818E4`, `33338B1F`, `272C8D38`). Each owns a distinct `onLoad` namespace
  (`ProcessInvoices` / `ProcessOrders` / `ProcessShipment`) and a different `*Defaults`
  handler / `tableId` / `tabId`. **The same two unsupported methods appear in all three
  onLoad variants**, so every member of this family is blocked for the same reason â€” not
  only this row. Once the two item-proxy methods exist, clone the migrated body into each
  member row per the cloning rule.
- **`view.okButton` / `.enable()` / `.isEnabled()`** appear in the Orders/Shipment onLoad
  variants (out of scope here) and are also undocumented; flag them when those rows are
  attempted.
