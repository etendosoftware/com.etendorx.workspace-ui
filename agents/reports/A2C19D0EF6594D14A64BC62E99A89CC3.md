## Generated code per field

### `em_etmeta_payscript_logic` — process (`obuiapp_process` `A2C19D0EF6594D14A64BC62E99A89CC3`)
Shared module body. Holds the three in-scope helpers + the small reused utilities (`bd`, `ZERO`, `ONE`,
`isNum`, `appliesUom`, `showOutOfRange`). The two validators (`RMOrderQtyValidate` →
`orderQtyValidate`, `RMReturnedUOMValidate` → `returnedUomValidate`) and the selection handler
(`RMOrderSelectionChange` → `orderSelectionChange`) are exported by bare name so both `onGridLoad`
bodies reference them directly. The repeated `OutOfRange` message branch (identical in both classic
validators) is factored into one `showOutOfRange` helper (no duplication). Money math stays on
`BigDecimal` verbatim. `isc.isA.Number` is reproduced by the local `isNum` helper. Parse-checked as a
module body ending in `return { … }`.
```
const UOM_MANAGEMENT = 'UomManagement';
const RETURNABLE_HANDLER =
  'org.openbravo.common.actionhandler.ServiceReturnableActionHandler';
const LABEL_OUT_OF_RANGE = 'OBUIAPP_RM_OutOfRange';
const LABEL_OUT_OF_RANGE_AUM = 'OBUIAPP_RM_OutOfRange_AUM';

const ZERO = () => BigDecimal.prototype.ZERO;
const ONE = () => BigDecimal.prototype.ONE;
const bd = (value) => new BigDecimal(String(value));

// Faithful reproduction of the classic SmartClient `isc.isA.Number(x)` type-check,
// which the new-UI `isc` shim does not expose. Pure, self-contained, no platform dependency.
const isNum = (value) => typeof value === 'number' && isFinite(value);

const isUomManagementEnabled = () =>
  OB.PropertyStore.get(UOM_MANAGEMENT) === 'Y';

const appliesUom = (record) =>
  isUomManagementEnabled() && record.returnedUOM !== record.uOM;

const showOutOfRange = (view, record, remaining, newReturnedQty) => {
  if (appliesUom(record)) {
    view.messageBar.setMessage(
      isc.OBMessageBar.TYPE_ERROR,
      null,
      OB.I18N.getLabel(LABEL_OUT_OF_RANGE_AUM, [
        newReturnedQty.toString(),
        record.uOM$_identifier,
        remaining.toString()
      ])
    );
  } else {
    view.messageBar.setMessage(
      isc.OBMessageBar.TYPE_ERROR,
      null,
      OB.I18N.getLabel(LABEL_OUT_OF_RANGE, [remaining.toString()])
    );
  }
};

const returnedUomValidate = (item, validator, value, record) => {
  const view = item.grid.view;
  const movementQty = isNum(record.movementQuantity)
    ? bd(record.movementQuantity)
    : ZERO();
  const returnedQty = isNum(record.returnQtyOtherRM)
    ? bd(record.returnQtyOtherRM)
    : ZERO();
  let newReturnedQty = isNum(record.returned) ? bd(record.returned) : ZERO();

  if (appliesUom(record)) {
    newReturnedQty = newReturnedQty.multiply(
      isNum(record.aumConversionRate) ? bd(record.aumConversionRate) : ONE()
    );
  }

  const remaining = movementQty.subtract(returnedQty);
  if (newReturnedQty.compareTo(remaining) <= 0 && record.returned > 0) {
    view.messageBar.hide(true);
    return true;
  }
  showOutOfRange(view, record, remaining, newReturnedQty);
  return false;
};

const orderQtyValidate = (item, validator, value, record) => {
  const view = item.grid.view;
  if (!isNum(value)) {
    return false;
  }
  if (record.goodsShipmentLine === null || record.goodsShipmentLine === '') {
    return value !== null && value > 0;
  }
  const movementQty =
    record.movementQuantity !== null ? bd(record.movementQuantity) : ZERO();
  const returnedQty =
    record.returnQtyOtherRM !== null ? bd(record.returnQtyOtherRM) : ZERO();
  let newReturnedQty = bd(value);

  if (appliesUom(record)) {
    newReturnedQty = newReturnedQty.multiply(bd(record.aumConversionRate));
  }

  const remaining = movementQty.subtract(returnedQty);
  if (value !== null && newReturnedQty.compareTo(remaining) <= 0 && value > 0) {
    view.messageBar.hide(true);
    return true;
  }
  showOutOfRange(view, record, remaining, newReturnedQty);
  return false;
};

const orderSelectionChange = (grid, view, record, state) => {
  if (!state) {
    return;
  }
  let contextInfo = view.parentWindow.activeView.getContextInfo(
    false,
    true,
    true,
    true
  );
  if (!contextInfo.inpdateordered) {
    contextInfo = view.parentWindow.activeView.parentView.getContextInfo(
      false,
      true,
      true,
      true
    );
  }
  if (!record.returnReason) {
    record.returnReason = contextInfo.inpcReturnReasonId;
  }
  OB.RemoteCallManager.call(
    RETURNABLE_HANDLER,
    {
      rfcOrderDate: contextInfo.inpdateordered,
      goodsShipmentId: record.id,
      productId: record.product
    },
    {},
    (response, data) => {
      if (!data.message) {
        return;
      }
      if (data.message.severity === isc.OBMessageBar.TYPE_ERROR) {
        view.messageBar.setMessage(
          isc.OBMessageBar.TYPE_ERROR,
          data.message.title,
          data.message.text
        );
        grid.deselectRecord(grid.getRecordIndex(record));
      } else {
        view.messageBar.setMessage(
          isc.OBMessageBar.TYPE_WARNING,
          data.message.title,
          data.message.text
        );
      }
    }
  );
};

return {
  returnedUomValidate,
  orderQtyValidate,
  orderSelectionChange
};
```

## Advisories (non-blocking)
- **`OB.RM.onLoad` is a stale/dead `on_load_function` binding.** It is declared in
  `OBUIAPP_PROCESS.xml` for this process but **defined nowhere** in the repo (verified by a repo-wide
  search; the classic file defines no `onLoad`). In Classic this resolves to `undefined` → no-op.
  Migrated as empty `em_etmeta_onload`. No behavior is lost. (If a future cleanup wants to remove the
  dangling XML binding, that is a separate, non-UI change.)
- **The file is a shared module for three Pick&Edit processes.** Six of the nine `OB.RM.*` functions
  (`RMReceipt*`, `RMShipment*`) are bound to the **RM Receipt** (`5E9F9D7EECC24E4FBB2C60840FF613BE`) and
  **RM Shipment** (`4AD70293357245AB96E59C2CDB43A35D`) processes and are **dead code for this process**
  (Section 10.3). They will migrate on their own rows. Note the Receipt/Shipment functions use
  `grid.getSelectionObject().lastSelectionItem`, `grid.getEditRow()`, and `item.pickList.getSelection()`
  (AUM unit swap) — those need their own coverage analysis when those two processes are migrated; they
  are **not** assessed here.
- **`isc.isA.Number` reproduced in-module (not a substrate gap).** The new-UI `isc` shim is
  `{ confirm, ask, warn, say, OBMessageBar }` and does not expose `isc.isA`. Because `isc.isA.Number(x)`
  is a pure type predicate with trivial semantics, it is reproduced faithfully by the local `isNum`
  helper rather than blocked. If `isc.isA.*` becomes common across migrations, adding it to the shim
  would let scripts keep the literal Classic call — optional, not required.
- **`grid.deselectRecord(record)` → index port.** The substrate API is index-based
  (`deselectRecord(index)`); the classic call passes a record. Ported to
  `grid.deselectRecord(grid.getRecordIndex(record))`, matching the C4265E27 reference. Behaviorally
  identical.
- **Tab-selection `(grid, record, state)` → `onSelectionToggle((record, state))`.** The classic
  selection function received `grid` as its first argument; the new-UI `onSelectionToggle` callback
  receives only `(record, state)`. `grid` and `view` are captured from the `onGridLoad` closure and
  passed through to the shared `orderSelectionChange(grid, view, record, state)` helper — equivalent.
- **`OutOfRange` message branch de-duplicated.** Both classic validators contained the identical
  AUM-vs-plain `setMessage` branch; factored into the single `showOutOfRange` helper. No behavior change.

### 2026-06-17 — SUBSTRATE FIX: P&E grid fetched **zero records** (missing `buttonOwnerViewTabId`)

**Symptom (QA).** Opening *RFC/RTV HQL Pick and Edit Lines* rendered both grids but the
`/api/datasource` fetch for entity `C_RM_ReturnToVendorHQL` returned **no rows**, while the Classic
form-encoded request to the same datasource returned the expected lines. The migrated JS is correct and
plays no part — the defect is 100% platform-side, in how the new UI assembles the P&E fetch.

**Root cause.** The new-UI P&E request body omits the `buttonOwnerViewTabId` parameter that Classic
always sends. In Classic, `OBPickAndExecuteGrid.transformRequest`
(`erp/.../js/process/ob-pick-and-execute-grid.js:220-223`) appends
`dsRequest.params.buttonOwnerViewTabId = me.view.buttonOwnerView.tabId` — the tabId of the view that
owns the process button (the tab the process was launched from, **not** the P&E grid's own tab). The
backend HQL transformers branch on it:
`OrderLinePickEditTransformer.java:50`, `ReturnToFromCustomerVendorHQLTransformer.java:120-127`,
`ReturnFromCustomerHQLInserter.java:42-48`. When it arrives `null` (new UI), the transformer takes the
wrong branch / skips the WHERE clause and the grid comes back empty. The user confirmed empirically that
adding `buttonOwnerViewTabId` to the request makes the new UI behave like Classic.

**FIXED (ETP-3748, uncommitted working tree, 2026-06-17, JS untouched) — general for all P&E processes.**
The single component that renders every P&E grid is `WindowReferenceGrid.tsx`, so the fix is general:
- New constant `BUTTON_OWNER_VIEW_TAB_ID_PARAM = "buttonOwnerViewTabId"` (no magic string).
- New pure exported helper `applyButtonOwnerViewTabId(options, originTab)`: sets the param to
  `originTab.id` only when an owner tab with an id exists (mirrors Classic's
  `buttonOwnerView && buttonOwnerView.tabId` guard).
- Called inside the `datasourceOptions` memo next to `isPickAndEdit = true`. `originTab` is the tab the
  process was launched from (`ProcessDefinitionModal` passes `originTab={tab}` from `useTabContext()`),
  i.e. the Classic `buttonOwnerView`. Added `originTab` to the memo deps and the new key to
  `datasourceRefetchKey` (stable for the modal's life → no extra refetch).
- The param flows untouched through `useDatasource` → `datasource.get` (`buildParams` does not filter it)
  → `/api/datasource` route (`createFormData` forwards all non-null keys) → backend servlet. No change to
  `useDatasource.ts`, `datasource.ts`, or the API route.

**Classic parity.** Same parameter name, same value (owner tab id), same conditional guard ⇒ the backend
receives what it expects and branches the HQL exactly as Classic does.

**Non-regressive.** When `originTab`/`originTab.id` is absent the param is simply not added (identical to
today's behavior), matching Classic's guard. No other parameter, filter branch, hook, or the create-row
