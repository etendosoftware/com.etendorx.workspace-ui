## In-scope code

### `OB.Jobs.ProcessInvoices.onLoad(view)` (lines 6–64)

1. Reads `DocAction`, `VoidDate`, `VoidAccountingDate`, `POReference` items and the selected records
   from the launching grid.
2. Builds distinct `documentStatuses` / `documentActions` from the selection.
3. **Hides** `VoidDate`, `VoidAccountingDate`, `POReference` (these are revealed later by the onChange).
4. Calls `com.smf.jobs.defaults.ProcessInvoicesDefaults` via `OB.RemoteCallManager.call` with
   `tableId = "318"`.
5. In the callback: filters the `DocAction` value map to the server-returned `actions`, applies it with
   `setValueMap(newValues)`, then programmatically selects an option —
   `setValueProgrammatically(documentActions[0])` if available, else
   `setValueProgrammatically(getFirstOptionValue())`.

Note: unlike the Orders variant, `ProcessInvoices.onLoad` does **not** touch the OK button — there is no
`view.okButton` manipulation to port here.

### `OB.Jobs.ProcessInvoices.onChangeDocumentAction(item, view, form, grid)` (lines 66–77)

Fires when the "Document Action" value is committed. When the dialog was launched from window `183`
(Purchase Invoice) **and** the chosen action is `VO` (Void) or `RC` (Reverse Correct):
- shows `VoidDate` and `VoidAccountingDate`;
- additionally shows `POReference` only if fewer than 2 records are selected.

It never hides anything (the onLoad already hid them); it only reveals fields under those conditions.

### Process `em_etmeta_onload` — entity `obuiapp_process` (PASTE)

Faithful port of `OB.Jobs.ProcessInvoices.onLoad`. Two-parameter signature `(_process, view) => { … }`
because the runtime calls `fn(processDefinition, onLoadView)` — the canonical `view` (carrying
`theForm`, `selectedRecords`) is the second positional argument; a one-parameter `(view) =>` would bind
`view` to `processDefinition`, making `view.theForm` `undefined` and throwing on the first executable
line. Remaps `view.parentWindow.view.viewGrid.getSelectedRecords()` → `view.selectedRecords`,
`Array.add` → `.push`, and the `currentValues[action]` object idiom → `currentValues.find(...)` (because
`getValueMap()` returns a `ListOption[]` array). The field-hiding (`VoidDate`/`VoidAccountingDate`/
`POReference`) is preserved exactly. The repeated handler name and table id are hoisted to constants.

```js
(_process, view) => {
  const INVOICE_DEFAULTS_HANDLER = 'com.smf.jobs.defaults.ProcessInvoicesDefaults';
  const INVOICE_TABLE_ID = '318';
  const form = view.theForm;
  const docActionField = form.getItem('DocAction');
  const voidDateField = form.getItem('VoidDate');
  const voidAcctDateField = form.getItem('VoidAccountingDate');
  const supplierReferenceField = form.getItem('POReference');
  const selectedRecords = view.selectedRecords || [];
  const documentStatuses = [];
  const documentActions = [];

  for (let i = 0; i < selectedRecords.length; i++) {
    const record = selectedRecords[i];
    if (!documentStatuses.includes(record.documentStatus)) {
      documentStatuses.push(record.documentStatus);
    }
    if (!documentActions.includes(record.documentAction)) {
      documentActions.push(record.documentAction);
    }
  }

  voidDateField.hide();
  voidAcctDateField.hide();
  supplierReferenceField.hide();

  OB.RemoteCallManager.call(
    INVOICE_DEFAULTS_HANDLER,
    { documentStatuses: documentStatuses, isProcessing: '', tableId: INVOICE_TABLE_ID },
    {},
    (response, data, request) => {
      const actions = data.actions || [];
      const currentValues = docActionField.getValueMap();
      const newValues = {};

      for (let i = 0; i < actions.length; i++) {
        const action = actions[i];
        const match = currentValues.find((option) => option.value === action || option.id === action);
        if (match) {
          newValues[action] = match.label;
        }
      }

      docActionField.setValueMap(newValues);
      // Avoid setting an action that is on the record(s) but not in the dropdown list.
      if (newValues[documentActions[0]]) {
        docActionField.setValueProgrammatically(documentActions[0]);
      } else {
        docActionField.setValueProgrammatically(docActionField.getFirstOptionValue());
      }
    }
  );
}
```

Compile-check: written to `/tmp/babel_inv_onload.js` and validated as a bare arrow function
(`typeof f === "function"` → OK).

### "Document Action" parameter (`DDCA2212D7AF4EE4AD51B1E968CB95F5`) — `em_etmeta_on_parameter_change` (PASTE)

Faithful port of `OB.Jobs.ProcessInvoices.onChangeDocumentAction`. Keeps the four-argument signature
`(item, view, form, grid)`. The action codes and window id are hoisted to constants. **Re-emitted (QA
fix):** selection is now read through `view.getSelection()` — the live path in the `on_parameter_change`
context — instead of the dead Classic idiom `view.parentWindow.view.viewGrid.getSelectedRecords()` (see
the coverage row; that grid/selection path is never wired into the onChange view proxy and collapsed to
`[]`, forcing POReference visible even with 2+ records). Guarded for absence. Behavior is identical:
reveal Void/Accounting-Date on `VO`/`RC` in window `183`, and additionally `POReference` only when fewer
than 2 records are selected.

```js
(item, view, form, grid) => {
  const PURCHASE_INVOICE_WINDOW_ID = '183';
  const VOID = 'VO';
  const REVERSE_CORRECT = 'RC';
  const docActionField = form.getItem('DocAction');
  const docAction = docActionField.getValue();
  const selectedRecords = (view.getSelection && view.getSelection()) || [];
  const isVoidLikeAction = docAction === VOID || docAction === REVERSE_CORRECT;

  if (view.windowId === PURCHASE_INVOICE_WINDOW_ID && isVoidLikeAction) {
    form.getItem('VoidDate').show();
    form.getItem('VoidAccountingDate').show();
    if (selectedRecords.length < 2) {
      form.getItem('POReference').show();
    }
  }
}
```

## Advisories (non-blocking)

1. **Real (not dangling) onChange — unlike the Shipment sibling.** This process's
   `onchangefunction = OB.Jobs.ProcessInvoices.onChangeDocumentAction` resolves to a function that
   **exists** in `processRecords.js` (lines 66–77). It is migrated. (Contrast: the Shipment row's
   homonymous binding is dangling and was correctly left EMPTY.)
2. **`Array.add` → `.push`.** The onLoad loop uses `documentStatuses.add(...)` / `documentActions.add(...)`
   (SmartClient `Array` augmentation); ported to `.push(...)`. Not a platform gap.
3. **Selection access differs by hook (QA fix).** In `onLoad` selection comes from `view.selectedRecords`
   (onLoad `hookData`). In `on_parameter_change` the Classic idiom
   `view.parentWindow.view.viewGrid.getSelectedRecords()` is a **dead path**: `runHook` builds the view
   proxy without `grid`/`hookData`, and `buildParentWindow` overwrites `view` with only
   `{ getContextInfo, getView }`, so `view.parentWindow.view.viewGrid` is `undefined` and the fallback
   collapsed to `[]` — making `[].length < 2` always true and showing POReference even with 2+ records.
   The onChange now reads selection via the live `view.getSelection()` (wired to
   `controller.getSelection()` → `graph.getSelectedMultiple(tab)`). Both hooks remain guarded against
   absence. This was a real substrate defect, not just a style change; the original coverage row marked
   the dead path `supported` incorrectly and is now corrected.
4. **Window-`183` gate preserved verbatim.** The onChange only reveals fields when launched from the
   Purchase Invoice window (`view.windowId === '183'`). This is intentional Classic behavior (the Void
   fields apply to purchase-invoice voiding); reproduced 1:1, not generalized.
5. **No OK-button manipulation.** Unlike `ProcessOrders.onLoad`, the Invoices onLoad does not touch
   `view.okButton`, so that capability is not exercised here. The button enables itself reactively once
   `DocAction` has a value.
6. **Family cloning.** The `processRecords` family shares the `onLoad` shape; each process is migrated on
   its own row. This row's onLoad differs from the Orders/Shipment ones (handler `ProcessInvoicesDefaults`,
   `tableId = "318"`, field-hiding) and it is the only family member with a real parameter `onChange`. A
   future fix to family logic must be re-applied per row.
