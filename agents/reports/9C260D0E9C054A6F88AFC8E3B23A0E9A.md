## Generated code per field

### `em_etmeta_payscript_logic` — process (module body)
```
const PURCHASE_INVOICE = 'OBFBPS_PurchaseInvoice';
const SALES_INVOICE = 'OBFBPS_SalesInvoice';
const PURCHASE_AMOUNT = 'OBFBPS_PurchaseAmount';
const SALES_AMOUNT = 'OBFBPS_SalesAmount';
const PEND_AMOUNT = 'OBFBPS_PendAmount';
const AMOUNT_EXCEED_PENDING = 'OBFBPS_Err_AmountExceedPending';
const SETTLEMENT_AMOUNT = 'settlementAmount';

const getInvoiceGrid = (view, paramName) => view.theForm.getItem(paramName).canvas.viewGrid;

const getSelectedAmount = grid => {
  let amount = BigDecimal.prototype.ZERO;
  const records = grid.getSelectedRecords();
  records.forEach(record => {
    const index = grid.getRecordIndex(record);
    const item = grid.getEditValues(index);
    const editedAmount = item && item[SETTLEMENT_AMOUNT];
    const value = editedAmount ? editedAmount : record[SETTLEMENT_AMOUNT] || 0;
    amount = amount.add(new BigDecimal(String(value)));
  });
  return Number(amount.toString());
};

const checkInvoiceAmount = grid => {
  let result = null;
  const records = grid.getSelectedRecords();
  records.every(record => {
    const index = grid.getRecordIndex(record);
    const item = grid.getEditValues(index);
    const editedAmount = item && item[SETTLEMENT_AMOUNT];
    if (editedAmount && editedAmount > record.amount) {
      result = record.documentNo;
      return false;
    }
    return true;
  });
  return result;
};

const checkAmounts = (view, purchaseInvoiceGrid, salesInvoiceGrid) => {
  view.messageBar.hide();
  const grids = [purchaseInvoiceGrid, salesInvoiceGrid];
  for (let i = 0; i < grids.length; i++) {
    const invoice = checkInvoiceAmount(grids[i]);
    if (invoice) {
      view.messageBar.setMessage(
        isc.OBMessageBar.TYPE_ERROR,
        null,
        OB.I18N.getLabel(AMOUNT_EXCEED_PENDING, [invoice])
      );
      return false;
    }
  }
  return true;
};

const setAmounts = view => {
  const purchaseInvoiceGrid = getInvoiceGrid(view, PURCHASE_INVOICE);
  const salesInvoiceGrid = getInvoiceGrid(view, SALES_INVOICE);
  const amountPurchase = getSelectedAmount(purchaseInvoiceGrid);
  const amountSales = getSelectedAmount(salesInvoiceGrid);
  view.theForm.getItem(PURCHASE_AMOUNT).setValue(amountPurchase);
  view.theForm.getItem(SALES_AMOUNT).setValue(amountSales);
  view.theForm.getItem(PEND_AMOUNT).setValue(amountSales - amountPurchase);
};

return {
  PURCHASE_INVOICE,
  SALES_INVOICE,
  PURCHASE_AMOUNT,
  SALES_AMOUNT,
  getInvoiceGrid,
  getSelectedAmount,
  checkInvoiceAmount,
  checkAmounts,
  setAmounts
};
```

### `em_etmeta_onprocess` — process
```
async (process, view) => {
  const AMOUNT_NOT_EQUALS = 'OBFBPS_Err_AmountNotEquals';
  const SELECT_INVOICE = 'OBFBPS_Err_SelectInvoice';
  const purchaseInvoiceGrid = getInvoiceGrid(view, PURCHASE_INVOICE);
  const salesInvoiceGrid = getInvoiceGrid(view, SALES_INVOICE);
  const amountPurchase = getSelectedAmount(purchaseInvoiceGrid);
  const amountSales = getSelectedAmount(salesInvoiceGrid);
  view.theForm.getItem(PURCHASE_AMOUNT).setValue(amountPurchase);
  view.theForm.getItem(SALES_AMOUNT).setValue(amountSales);
  if (!checkAmounts(view, purchaseInvoiceGrid, salesInvoiceGrid)) {
    return { severity: 'error' };
  }
  if (amountPurchase !== amountSales) {
    const text = OB.I18N.getLabel(AMOUNT_NOT_EQUALS);
    view.messageBar.setMessage(isc.OBMessageBar.TYPE_ERROR, null, text);
    return { severity: 'error', text };
  }
  if (amountPurchase === 0) {
    const text = OB.I18N.getLabel(SELECT_INVOICE);
    view.messageBar.setMessage(isc.OBMessageBar.TYPE_ERROR, null, text);
    return { severity: 'error', text };
  }
  const response = await view.executeProcess();
  return response && response.message;
}
```

## Advisories (non-blocking)
- **Dead code dropped:** `doSelectionChanged(record, state)` and `amountChange(item, view, form, grid)`
  carry no metadata binding (no `onchangefunction`/`on_refresh_function`), so they are unreachable in
  the new UI and were not migrated. Their net effect (recompute totals + revalidate) is already covered
  by `setAmounts` on grid load and by `onProcess` validation. If, in Classic, recalculation occurred
  live on selection toggle or on amount edit, the new UI recomputes on grid (re)load and at submit
  instead — confirm during QA that the amount banner timing is acceptable.
- **`onLoad` left empty** intentionally (see field section).
- **`selectedIds` + `data.localData.find('id', id)` rewrite:** ported to `grid.getSelectedRecords()`,
  which returns the same selected rows; this is behaviorally equivalent and avoids the unsupported
  SmartClient `selectedIds` array and ResultSet `find(property, value)`.
- **OBFBPS cluster:** the sibling process *Add Credit Payments* (`DF7F70B82C514F639F06495E0B818A53`,
  `ob-obfbps-addpayments.js`) is currently `blocked` with empty columns, so no cluster merge was
  needed; this process's module scope is self-contained.

## Manual-test checklist
1. **Open dialog.** Launch *Add Invoices*. The Purchase Invoice and Sales Invoice grids load; the
   Purchase Amount, Sales Amount and Pending Amount fields are present. No error banner on open.
2. **Grid load recompute.** When each grid finishes loading, confirm Purchase Amount / Sales Amount /
   Pending Amount reflect the currently selected rows (Pending = Sales − Purchase). With nothing
   selected they show 0.
3. **Settlement amount edit > invoice amount.** In one grid, edit a row's settlement amount above its
   invoice amount and press Done. Expect an error banner
   `OBFBPS_Err_AmountExceedPending` naming that row's document number, and the dialog stays open.
4. **Purchase ≠ Sales.** Select rows so purchase total ≠ sales total and press Done. Expect the
   `OBFBPS_Err_AmountNotEquals` error banner; dialog stays open.
5. **Nothing selected.** With both totals 0, press Done. Expect the `OBFBPS_Err_SelectInvoice` error
   banner; dialog stays open.
6. **Valid submit.** Select purchase and sales rows with equal, non-zero totals within each invoice's
   amount and press Done. Expect the server process to run (`view.executeProcess`) and the resulting
   server message to be surfaced exactly once.
7. **Amount math parity.** Use amounts with decimals (e.g. 100.10 + 50.05) and confirm totals match the
   server-side computation (BigDecimal parity, no float drift).

## Updates

**Correction: this process is a Pick&Execute / Window-Reference process** (two Window Reference grids,
`OBFBPS_PurchaseInvoice` / `OBFBPS_SalesInvoice` ⇒ `hasWindowReference === true`). Its `onProcess`
validations therefore hit the **same substrate gap as `DF7F70B8` (Add Credit Payments)**: before
ETP-3748, `em_etmeta_onprocess` was never evaluated on the Done button for these processes, so the
"populated `em_etmeta_onprocess`" reported here would have been a dead path.

**Platform support implemented (ETP-3748).** `em_etmeta_onprocess` now runs as a **pre-submit
validation hook** before the Window-Reference submit (see `useProcessExecution.ts` `runPreSubmitValidation`;
`new-javascript-code.md` §5.2.2).

**What babel must do to confirm/unblock:**

- Re-verify the migrated `em_etmeta_onprocess` follows the **pre-submit contract**: `return { severity:
  'error', text }` to abort, `undefined` to proceed; **do not** call `view.executeProcess()` (the
  platform performs the submit for Window-Reference processes).
- The grid-load path is unaffected.
