## Process classification
This is a **Pick&Execute / Window-Reference process**: two Window Reference grid parameters
(`OBFBPS_PurchaseInvoice` / `OBFBPS_SalesInvoice`) ⇒ `hasWindowReference === true`. The Done button posts
the grid selection straight to the Java handler via `handleWindowReferenceExecute`. The migrated

below), not as a standard `onProcess` that drives the submit.

**Substrate evidence for the selection-toggle migration** (`client/packages/MainUI/components/ProcessModal/WindowReferenceGrid.tsx`):
- The grid proxy exposes `onSelectionToggle(fn)`: `if (!selectionToggleSubs.includes(fn)) selectionToggleSubs.push(fn);` — registration is **reference-deduped**.
- The selection effect fires on every `rowSelection` change; when `selectionToggleSubsRef.current.length > 0` it calls `emitSelectionToggles(selection)`.
- `emitSelectionToggles` diffs previous vs current selection and calls `fn(record, state)` once per added (`true`) / removed (`false`) id, for every registered subscriber.
- `evaluateModuleScope` runs the module body **once** per modal open (`ProcessDefinitionModal.tsx`, memoized), so `recomputeAmounts` is one stable reference passed to every grid (re)load → `.includes` collapses repeats to a single registration per grid handle. No subscriber accumulation across reloads.

**Substrate evidence for the `amountChange` migration** (`client/packages/MainUI/components/ProcessModal/WindowReferenceGrid.tsx`):
- The grid proxy exposes `setColumnOnChange(colName, fn)`: `columnOnChange.set(colName, fn)` — a `Map.set`, so re-registering the same column key is idempotent (replaces, never accumulates); passing the single stable `onSettlementAmountChange` reference from both grids is safe.
- `fireCellEditHooks` is the cell-edit hook dispatcher: it loops `for (const colName of Object.keys(changes))`, looks up `columnOnChangeRef.current.get(colName)`, and invokes it as `onChange(item, view, view.theForm, grid)` — **identical** to classic `amountChange(item, view, form, grid)`. The re-entrancy guard `inColumnOnChangeRef` prevents a column onChange that writes grid cells from looping.
- `handleRecordChange` is the cell-edit entry point and calls `fireCellEditHooks`; it is wired into the editor via `StableGridCellEditorRenderer`'s `onRecordChange={handleRecordChangeRef.current}`.

The registration key and the dispatch key agree — no ambiguity.

## Design note — Window-Reference pre-submit hook (ETP-3748)
For Pick&Execute / Window-Reference processes the platform performs the submit itself; `em_etmeta_onprocess`
runs as a **pre-submit validation hook** before that submit (`new-javascript-code.md` §5.2.2). Confirmed
in `client/packages/MainUI/components/ProcessModal/hooks/useProcessExecution.ts`:

- `handleExecute` — the `if (hasWindowReference)` branch runs `em_etmeta_onprocess` via
  `runPreSubmitValidation(actionValue)` **before** `handleWindowReferenceExecute(actionValue)`. If it
  returns `false`, the function returns early and never submits.
- `runPreSubmitValidation` — executes `etmetaOnprocess`, reads `extractResponseMessage`; on
  `msgType === 'error'` it shows the message bar, sets the failed result, and returns `false` (abort);
  otherwise returns `true` (proceed). It never calls `view.executeProcess()`.
- `buildOnProcessView` — builds the canonical `view` proxy (`theForm`, `messageBar`, `gridResolver`
  → `getItem('<grid>').canvas.viewGrid`, etc.), so the migrated script reads the grids and writes the
  amount fields exactly as in Classic.
- `extractResponseMessage` — maps a returned `{ severity, text }` to `{ msgType, msgText }`, so
  `return { severity: 'error', text }` aborts and `return undefined` proceeds.
- `shouldRunProcessLifecycleHooks({ tab, callerField })` — gates the hook (top-level open only).

**Contract for the hook:** `async (process, view) => {...}`; `return { severity: 'error', text }` to abort,
`return undefined` to proceed. It must **not** call `view.executeProcess()` — the platform performs the
Window-Reference submit.

**Fidelity of the Classic `onProcess` under this contract.** The Classic `onProcess`
(`ob-obfbps-addinvoices.js` lines 92–139) does exactly:

1. Reads both Window-Reference grids and recomputes `amountPurchase` / `amountSales`.
2. `purchaseAmount.setValue(...)` / `salesAmount.setValue(...)` — **display refresh** of two read-only
   amount fields (pure UI sync; no effect on the submit).
3. `checkAmounts(...)` → on a row whose edited settlement amount exceeds its invoice amount: error banner
   `OBFBPS_Err_AmountExceedPending` + `clientSideValidationFail()` → **abort**.
4. `amountPurchase !== amountSales` → error banner `OBFBPS_Err_AmountNotEquals` +
   `clientSideValidationFail()` → **abort**.
5. `amountPurchase === 0` → error banner `OBFBPS_Err_SelectInvoice` + `clientSideValidationFail()` →
   **abort**.
6. Otherwise → `actionHandlerCall(view)` → let the **platform** submit the grid selection.

Every terminal branch is either `clientSideValidationFail()` (a pure client-side abort) or
`actionHandlerCall(view)` (hand control to the platform submit). The only non-abort side effect is the two
display `setValue` calls, which the pre-submit hook performs verbatim through `view.theForm`. The script
does **no** work the validation-only hook cannot express and does **not** itself perform the action.
Therefore it maps faithfully: the three aborts return `{ severity: 'error', text }`, the valid branch
returns `undefined`.

## Generated code per field

### `em_etmeta_payscript_logic` — process (module body)
```
const PURCHASE_INVOICE = 'OBFBPS_PurchaseInvoice';
const SALES_INVOICE = 'OBFBPS_SalesInvoice';
const PURCHASE_AMOUNT = 'OBFBPS_PurchaseAmount';
const SALES_AMOUNT = 'OBFBPS_SalesAmount';
const PEND_AMOUNT = 'OBFBPS_PendAmount';
const AMOUNT_EXCEED_PENDING = 'OBFBPS_Err_AmountExceedPending';
const SETTLEMENT_AMOUNT = 'settlementAmount';

const getInvoiceGrid = (view, paramName) => view.theForm.getItem(paramName).canvas.viewGrid;

const getSelectedAmount = grid => {
  let amount = BigDecimal.prototype.ZERO;
  const records = grid.getSelectedRecords();
  records.forEach(record => {
    const index = grid.getRecordIndex(record);
    const item = grid.getEditValues(index);
    const editedAmount = item && item[SETTLEMENT_AMOUNT];
    const value = editedAmount ? editedAmount : record[SETTLEMENT_AMOUNT] || 0;
    amount = amount.add(new BigDecimal(String(value)));
  });
  return Number(amount.toString());
};

const checkInvoiceAmount = grid => {
  let result = null;
  const records = grid.getSelectedRecords();
  records.every(record => {
    const index = grid.getRecordIndex(record);
    const item = grid.getEditValues(index);
    const editedAmount = item && item[SETTLEMENT_AMOUNT];
    if (editedAmount && editedAmount > record.amount) {
      result = record.documentNo;
      return false;
    }
    return true;
  });
  return result;
};

const checkAmounts = (view, purchaseInvoiceGrid, salesInvoiceGrid) => {
  view.messageBar.hide();
  const grids = [purchaseInvoiceGrid, salesInvoiceGrid];
  for (let i = 0; i < grids.length; i++) {
    const invoice = checkInvoiceAmount(grids[i]);
    if (invoice) {
      const text = OB.I18N.getLabel(AMOUNT_EXCEED_PENDING, [invoice]);
      view.messageBar.setMessage(isc.OBMessageBar.TYPE_ERROR, null, text);
      return invoice;
    }
  }
  return null;
};

const setAmounts = view => {
  const purchaseInvoiceGrid = getInvoiceGrid(view, PURCHASE_INVOICE);
  const salesInvoiceGrid = getInvoiceGrid(view, SALES_INVOICE);
  const amountPurchase = getSelectedAmount(purchaseInvoiceGrid);
  const amountSales = getSelectedAmount(salesInvoiceGrid);
  view.theForm.getItem(PURCHASE_AMOUNT).setValue(amountPurchase);
  view.theForm.getItem(SALES_AMOUNT).setValue(amountSales);
  view.theForm.getItem(PEND_AMOUNT).setValue(amountSales - amountPurchase);
};

// Live selection-toggle recompute. The module body runs once per modal open, so
// recomputeAmounts is a single stable reference; passing that same reference to
// every grid's grid.onSelectionToggle lets the substrate's reference-dedup
// collapse repeated registrations across grid reloads. It reads the latest view
// captured on grid load, so it always recomputes against a current view proxy.
let lastView = null;
const setLastView = view => { lastView = view; };
const recomputeAmounts = () => { if (lastView) setAmounts(lastView); };

// Live recompute when the user edits a row's Settlement Amount cell. Ported
// 1:1 from classic amountChange(item, view, form, grid): recompute every total
// (setAmounts) and revalidate the over-pending banner (checkAmounts), ignoring
// the return value exactly as classic did. Registered via grid.setColumnOnChange
// under SETTLEMENT_AMOUNT, so it is one stable module-scope reference reused by
// both grids' on_grid_load (re-registration is idempotent by column key).
const onSettlementAmountChange = (item, view, form, grid) => {
  setAmounts(view);
  const purchaseInvoiceGrid = getInvoiceGrid(view, PURCHASE_INVOICE);
  const salesInvoiceGrid = getInvoiceGrid(view, SALES_INVOICE);
  checkAmounts(view, purchaseInvoiceGrid, salesInvoiceGrid);
};

return {
  PURCHASE_INVOICE,
  SALES_INVOICE,
  PURCHASE_AMOUNT,
  SALES_AMOUNT,
  AMOUNT_EXCEED_PENDING,
  SETTLEMENT_AMOUNT,
  getInvoiceGrid,
  getSelectedAmount,
  checkInvoiceAmount,
  checkAmounts,
  setAmounts,
  setLastView,
  recomputeAmounts,
  onSettlementAmountChange
};
```

### `em_etmeta_onprocess` — process (pre-submit validation hook)
The three aborts return `{ severity: 'error', text }`; the valid branch returns `undefined` so the platform
performs the Window-Reference submit. In the over-pending branch the informative document text is returned
so `extractResponseMessage` re-surfaces the same `OBFBPS_Err_AmountExceedPending` message `checkAmounts`
already showed (see the note below). Compile-checked as a bare arrow function.
```
async (process, view) => {
  const AMOUNT_NOT_EQUALS = 'OBFBPS_Err_AmountNotEquals';
  const SELECT_INVOICE = 'OBFBPS_Err_SelectInvoice';
  const purchaseInvoiceGrid = getInvoiceGrid(view, PURCHASE_INVOICE);
  const salesInvoiceGrid = getInvoiceGrid(view, SALES_INVOICE);
  const amountPurchase = getSelectedAmount(purchaseInvoiceGrid);
  const amountSales = getSelectedAmount(salesInvoiceGrid);
  view.theForm.getItem(PURCHASE_AMOUNT).setValue(amountPurchase);
  view.theForm.getItem(SALES_AMOUNT).setValue(amountSales);
  const exceedInvoice = checkAmounts(view, purchaseInvoiceGrid, salesInvoiceGrid);
  if (exceedInvoice) {
    return { severity: 'error', text: OB.I18N.getLabel(AMOUNT_EXCEED_PENDING, [exceedInvoice]) };
  }
  if (amountPurchase !== amountSales) {
    const text = OB.I18N.getLabel(AMOUNT_NOT_EQUALS);
    view.messageBar.setMessage(isc.OBMessageBar.TYPE_ERROR, null, text);
    return { severity: 'error', text };
  }
  if (amountPurchase === 0) {
    const text = OB.I18N.getLabel(SELECT_INVOICE);
    view.messageBar.setMessage(isc.OBMessageBar.TYPE_ERROR, null, text);
    return { severity: 'error', text };
  }
  return undefined;
}
```

**Note on the `checkAmounts` over-pending branch.** `checkAmounts` sets the informative banner
`OBFBPS_Err_AmountExceedPending [documentNo]` and returns the offending document number (truthy) or `null`.
The hook returns that text via `{ severity: 'error', text }`, so `extractResponseMessage` re-surfaces the
same message — there is no blank overwrite. (Returning `{ severity: 'error' }` with no `text` would cause
`runPreSubmitValidation` to call `messageBar.setMessage('error', null, undefined)`, which — since
`setMessage` does `html: sanitizeMessageHtml(text ?? "")` and **replaces** the current message — would blank
out the informative banner. Returning the text avoids that.) `AMOUNT_EXCEED_PENDING` is exported from
`em_etmeta_payscript_logic` so the hook can build the label.

## Advisories (non-blocking)
- **`doSelectionChanged` is now migrated (no longer dropped).** Classic recomputes the amount fields live
  on every selection toggle of the Window-Reference grids via `doSelectionChanged(record, state) ->
  setAmounts()`. The earlier migration dropped this as "unbound dead code" and only wired `setAmounts(view)`
  into `em_etmeta_on_grid_load`, which fires once per datasource payload — selecting a row does NOT refetch
  the datasource, so the live selection-driven recompute was lost. The substrate has since gained the
  additive per-toggle subscriber `grid.onSelectionToggle((record, state) => …)` (the 1:1 equivalent of
  classic `doSelectionChanged`), so the recompute is now wired by subscribing `recomputeAmounts` from both
  grids' `em_etmeta_on_grid_load`. Classic `doSelectionChanged` ignores its `(record, state)` args and just
  calls `setAmounts()` (a full recompute), so the subscriber ignores its args too — **no per-record
  incremental logic was dropped**. Duplicate-subscription risk (a fresh inline arrow per grid reload would
  accumulate, since `.includes` dedups by reference) is avoided by passing the single module-scope
  `recomputeAmounts` reference, which the substrate's reference-dedup collapses to one registration per
  grid handle.
- **`amountChange` is now migrated (no longer dropped).** Classic `amountChange(item, view, form, grid)`
  (`ob-obfbps-addinvoices.js:177-185`) recomputes all totals (`setAmounts()`) and revalidates the
  over-pending banner (`checkAmounts()`) live when the user edits a row's Settlement Amount cell. It carries
  no parameter `onchangefunction` (re-verified: `em_etmeta_on_parameter_change` null on all 8 parameters), so
  it is not wired through a parameter onChange column — instead it is migrated as a per-column cell-edit hook
  via the additive substrate API `grid.setColumnOnChange(SETTLEMENT_AMOUNT, onSettlementAmountChange)`,
  registered from both grids' `em_etmeta_on_grid_load`. The substrate's `fireCellEditHooks` invokes it as
  `fn(item, view, view.theForm, grid)` (signature identical to classic) on every Settlement Amount cell
  edit, so the live in-cell-edit recompute is now faithfully reproduced (no deselect/reselect needed). The
  classic body ignores its `(item, ...)` args and just calls `setAmounts()` + `checkAmounts()`, and the
  ported handler does the same — no per-cell incremental logic was dropped. The single stable module-scope
  reference is reused by both grids; `setColumnOnChange` is a `Map.set` keyed by column, so re-registration
  on grid reload is idempotent (no accumulation). The change is faithful: `checkAmounts` is called for its
  banner side-effect (hide-then-maybe-show `OBFBPS_Err_AmountExceedPending`) exactly as classic did,
  ignoring its return value.
- **`onLoad` left empty** intentionally (see field section).
- **`selectedIds` + `data.localData.find('id', id)` rewrite:** ported to `grid.getSelectedRecords()`,
  which returns the same selected rows; this is behaviorally equivalent and avoids the unsupported
  SmartClient `selectedIds` array and ResultSet `find(property, value)`.
- **`em_etmeta_on_grid_load` sees `rowCount = 0` on open** because record seeding does not go through the
  `onDataArrived` path. This is a display-timing fidelity note only, not a functional gap: the migrated
  `grid.onSelectionToggle(recomputeAmounts)` and `grid.setColumnOnChange(...)` recompute the totals on the
  first selection toggle / settlement-amount edit, and the pre-submit `em_etmeta_onprocess` reads
  `record.amount` directly off the grid rows at submit.
- **OBFBPS cluster:** the sibling process *Add Credit Payments* (`DF7F70B82C514F639F06495E0B818A53`,
  `ob-obfbps-addpayments.js`) has empty columns, so no cluster merge was needed; this process's module
  scope is self-contained.
