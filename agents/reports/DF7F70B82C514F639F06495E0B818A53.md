### 3.1 Why `em_etmeta_onprocess` is unsupported here (substrate trace)

This is `uipattern = OBUIAPP_PickAndExecute` and has two Window Reference grid parameters (Credit In /
Credit Out, reference `FF80818132D8F0F30132D9BC395D0038`). At runtime:

- `isPickAndExecute(process)` (`utils/processes/definition/pickAndExecute.ts:38-42`) returns `true`
  (matches both signals: `uIPattern === 'OBUIAPP_PickAndExecute'` **and** a Window Reference parameter).
- `ProcessDefinitionModal.tsx:1096` sets `hasWindowReference = isPE`, where `isPE = isPickAndExecute(...)`
  (`:552`). So `hasWindowReference` is `true`.
- In `useProcessExecution.ts → handleExecute` (~line 713), the Done button hits:
  `if (hasWindowReference) { await handleWindowReferenceExecute(actionValue); return; }` (~line 722).
  This **returns** before the only site that evaluates the column:
  `executeStringFunction(etmetaOnprocess, …)` (~line 732+).
- `handleWindowReferenceExecute` (~line 575) builds the P&E payload and posts directly to
  `AddCreditPaymentsActionHandler` via `executeJavaProcess`, never reading `em_etmeta_onprocess`.
- `em_etmeta_custom_component` is empty (`has_custom = f`), so the `GenericWarehouseProcess` custom
  component path (the only other evaluator of `etmetaOnprocess`) does not apply either.

**Consequence:** any code pasted into `em_etmeta_onprocess` for this process is dead — it would never
run on the Done button. The three Classic client-side validations that `onProcess` performed via
`clientsidevalidation` (exceed-pending → `OBFBPS_Err_CreditExceedPending`, credit-in≠credit-out →
`OBFBPS_Err_CreditNotEquals`, zero-credit → `OBFBPS_Err_SelectCredit`) therefore have **no faithful
migration target** in the current substrate. Emitting them as "paste this" would silently drop real,
observable Classic behavior, so per the conservative feasibility gate the column is left empty and
flagged BLOCKED pending a substrate fix (i.e. the runtime evaluating `em_etmeta_onprocess` — or an
equivalent pre-submit hook — before `handleWindowReferenceExecute`).

> Note the contrast: `em_etmeta_on_grid_load` **is** wired for P&E grids
> (`WindowReferenceGrid.tsx:2152` invokes `onGridLoadHook(grid, view, parameters)`), so the grid-load
> deliverables and the payscript helpers they depend on are valid.

## 4. Code to paste (per column)

### `em_etmeta_payscript_logic` — entity `obuiapp_process`

This module is now scoped to the **grid-load path only** (the `em_etmeta_onprocess` path is BLOCKED,
so its exclusive helpers `checkCreditAmount` and `checkAmounts` are dropped). The module stands on its
own: `setAmounts` depends only on `getGrid` and `getSelectedAmount`, all defined here.

```js
const CREDIT_IN_GRID = 'OBFBPS_CreditIn';
const CREDIT_OUT_GRID = 'OBFBPS_CreditOut';
const CREDIT_IN_AMOUNT = 'OBFBPS_CreditInAmount';
const CREDIT_OUT_AMOUNT = 'OBFBPS_CreditOutAmount';
const CREDIT_PEND_AMOUNT = 'OBFBPS_CreditPendAmount';

const getGrid = (view, paramName) => view.theForm.getItem(paramName).canvas.viewGrid;

const getSelectedAmount = grid => {
  let amount = BigDecimal.prototype.ZERO;
  grid.getSelectedRecords().forEach(record => {
    const index = grid.getRecordIndex(record);
    const editValues = grid.getEditValues(index);
    const settlementAmount = editValues.settlementAmount
      ? editValues.settlementAmount
      : grid.getRecord(index).settlementAmount || 0;
    amount = amount.add(new BigDecimal(String(settlementAmount)));
  });
  return Number(amount);
};

const setAmounts = view => {
  const creditInGrid = getGrid(view, CREDIT_IN_GRID);
  const creditOutGrid = getGrid(view, CREDIT_OUT_GRID);
  const amountCreditIn = getSelectedAmount(creditInGrid);
  const amountCreditOut = getSelectedAmount(creditOutGrid);
  view.theForm.getItem(CREDIT_IN_AMOUNT).setValue(amountCreditIn);
  view.theForm.getItem(CREDIT_OUT_AMOUNT).setValue(amountCreditOut);
  view.theForm.getItem(CREDIT_PEND_AMOUNT).setValue(amountCreditOut - amountCreditIn);
};

return {
  CREDIT_IN_GRID,
  CREDIT_OUT_GRID,
  CREDIT_IN_AMOUNT,
  CREDIT_OUT_AMOUNT,
  getGrid,
  getSelectedAmount,
  setAmounts,
};
```

## 5. Advisories (non-blocking)

- **A1 — selection idiom translated.** Classic iterates `grid.selectedIds` and resolves each row with
  `grid.data.localData.find('id', id)`. The new-UI grid proxy exposes `getSelectedRecords()` (full
  records) but not `selectedIds`, and `data.localData` is a plain JS array (the SmartClient two-arg
  `Array.find(field, value)` form is not native). The migration iterates `getSelectedRecords()` and
  derives `index = getRecordIndex(record)`, then reads `getEditValues(index)` / `getRecord(index)` — the
  same supported accessors, same per-row data, same result. No behavior change.
- **A2 — dead code dropped.** `doSelectionChanged` and `amountChange` are not referenced by any metadata
  binding (no parameter has `onchangefunction`) nor by any in-scope function, so they are not migrated.
  In Classic the live amount recompute on selection/edit flows through the grids' `ongridloadfunction`
  (`onLoad*Grid → setAmounts`) and the pre-submit `onProcess`; this parity is preserved.
- **A3 — `onLoad` is empty by design.** The Classic `onLoad` only stored `view` in the namespace so the
  helpers could read it later. Every new-UI hook receives `view` as an argument and the helpers take it
  explicitly, so no caching is needed and `em_etmeta_onload` stays empty. No observable behavior is lost.
- **A4 — `clientSideValidationFail()` is NOT migratable here (substrate block).** In Classic the three
  pre-submit guards aborted the Done submit via `clientSideValidationFail()` inside `onProcess`. In the
  current substrate `em_etmeta_onprocess` is never evaluated for this P&E / Window Reference process
  (Section 3.1), so there is no hook in which a client-side abort can fire before
  `handleWindowReferenceExecute` posts to the Java handler. These validations are therefore **dropped from
  the deliverable** and the behavior cannot be reproduced today. A faithful migration requires a substrate
  change (a pre-submit hook that runs before the Window-Reference submit). Until then, only the Java
  handler's server-side checks (if any) apply on Done.
- **A5 — `actionHandlerCall(view)` is NOT migratable here (substrate block).** It would map to
  `view.executeProcess()`, but since `em_etmeta_onprocess` is bypassed for P&E / Window Reference
  processes, that call site never runs. The actual submit is performed by the substrate's
  `handleWindowReferenceExecute` → `executeJavaProcess(payload, 'process')`, which posts straight to
  `AddCreditPaymentsActionHandler` independently of any migrated onProcess. No onProcess code is needed
  (or possible) for the submit itself.
- **A7 — substrate gap to record (self-improvement).** `new-javascript-code.md` does not flag that
  `em_etmeta_onprocess` is unreachable for `OBUIAPP_PickAndExecute` / Window Reference processes
  (`hasWindowReference === true`). This caused an initial over-optimistic "supported" classification.
  Recommend the architecture doc explicitly document: for P&E / Window-Reference processes, the only
  wired client-side hooks are `em_etmeta_on_grid_load` (and grid-level hooks); `em_etmeta_onprocess`
  (and thus migrated pre-submit `clientsidevalidation`) is currently a dead path. Babel does not edit the
  architecture doc unilaterally; this is surfaced for the human to decide.
- **A6 — no `qa-passed` reference available.** The inventory currently lists no process in `qa-passed`
  state, so this migration relies on the architecture doc directly. The REM `ongridloadfunction` cluster
  (`C88AB6CBA1694000AFF5706A31B08AE1`, status `migrated`) uses the same `BigDecimal` + embedded-grid
  patterns and was used only as a style hint, not as an authoritative reference.

## 6. Manual-test checklist

Open the **Add Credit Payments** process (Pick & Execute modal) from its launch context.

1. **Modal opens cleanly.** No console error; the two grids (Credit In, Credit Out) and the three amount
   fields (Credit In Amount, Credit Out Amount, Credit Pending) render. (onLoad is empty — verify there
   is no regression from it being empty.)
2. **Grid load recomputes amounts.** When each grid finishes loading rows, Credit In Amount, Credit Out
   Amount and Credit Pending are recomputed from the current selection (initially 0 / 0 / 0 if nothing is
   pre-selected). `Credit Pending = Credit Out Amount − Credit In Amount`.
3. **Selecting rows updates totals.** Select one or more rows in **Credit In**: Credit In Amount sums the
   selected rows' settlement amounts (using the edited value when present, else the row's
   `settlementAmount`). Do the same in **Credit Out**. Credit Pending updates to `out − in`.
4. **Edited settlement amount is honored.** Edit a row's settlement amount cell in a selected row; the
   totals reflect the edited value (not the original row value).
5. **Happy path / submit.** Select rows so Credit In Amount == Credit Out Amount and both > 0 and within
   pending, press Done. The process submits to `AddCreditPaymentsActionHandler` and the result is shown.
   (The submit goes through the substrate's Window-Reference P&E path, not through any migrated onProcess.)

## Updates

**Platform support implemented (ETP-3748) — `em_etmeta_onprocess` is now reachable for P&E / Window-Reference processes.**

`useProcessExecution.ts` (`handleExecute`) now evaluates `em_etmeta_onprocess` as a **pre-submit
validation hook** *before* `handleWindowReferenceExecute` posts to the Java handler. A shared
`buildOnProcessView` builds the same `view` proxy for both paths; the new `runPreSubmitValidation`
runs the script and, if it returns an `error` severity, shows the message and **aborts the submit**;
returning `undefined` lets the submit proceed. This resolves advisories A4 / A5 / A7 (the substrate
gap that made the column a dead path). Documented in `new-javascript-code.md` §5.2.2.

**What babel must do to unblock this process (status `migrated (partial)` → `migrated`):**

- Populate `em_etmeta_onprocess` with the three pre-submit guards (`OBFBPS_Err_CreditExceedPending`,
  `OBFBPS_Err_CreditNotEquals`, `OBFBPS_Err_SelectCredit`) as a **validation-only** hook:
  `return { severity: 'error', text }` to abort, `undefined` to proceed.
- Do **not** call `view.executeProcess()` in this hook — the platform performs the Window-Reference
  submit. The helpers `checkCreditAmount` / `checkAmounts` move into `em_etmeta_payscript_logic`.
- The grid-load path (`em_etmeta_on_grid_load` + `em_etmeta_payscript_logic`) is unchanged.
