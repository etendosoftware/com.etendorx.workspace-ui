### 3.1 Why `em_etmeta_onprocess` is now supported here (substrate trace)

This is `uipattern = OBUIAPP_PickAndExecute` and has two Window Reference grid parameters (Credit In /
Credit Out), so `hasWindowReference === true` at runtime. With ETP-3748 the substrate evaluates
`em_etmeta_onprocess` as a **pre-submit validation hook** before submitting:

- In `useProcessExecution.ts → handleExecute` (~line 818) the Done button now hits, inside the
  `if (hasWindowReference)` branch (~line 827):
  ```
  if (etmetaOnprocess && shouldRunProcessLifecycleHooks({ tab, callerField })) {
    const canProceed = await runPreSubmitValidation(actionValue);
    if (!canProceed) return; // message already shown; do not submit
  }
  await handleWindowReferenceExecute(actionValue);
  ```
  i.e. the migrated `em_etmeta_onprocess` runs **before** `handleWindowReferenceExecute`.
- `runPreSubmitValidation` (~line 781) builds the `view` proxy via the shared `buildOnProcessView`
  (~line 720, identical to the standard path), runs the script with `executeStringFunction`, then
  reads the result via `extractResponseMessage`. If `msgType === 'error'` it calls
  `messageBar.setMessage('error', null, msgText)` and returns `false` → the submit is **aborted**.
  A non-error / `undefined` return resolves to `true` → the submit **proceeds** to
  `handleWindowReferenceExecute`.
- The gate `shouldRunProcessLifecycleHooks({ tab, callerField })`
  (`utils/processes/definition/processLifecycle.ts:36`) returns `true` for a top-level launch
  (tab present), so the hook fires for the normal P&E open. (Inside a nested modal `tab` is undefined
  and a `callerField` is present → the hook would not fire; this process is launched top-level.)

**Return-value contract (important for the deliverable):** the substrate **already shows** the returned
error message. The migrated hook therefore returns `{ severity: 'error', text }` and does **not** call
`view.messageBar.setMessage(...)` for the error itself — it only calls `view.messageBar.hide()` once at
the start to clear any stale banner (mirroring the Classic `checkAmounts` `messageBar.hide()`),
matching Classic with a single visible message and no double display.

**Consequence:** the three Classic client-side validations (`OBFBPS_Err_CreditExceedPending`,
`OBFBPS_Err_CreditNotEquals`, `OBFBPS_Err_SelectCredit`) have a faithful migration target and are
delivered in `em_etmeta_onprocess` (logic in `validateAmounts` / `getExceedingDocumentNo` in
`em_etmeta_payscript_logic`). The hook does **not** call `view.executeProcess()` — the platform performs
the Window-Reference submit.

> The grid-load path is also wired for P&E grids (`WindowReferenceGrid.tsx` invokes
> `onGridLoadHook(grid, view, parameters)`), so the grid-load deliverables and the payscript helpers
> they depend on are valid and unchanged.

## 4. Code to paste (per column)

### `em_etmeta_payscript_logic` — entity `obuiapp_process`

This module now serves **both** paths: the grid-load path (`setAmounts` via `getGrid` /
`getSelectedAmount`) and the onProcess pre-submit validation path (`validateAmounts` via
`getExceedingDocumentNo` — the migration of Classic `checkAmounts` / `checkCreditAmount`). The module
stands on its own; every helper resolves by bare name in all hooks.

```js
const CREDIT_IN_GRID = 'OBFBPS_CreditIn';
const CREDIT_OUT_GRID = 'OBFBPS_CreditOut';
const CREDIT_IN_AMOUNT = 'OBFBPS_CreditInAmount';
const CREDIT_OUT_AMOUNT = 'OBFBPS_CreditOutAmount';
const CREDIT_PEND_AMOUNT = 'OBFBPS_CreditPendAmount';

const ERR_EXCEED_PENDING = 'OBFBPS_Err_CreditExceedPending';
const ERR_NOT_EQUALS = 'OBFBPS_Err_CreditNotEquals';
const ERR_SELECT_CREDIT = 'OBFBPS_Err_SelectCredit';

const getGrid = (view, paramName) => view.theForm.getItem(paramName).canvas.viewGrid;

const getSelectedAmount = grid => {
  let amount = BigDecimal.prototype.ZERO;
  grid.getSelectedRecords().forEach(record => {
    const index = grid.getRecordIndex(record);
    const editValues = grid.getEditValues(index);
    const settlementAmount = editValues.settlementAmount
      ? editValues.settlementAmount
      : grid.getRecord(index).settlementAmount || 0;
    amount = amount.add(new BigDecimal(String(settlementAmount)));
  });
  return Number(amount);
};

const setAmounts = view => {
  const creditInGrid = getGrid(view, CREDIT_IN_GRID);
  const creditOutGrid = getGrid(view, CREDIT_OUT_GRID);
  const amountCreditIn = getSelectedAmount(creditInGrid);
  const amountCreditOut = getSelectedAmount(creditOutGrid);
  view.theForm.getItem(CREDIT_IN_AMOUNT).setValue(amountCreditIn);
  view.theForm.getItem(CREDIT_OUT_AMOUNT).setValue(amountCreditOut);
  view.theForm.getItem(CREDIT_PEND_AMOUNT).setValue(amountCreditOut - amountCreditIn);
};

const getExceedingDocumentNo = grid => {
  let documentNo = null;
  grid.getSelectedRecords().every(record => {
    const index = grid.getRecordIndex(record);
    const editValues = grid.getEditValues(index);
    if (editValues.settlementAmount && editValues.settlementAmount > grid.getRecord(index).outstandingAmount) {
      documentNo = grid.getRecord(index).documentNo;
      return false;
    }
    return true;
  });
  return documentNo;
};

const validateAmounts = view => {
  const creditInGrid = getGrid(view, CREDIT_IN_GRID);
  const creditOutGrid = getGrid(view, CREDIT_OUT_GRID);
  const amountCreditIn = getSelectedAmount(creditInGrid);
  const amountCreditOut = getSelectedAmount(creditOutGrid);

  const exceedIn = getExceedingDocumentNo(creditInGrid);
  if (exceedIn) {
    return { severity: 'error', text: OB.I18N.getLabel(ERR_EXCEED_PENDING, [exceedIn]) };
  }
  const exceedOut = getExceedingDocumentNo(creditOutGrid);
  if (exceedOut) {
    return { severity: 'error', text: OB.I18N.getLabel(ERR_EXCEED_PENDING, [exceedOut]) };
  }
  if (amountCreditIn !== amountCreditOut) {
    return { severity: 'error', text: OB.I18N.getLabel(ERR_NOT_EQUALS) };
  }
  if (amountCreditIn === 0) {
    return { severity: 'error', text: OB.I18N.getLabel(ERR_SELECT_CREDIT) };
  }
  return null;
};

return {
  CREDIT_IN_GRID,
  CREDIT_OUT_GRID,
  CREDIT_IN_AMOUNT,
  CREDIT_OUT_AMOUNT,
  CREDIT_PEND_AMOUNT,
  getGrid,
  getSelectedAmount,
  setAmounts,
  getExceedingDocumentNo,
  validateAmounts,
};
```

## 5. Advisories (non-blocking)

- **A1 — selection idiom translated.** Classic iterates `grid.selectedIds` and resolves each row with
  `grid.data.localData.find('id', id)`. The new-UI grid proxy exposes `getSelectedRecords()` (full
  records) but not `selectedIds`, and `data.localData` is a plain JS array (the SmartClient two-arg
  `Array.find(field, value)` form is not native). The migration iterates `getSelectedRecords()` and
  derives `index = getRecordIndex(record)`, then reads `getEditValues(index)` / `getRecord(index)` — the
  same supported accessors, same per-row data, same result. No behavior change.
- **A2 — dead code dropped.** `doSelectionChanged` and `amountChange` are not referenced by any metadata
  binding (no parameter has `onchangefunction`) nor by any in-scope function, so they are not migrated.
  In Classic the live amount recompute on selection/edit flows through the grids' `ongridloadfunction`
  (`onLoad*Grid → setAmounts`) and the pre-submit `onProcess`; this parity is preserved.
- **A3 — `onLoad` is empty by design.** The Classic `onLoad` only stored `view` in the namespace so the
  helpers could read it later. Every new-UI hook receives `view` as an argument and the helpers take it
  explicitly, so no caching is needed and `em_etmeta_onload` stays empty. No observable behavior is lost.
- **A4 — `clientSideValidationFail()` is now migratable (substrate fixed, ETP-3748).** The three
  pre-submit guards abort the Done submit by returning `{ severity:'error', text }` from
  `em_etmeta_onprocess`, which `runPreSubmitValidation` honors (`return false` -> no submit) and shows.
  Order preserved from Classic: exceed-pending (`OBFBPS_Err_CreditExceedPending`, Credit In then Credit
  Out) -> credit-in != credit-out (`OBFBPS_Err_CreditNotEquals`) -> zero-credit
  (`OBFBPS_Err_SelectCredit`). No behavior dropped.
- **A4b — single-message, no double display.** The substrate auto-shows the returned error
  (`messageBar.setMessage('error', null, msgText)`). To avoid showing the banner twice, the migrated
  hook does **not** call `setMessage` for the error; it returns it. It calls `messageBar.hide()` once
  at the start, mirroring Classic `checkAmounts`'s leading `messageBar.hide()`. Net result is the same
  single visible error Classic produced.
- **A5 — `actionHandlerCall(view)` is intentionally NOT emitted (correct for P&E).** For a
  Window-Reference process the platform performs the submit itself (`handleWindowReferenceExecute` ->
  `AddCreditPaymentsActionHandler`). Per architecture-doc Section 5.2.2 the pre-submit hook must **not**
  call `view.executeProcess()` (it would double-submit). So `actionHandlerCall` has no counterpart in
  the migrated `onProcess` — this is by design, not a gap.
- **A7 — substrate gap RESOLVED.** The earlier report flagged that `em_etmeta_onprocess` was unreachable
  for `OBUIAPP_PickAndExecute` / Window-Reference processes. ETP-3748 added `runPreSubmitValidation`,
  and `new-javascript-code.md` §5.2.2 now documents the pre-submit-hook semantics. No outstanding
  doc/substrate gap remains for this process.
- **A6 — no `qa-passed` reference available.** The inventory currently lists no process in `qa-passed`
  state, so this migration relies on the architecture doc directly. The REM `ongridloadfunction` cluster
  (`C88AB6CBA1694000AFF5706A31B08AE1`, status `migrated`) uses the same `BigDecimal` + embedded-grid
  patterns and was used only as a style hint, not as an authoritative reference.

## 6. Manual-test checklist

Open the **Add Credit Payments** process (Pick & Execute modal) from its launch context.

1. **Modal opens cleanly.** No console error; the two grids (Credit In, Credit Out) and the three amount
   fields (Credit In Amount, Credit Out Amount, Credit Pending) render. (onLoad is empty — verify there
   is no regression from it being empty.)
2. **Grid load recomputes amounts.** When each grid finishes loading rows, Credit In Amount, Credit Out
   Amount and Credit Pending are recomputed from the current selection (initially 0 / 0 / 0 if nothing is
   pre-selected). `Credit Pending = Credit Out Amount − Credit In Amount`.
3. **Selecting rows updates totals.** Select one or more rows in **Credit In**: Credit In Amount sums the
   selected rows' settlement amounts (using the edited value when present, else the row's
   `settlementAmount`). Do the same in **Credit Out**. Credit Pending updates to `out − in`.
4. **Edited settlement amount is honored.** Edit a row's settlement amount cell in a selected row; the
   totals reflect the edited value (not the original row value).
5. **Exceed-pending guard (Credit In).** Select a Credit In row and edit its settlement amount **above**
   its outstanding amount, then press Done. The submit is **aborted** and the message bar shows
   `OBFBPS_Err_CreditExceedPending` naming the offending `documentNo`. No call to the Java handler occurs.
6. **Exceed-pending guard (Credit Out).** Same as test 5 but on a **Credit Out** row (with Credit In
   within pending). Done aborts with `OBFBPS_Err_CreditExceedPending` for the Credit Out document.
7. **Credit-not-equal guard.** Select rows so Credit In Amount != Credit Out Amount (both within pending),
   press Done. The submit is **aborted** with `OBFBPS_Err_CreditNotEquals`.
8. **Zero-credit guard.** With nothing selected (or selections summing to 0 on both sides), press Done.
   The submit is **aborted** with `OBFBPS_Err_SelectCredit`.
9. **Stale banner clears.** Trigger a guard (e.g. test 7) to show the error, then fix the selection so it
   is valid and press Done again: the previous banner is cleared (`messageBar.hide()`) and the submit
   proceeds.
10. **Happy path / submit.** Select rows so Credit In Amount == Credit Out Amount, both > 0 and within
    pending, press Done. The pre-submit validation passes (returns nothing), and the process submits to
    `AddCreditPaymentsActionHandler` through the substrate's Window-Reference P&E path; the result is shown.
