### `em_etmeta_onprocess` — entity `obuiapp_process` (P&E pre-submit hook)

```js
async (process, view) => {
  const TYPE_DEPOSIT = 'BPD';
  const TYPE_WITHDRAWAL = 'BPW';
  const record = view.callerField && view.callerField.record;

  // Classic guard: only the Match Statement launch (record.match) gates the submit.
  if (!record || !record.match) {
    return undefined;
  }

  // The process-form values must be read through `view.theForm.getItem(<columnName>)`.
  // `view.getContextInfo()` spreads `form.getValues()`, which is keyed by the parameter
  // DISPLAY name (e.g. "Deposit Amount"), NOT the column name — so `info.depositamt`
  // would be `undefined`. `getItem(columnName)` resolves columnName -> parameter.name
  // and coerces Amount references to a Number (see verification note V1).
  const form = view.theForm;
  const toNumber = (v) => (typeof v === 'number' ? v : Number(v) || 0);
  const blineAmt = toNumber(record.amount);
  const trxAmt =
    toNumber(form.getItem('depositamt').getValue()) -
    toNumber(form.getItem('withdrawalamt').getValue());
  const paymentId = form.getItem('fin_payment_id').getValue();
  const glitemId = form.getItem('c_glitem_id').getValue();
  const trxType = form.getItem('trxtype').getValue();

  if (
    (trxType === TYPE_DEPOSIT || trxType === TYPE_WITHDRAWAL) &&
    !glitemId &&
    !paymentId
  ) {
    const text = OB.I18N.getLabel('APRM_INVALID_TRANSACTION');
    view.messageBar.setMessage(isc.OBMessageBar.TYPE_ERROR, null, text);
    return { severity: 'error', text: text };
  }

  if (trxAmt === blineAmt) {
    return undefined;
  }

  const hideSplitConfirmation = OB.PropertyStore.get(
    'APRM_MATCHSTATEMENT_HIDE_PARTIALMATCH_POPUP',
    view.windowId
  );
  if (hideSplitConfirmation === 'Y') {
    return undefined;
  }

  const formatAmount = (amount) =>
    OB.Utilities.Number.JSToOBMasked(
      amount,
      OB.Format.defaultNumericMask,
      OB.Format.defaultDecimalSymbol,
      OB.Format.defaultGroupingSymbol,
      OB.Format.defaultGroupingSize
    );

  const confirmed = await confirm(
    OB.I18N.getLabel('APRM_SplitBankStatementLineConfirm', [
      formatAmount(blineAmt),
      formatAmount(trxAmt)
    ])
  );

  if (!confirmed) {
    return { severity: 'error', text: '' };
  }
  return undefined;
}
```

### `em_etmeta_on_parameter_change` — parameter `Transaction Type` (`trxtype`)

```js
(item, view, form, grid) => {
  const TYPE_DEPOSIT = 'BPD';
  const TYPE_WITHDRAWAL = 'BPW';
  const value = item.getValue();
  const depositamt = form.getItem('depositamt');
  const withdrawalamt = form.getItem('withdrawalamt');

  if (value === TYPE_WITHDRAWAL) {
    depositamt.setDisabled(true);
    withdrawalamt.setDisabled(false);
    depositamt.setValue(0);
  } else if (value === TYPE_DEPOSIT) {
    depositamt.setDisabled(false);
    withdrawalamt.setDisabled(true);
    withdrawalamt.setValue(0);
  } else {
    depositamt.setDisabled(false);
    withdrawalamt.setDisabled(false);
    form.getItem('description').setValue('');
  }
}
```

## Advisories (non-blocking)

- **A1 — shared `amtOnChangeFunction` split into two bodies.** Classic binds the same
  `amtOnChangeFunction` to both *Deposit Amount* and *Withdrawal Amount*, branching on
  `item.name === 'depositamt' / 'withdrawalamt'`. In the new UI `item.name` resolves to the
  parameter's **display name** (`createItemProxy(..., name)` where `name` is the form-watch key =
  `parameter.name`), not the column name, so the Classic comparison would never match. The handler is
  therefore split into one body per parameter, each implicitly knowing its own identity. Behavior is
  identical: when `trxtype === 'BF'`, editing one amount zeroes the other.
- **A2 — `setDateParameterValue` → `setValue`.** The item proxy exposes no `setDateParameterValue`.
  `trxDateOnChangeFunction` copies the Transaction Date into the Accounting Date; both are date
  references, so `form.getItem('dateacct').setValue(item.getValue())` passes the committed date value
  through directly. The Classic `new Date(...)` wrap existed only to satisfy the SmartClient date item.
- **A3 — `c_bpartner_id` set by id only.** `paymentOnChangeFunction` sets the Business Partner with
  `setValue(data.cBpartnerId)` (id, no label) — reproduced verbatim. This matches Classic; the
  selector renders the identifier once the value resolves. If QA reports a missing BP label, switch to
  `setValueFromRecord({ id: data.cBpartnerId, _identifier: <label> })` once the handler returns a label
  (it currently returns only `cBpartnerId`).
- **A4 — split-cancel shows an empty error banner (substrate limitation, corrected).** When the user
  cancels the split confirmation (`APRM_SplitBankStatementLineConfirm`), the hook returns
  `{ severity: 'error', text: '' }` to abort. The substrate's only abort signal for a P&E pre-submit
  hook is an `error` response, and `runPreSubmitValidation` calls `messageBar.setMessage("error", null, "")`
  unconditionally for any `error` (useProcessExecution.ts:795-799), which sets a non-null banner with
  empty html — i.e. a visible empty error box. Classic `clientSideValidationFail()` showed nothing on
  cancel. This is **not** fixable from the migrated JS (there is no silent-abort return value) and is
  **not specific to this process** — it affects every P&E pre-submit cancel. Minor cosmetic deviation;
  does not block. A clean fix would be platform-side: have `messageBar.setMessage` (or the banner host)
  skip rendering when both title and text are empty.
- **A5 — hidden `bankStatementLineId` parameter.** Classic injects a hidden text field carrying the
  bank statement line id into the submitted `_params`. Reproduced by `addField` (dynamic parameter,
  `reference: '10'` = String) + seeding its value through the onLoad return map + `hideItem`. The
  injected parameter flows into the P&E payload via `getMergedProcessValues()` → form values → `_params`.

## Manual-test checklist (new UI)

> Launch context: open the **Add Transaction** process from a **Match Statement** bank statement line
> (the launcher whose record carries `match`, `amount`, and the context info `depositamt`,
> `withdrawalamt`, `fin_payment_id`, `c_glitem_id`, `trxtype`).

1. **onLoad — org mirror + hidden field.** Open the dialog. Confirm *Process Organization*
   (`ad_org_id_process`) is set equal to *Organization* (`ad_org_id`) on open, and that no
   `bankStatementLineId` field is visible. Submit a valid transaction and confirm the bank statement
   line id reaches the backend (the line gets matched).
2. **trxType = Withdrawal (BPW).** Set *Transaction Type* to a withdrawal type (`BPW`). Expect:
   *Deposit Amount* becomes disabled and is set to 0; *Withdrawal Amount* becomes enabled.
3. **trxType = Deposit (BPD).** Set *Transaction Type* to a deposit type (`BPD`). Expect:
   *Withdrawal Amount* becomes disabled and is set to 0; *Deposit Amount* becomes enabled.
4. **trxType = other.** Set *Transaction Type* to any other value. Expect: both amounts enabled and
   *Description* cleared.
5. **Payment onChange.** Select a *Payment*. Expect *Description*, *Deposit Amount*, *Withdrawal
   Amount* and *Business Partner* repopulate from the server (deposit vs withdrawal depends on the
   payment's transaction amount sign; the other amount is 0).
6. **G/L Item onChange.** Select a *G/L Item*. Expect *Description* updates from the server.
7. **Transaction Date onChange.** Change *Transaction Date*. Expect *Accounting Date* (`dateacct`)
   updates to the same date.
8. **Organization onChange.** Change *Organization*. Expect *Process Organization* mirrors the new
   organization value.
9. **Amount mutual-exclusion (trxType = BF).** Set *Transaction Type* to `BF`. Type a value in
   *Deposit Amount* → *Withdrawal Amount* becomes 0. Type a value in *Withdrawal Amount* → *Deposit
   Amount* becomes 0. With any other trxType, editing one amount must **not** zero the other.
10. **onProcess — invalid transaction guard.** With a matched line and `trxType` in {`BPD`,`BPW`} but
    **no** G/L Item and **no** Payment selected, press Done. Expect an error banner
    `APRM_INVALID_TRANSACTION` and the submit is aborted (modal stays open).
11. **onProcess — exact match (no split).** With a matched line where `depositamt - withdrawalamt`
    equals the bank statement line amount, press Done. Expect the process submits directly (no
    confirmation dialog).
12. **onProcess — split confirmation, accept.** With a matched line where the amounts differ and the
    preference `APRM_MATCHSTATEMENT_HIDE_PARTIALMATCH_POPUP` is **not** `Y`, press Done. Expect a
    confirmation dialog `APRM_SplitBankStatementLineConfirm` showing both masked amounts. Accept →
    submit proceeds.
13. **onProcess — split confirmation, cancel.** Same setup as #12, press Done, then cancel the
    confirmation. Expect the submit is aborted with no error banner (modal stays open).
14. **onProcess — split popup hidden by preference.** With differing amounts and
    `APRM_MATCHSTATEMENT_HIDE_PARTIALMATCH_POPUP = Y`, press Done. Expect the submit proceeds directly
    (no confirmation).
15. **onProcess — non-match launch.** Launch the process from a context **without** `match` on the
    caller record. Press Done. Expect no validation gating (the submit proceeds directly).

## Independent verification (verifier pass — 2026-06-17)

- **V1 — `em_etmeta_onprocess` context-read bug FOUND AND FIXED.** The first migrated `onProcess` read
  the process-form fields from `view.getContextInfo()` by **column name** (`info.depositamt`,
  `info.withdrawalamt`, `info.fin_payment_id`, `info.c_glitem_id`, `info.trxtype`). Verified against the
  substrate: the process form (react-hook-form) is registered with **display-name keys**
  (`field.hqlName = parameter.name`, e.g. `"Deposit Amount"`, `"Transaction Type"`) —
  `ProcessParameterSelector.tsx:190-198` + the dedicated `mapValuesByDBColumnName` remapper, and
  `assignViewData` builds `getContextInfo = () => ({ inpTabId, ...parentRecord, ...form.getValues() })`
  (`scriptProxies.ts:1026-1030`). DB confirms `parameter.name !== columnname` for all five
  (`Deposit Amount`/`depositamt`, `Withdrawal Amount`/`withdrawalamt`, `Transaction Type`/`trxtype`,
  `Payment`/`fin_payment_id`, `G/L Item`/`c_glitem_id`). So the column-name reads resolved to
  `undefined`, which would have (a) made `trxType` undefined → the `APRM_INVALID_TRANSACTION` guard never
  fire, and (b) made `trxAmt = undefined - undefined = NaN` → `NaN === blineAmt` always false → the split
  confirmation always trigger even on an exact match. **Fixed** by reading through
  `view.theForm.getItem(<columnName>).getValue()`, which `findParameter` resolves columnName →
  parameter.name (`scriptProxies.ts:724-741`) and which coerces Amount (ref id `12` ∈ numeric set,
  `form/constants.ts:134-145`) to a Number via `coerceParamValue` (`scriptProxies.ts:788-794`).
  `record.amount`/`record.match` are launcher-record fields (unchanged — correctly read off
  `view.callerField.record`).
- **Pre-submit abort contract confirmed.** `runPreSubmitValidation` (`useProcessExecution.ts:781-816`):
  `{ severity:'error', text }` → `extractResponseMessage` → `msgType === "error"` → abort (`return false`);
  `undefined` → proceed (`return true`). The hook must not call `view.executeProcess()`; the platform
  posts the P&E payload. Matches the migrated design.
- **onChange handlers verified faithful.** `form.getItem(<columnName>)` resolves correctly for all
  (`depositamt`, `withdrawalamt`, `trxtype`, `description`, `dateacct`, `ad_org_id`,
  `ad_org_id_process`, `c_bpartner_id`). A1 (item.name = display name) confirmed via `createItemProxy`
  (`name: paramName = parameter.name`) → the split of `amtOnChangeFunction` into two bodies is correct.
  The added `if (!data) return;` guards on the two remote-call callbacks are defensive and harmless.
- **Backend handlers exist** (`AddTransactionOnChangePaymentActionHandler`,
  `GLItemTransactionActionHandler`). DB parameter inventory matches the report.
