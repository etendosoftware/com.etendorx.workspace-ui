### `em_etmeta_onprocess` — entity `obuiapp_process`

```js
async (process, view) => {
  const PRECHECK_HANDLER = "org.openbravo.costing.CostingRuleProcessOnProcessHandler";
  const EXECUTE_HANDLER = "org.openbravo.costing.CostingRuleProcessActionHandler";
  view.messageBar.setMessage("info", null, OB.I18N.getLabel("CostingRuleHelp"));
  const ruleId = view.recordIds && view.recordIds[0];
  const runExecution = async () => {
    const response = await callAction(EXECUTE_HANDLER, { M_Costing_Rule_ID: ruleId }, { queryParams: { processId: process.id } });
    return response.data && response.data.message;
  };
  const precheck = await callAction(PRECHECK_HANDLER, { ruleId });
  const precheckMessage = precheck.data && precheck.data.message;
  const confirmText = precheckMessage && precheckMessage.text;
  if (!confirmText) {
    return await runExecution();
  }
  const accepted = await confirm(confirmText);
  if (!accepted) {
    return { severity: "info", text: "" };
  }
  return await runExecution();
}
```

## Advisories (non-blocking)

1. **Decline branch semantics.** Classic `closeClick()` closed the dialog when the user declined the
   confirmation. The new UI has no `closeClick()` in the script surface; the migrated decline branch
   returns `{ severity: "info", text: "" }`, so the modal stays open and does **not** report success.
   This is behaviorally safe (no execution happens, no false success) but the dialog is left open
   instead of auto-closed. If auto-close on decline is required, it would need a footer/close control
   exposed to scripts (not currently available). Validate this UX in QA test #3.
2. **Pre-check `callAction` vs `OB.RemoteCallManager.call`.** The classic file used
   `OB.RemoteCallManager.call(..., callback)`. The migration uses `await callAction(...)` for a linear,
   `async` flow (both are supported per the playbook). The response shape is unchanged
   (`{ message: { severity, title, text } }`).
3. **Record id source.** Classic read `view.parentWindow.view.lastRecordSelected.id`; the new UI
   provides the launching record(s) as `view.recordIds`. For a Pick & Execute launched on a single
   costing-rule record, `view.recordIds[0]` is the rule id. Confirm a single record is selected when
   launching (QA test #2).
4. **Correction applied — execution payload `processId`.**
   - **What was wrong.** The execution call read `await callAction(EXECUTE_HANDLER, { M_Costing_Rule_ID: ruleId })`
     with no third argument, so the request only carried `M_Costing_Rule_ID` in the JSON body.
   - **Why it failed.** Verified against source: `CostingRuleProcessActionHandler.doExecute`
     (`/home/luciano/projects/etendo/erp/src/org/openbravo/costing/CostingRuleProcessActionHandler.java`,
     line 53-54) reads `M_Costing_Rule_ID` from the body (`content`) but `processId` from the **kernel
     parameters** (`(String) parameters.get("processId")`), and then builds `new ProcessBundle(processId, vars)`.
     The runtime helper `callAction(handler, payload, options)`
     (`packages/MainUI/utils/processes/definition/utils.ts`, line 181-194) only appends kernel query
     params when `options.queryParams` is provided (`buildQuery(base, options.queryParams)`). With no
     third argument, `processId` arrived `null` → `new ProcessBundle(null, vars)` → execution failed.
   - **How it was fixed.** Pass `processId` via the third argument as a kernel query param, sourcing it
     from the hook's `process.id`:
     `await callAction(EXECUTE_HANDLER, { M_Costing_Rule_ID: ruleId }, { queryParams: { processId: process.id } })`.
     `process.id` is the `ProcessDefinition` id and is available as the first hook argument
     (verified in `packages/MainUI/hooks/types.ts`).
   - **Pre-check not affected.** `CostingRuleProcessOnProcessHandler.execute`
     (`/home/luciano/projects/etendo/erp/src/org/openbravo/costing/CostingRuleProcessOnProcessHandler.java`,
     line 54-55) reads only `ruleId` from the JSON body and never touches `parameters.get("processId")`,
     so the pre-check call `callAction(PRECHECK_HANDLER, { ruleId })` is correct as-is and was left
     unchanged. Confirm execution still succeeds in QA test #4.

## Manual-test checklist

Run these in the new UI against a costing rule record.

1. **onLoad banner.** Open the Validate Costing Rule process on a costing-rule record. Expect: the
   info message bar shows the `CostingRuleHelp` label immediately on open, before any interaction.
2. **Happy path (no confirmation needed).** With a rule that has a previous validated rule or no
   pending-period transactions (pre-check returns `message.text = null`), press Execute/OK. Expect: no
   confirmation dialog; the rule validation runs and a success toast appears; the modal closes.
3. **Confirmation path — decline.** With a rule whose pre-check returns a warning text (e.g. a legal
   entity with transactions whose movement date is in a closed period, or a null starting date with
   transactions), press Execute/OK. Expect: a confirm dialog with the warning text. Press Cancel.
   Expect: nothing executes, no success toast, the modal stays open (see advisory 1).
4. **Confirmation path — accept.** Same setup as #3 but press OK on the confirm dialog. Expect: the
   rule validation runs; a success (or the handler's error) toast/result appears matching classic.
5. **Info banner on submit.** On pressing Execute/OK, the info banner is (re)shown with the
   `CostingRuleHelp` label, matching classic onProcess line 44.
6. **Single-record guard.** Confirm the process is launched against exactly one costing-rule record so
   `view.recordIds[0]` resolves to the intended rule (advisory 3).
