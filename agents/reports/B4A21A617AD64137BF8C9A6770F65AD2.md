## In-scope code: `OB.Jobs.ProcessOrders.onLoad(view)` (lines 79–134)

Behavior:
1. Reads `DocAction` item and the selected records from the launching grid.
2. Builds distinct `documentStatuses` / `documentActions` from the selection.
3. Calls `com.smf.jobs.defaults.ProcessOrdersDefaults` via `OB.RemoteCallManager.call`.
4. In the callback: filters the `DocAction` value map to the server-returned `actions`, applies it
   with `docActionField.setValueMap(newValues)`, then **programmatically selects** an option:
   - `docActionField.setValueProgrammatically(documentActions[0])` if available, else
   - `docActionField.setValueProgrammatically(docActionField.getFirstOptionValue())`.
5. If `actions.length > 0` and the OK button is disabled, enables it:
   `view.okButton.enable()` (guarded by `view.okButton && !view.okButton.isEnabled()`).

### Process `em_etmeta_onload` (paste into the process)

Faithful port of `OB.Jobs.ProcessOrders.onLoad`. Remaps `view.parentWindow.view.viewGrid
.getSelectedRecords()` → `view.selectedRecords` and `view.processOwnerView.tabId` → `view.tabId` (both
provided in the onLoad `hookData`); `Array.add` → `.push`; and the `currentValues[action]` object idiom
→ `currentValues.find(...)` because `getValueMap()` returns a `ListOption[]` array.

```js
(view) => {
  const form = view.theForm;
  const docActionField = form.getItem('DocAction');
  const selectedRecords = view.selectedRecords || [];
  const tabId = view.tabId;
  const documentStatuses = [];
  const documentActions = [];

  for (let i = 0; i < selectedRecords.length; i++) {
    const record = selectedRecords[i];
    if (!documentStatuses.includes(record.documentStatus)) {
      documentStatuses.push(record.documentStatus);
    }
    if (!documentActions.includes(record.documentAction)) {
      documentActions.push(record.documentAction);
    }
  }

  OB.RemoteCallManager.call(
    'com.smf.jobs.defaults.ProcessOrdersDefaults',
    { documentStatuses: documentStatuses, isProcessing: '', tabId: tabId },
    {},
    (response, data, request) => {
      const actions = data.actions || [];
      const currentValues = docActionField.getValueMap();
      const newValues = {};

      for (let i = 0; i < actions.length; i++) {
        const action = actions[i];
        const match = currentValues.find((option) => option.value === action || option.id === action);
        if (match) {
          newValues[action] = match.label;
        }
      }

      docActionField.setValueMap(newValues);
      // Avoid setting an action that is on the record(s) but not in the dropdown list.
      if (newValues[documentActions[0]]) {
        docActionField.setValueProgrammatically(documentActions[0]);
      } else {
        docActionField.setValueProgrammatically(docActionField.getFirstOptionValue());
      }

      // If the OK button is not enabled but there are actions to select, enable it.
      if (actions.length > 0 && view.okButton && !view.okButton.isEnabled()) {
        view.okButton.enable();
      }
    }
  );
}
```

## Advisories (non-blocking)

1. **`Array.add` → `.push`.** The classic loops use `documentStatuses.add(...)` / `documentActions.add(...)`,
   which only work under SmartClient's `Array` augmentation. Any future migration must convert these to
   `.push(...)`. Not a platform gap.
2. **`processOwnerView.tabId` → `view.tabId`.** Resolved: the migrated onLoad reads `view.tabId`
   (provided in the onLoad `hookData`), not the classic `processOwnerView` member.
3. **Family cloning.** The `processRecords` family (Process Orders/Shipment/Invoices) clones the same
   `onLoad` shape. The capabilities added here unblock every member; each is still migrated on its own
   row (Process Invoices additionally binds an `onChangeDocumentAction` handler, out of scope here).

## Platform changes (2026-06-12)

Implemented in the client substrate to unblock this process:

- **`item.setValueProgrammatically(value)`** — live item method; sets value + displayed label, selecting
  the existing option (reuses `applyValue`). The new UI has no user `onChange` to suppress on the
  migrated-hook path (and `DocAction` binds none here), so this maps to the same value+identifier set as
  `setValue`. `scriptProxies.ts`.
- **`item.getFirstOptionValue()`** — live item method; returns the first option `value` of the current
  value map (`readValueMap(...)[0]?.value`). `scriptProxies.ts`.
- **`view.okButton`** — `{ isEnabled(), enable() }`. `isEnabled()` reads the live execute-button state
  (via `okEnabledRef`); `enable()` force-enables it through a new `ScriptButtonState.okForceEnabled`
  flag that overrides **only** the "mandatory empty" reason (not `isPending` / submit). `scriptProxies.ts`,
  `ProcessDefinitionModal.tsx`, `utils.ts`.
- **Key-robust value map / mandatory** — `withRefList` / `withMandatory` / `fieldController.getValueMap`
  now resolve the parameter by map-key OR `name` OR `dBColumnName` (the map is keyed by `dBColumnName`
  but `DocAction`'s name "Document Action" ≠ `dBColumnName` "DocAction"), so `setValueMap`/`getValueMap`
  actually update the field. `utils.ts`, `ProcessDefinitionModal.tsx`.

Tests: `scriptProxies.test.ts` (new item methods + `view.okButton`), `utils.test.ts` (`withOkForceEnabled`
and the name/dBColumnName resolution of `withRefList`/`withMandatory`). Jest green; `tsc` and Biome clean.
