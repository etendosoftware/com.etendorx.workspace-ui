## In-scope code: `OB.Jobs.ProcessOrders.onLoad(view)` (lines 79–134)

Behavior:
1. Reads `DocAction` item and the selected records from the launching grid.
2. Builds distinct `documentStatuses` / `documentActions` from the selection.
3. Calls `com.smf.jobs.defaults.ProcessOrdersDefaults` via `OB.RemoteCallManager.call`.
4. In the callback: filters the `DocAction` value map to the server-returned `actions`, applies it
   with `docActionField.setValueMap(newValues)`, then **programmatically selects** an option:
   - `docActionField.setValueProgrammatically(documentActions[0])` if available, else
   - `docActionField.setValueProgrammatically(docActionField.getFirstOptionValue())`.
5. If `actions.length > 0` and the OK button is disabled, enables it:
   `view.okButton.enable()` (guarded by `view.okButton && !view.okButton.isEnabled()`).

### Gate result

Three entries are **`unsupported`** (`setValueProgrammatically`, `getFirstOptionValue`,
`view.okButton.*`) and one is **`best-effort`** (`processOwnerView.tabId`). Per the conservative
feasibility gate, this process is **`blocked`**. No migrated code is generated.

**Relation to A5A9B914 (verified independently, not assumed):**
- **Shared blockers:** `setValueProgrammatically()` and `getFirstOptionValue()` — present in both
  `ProcessInvoices.onLoad` and `ProcessOrders.onLoad`. Same root cause: these item-proxy methods do
  not exist in the new UI.
- **Additional blocker unique to this process:** `view.okButton.enable()` / `isEnabled()`
  (`ProcessOrders.onLoad` lines 129–131). `ProcessInvoices.onLoad` has no OK-button manipulation, so
  A5A9B914 did not surface this gap. ProcessOrders is therefore **strictly more blocked** than the
  Invoice variant.

## Advisories (non-blocking)

1. **`Array.add` → `.push`.** The classic loops use `documentStatuses.add(...)` / `documentActions.add(...)`,
   which only work under SmartClient's `Array` augmentation. Any future migration must convert these to
   `.push(...)`. Not a platform gap.
2. **`processOwnerView.tabId` fallback.** When the unblocking work lands, the `tabId` resolution should
   be remapped onto a documented view-proxy accessor (`view.getContextInfo()` / `view.activeView.tabId`)
   rather than the classic `processOwnerView` member.
3. **Family cloning.** Per the playbook, the `processRecords` family (Create Inverse Invoice/Order,
   Process Orders/Shipment/Invoices) clones one migrated body into each process row. The OK-button and
   programmatic-selection blockers affect every member that uses `ProcessOrders.onLoad` /
   `ProcessShipment.onLoad` shape, so unblocking is shared work.

## What the new UI must add to unblock this process

1. An item-proxy method equivalent to Classic `setValueProgrammatically(value)` — set a selector's
   value to an existing option without firing onChange.
2. An item-proxy method equivalent to Classic `getFirstOptionValue()` — read the first available option
   value of a selector from a hook.
3. A view-proxy handle for the execute/OK button exposing `isEnabled()` and `enable()` (the
   `view.okButton` surface), so `onLoad` can enable the button after populating actions.
